# Kubernetes с нуля: оркестрация контейнеров

## Проблема Docker Compose в production

Docker Compose — отличный инструмент для локальной разработки и запуска простых стеков на одной машине. Однако, когда речь заходит о промышленной эксплуатации (production), проявляются его существенные ограничения:

1. **Масштабируемость (Scalability):** Compose ограничен ресурсами одного узла. Если ваше приложение требует больше CPU или RAM, чем может дать один сервер, Compose не поможет распределить нагрузку на кластер машин.
2. **Отказоустойчивость (High Availability):** Если физический сервер с Docker Compose упадет, все контейнеры остановятся. Нет встроенного механизма автоматического перезапуска контейнеров на другом исправном узле.
3. **Самовосстановление (Self-healing):** В K8s, если контейнер перестает отвечать на проверки (health checks), он будет перезапущен. Compose умеет перезапускать упавшие процессы (`restart: always`), но не справляется с распределенными сбоями.
4. **Сложное обновление (Rolling Updates):** Обновление приложения в Compose обычно требует кратковременной остановки сервиса. Kubernetes позволяет обновлять приложения без простоя (zero-downtime).
5. **Управление конфигурациями и секретами:** В Compose управление секретами реализовано базово. Kubernetes предлагает мощные абстракции ConfigMap и Secret для динамического управления настройками.

## Архитектура Kubernetes: Control Plane vs Worker Nodes

Kubernetes работает по модели Master-Worker. Кластер состоит из управляющего слоя (Control Plane) и рабочих узлов (Worker Nodes).

### Control Plane (Управляющий слой)
Это "мозг" кластера, принимающий решения о планировании и реагирующий на события.

*   **kube-apiserver:** Точка входа для всех запросов (CLI, UI, внутренние компоненты). Единственный компонент, который общается с `etcd`.
*   **etcd:** Распределенное хранилище "ключ-значение". Здесь хранится все состояние кластера (конфигурации, количество запущенных подов и т.д.).
*   **kube-scheduler:** Следит за новыми Подами, у которых не назначена Нода, и выбирает для них подходящий узел на основе ресурсов (CPU, RAM).
*   **kube-controller-manager:** Запускает процессы контроллеров, которые следят за состоянием кластера (например, Node Controller следит за доступностью узлов, Replication Controller — за количеством копий приложения).

### Worker Nodes (Рабочие узлы)
Здесь запускаются и работают ваши приложения.

*   **kubelet:** Агент, работающий на каждой ноде. Он следит, чтобы контейнеры были запущены в соответствии с инструкциями от Control Plane.
*   **kube-proxy:** Сетевой прокси, обеспечивающий сетевые правила на узлах (iptables/ipvs). Он позволяет подам общаться друг с другом и с внешним миром.
*   **Container Runtime:** Среда выполнения контейнеров (например, Docker, containerd, CRI-O). Kubernetes сам не запускает контейнеры, он отдает команды рантайму.

## Установка minikube и kubectl

Для локального обучения мы будем использовать **minikube** — инструмент, запускающий одноузловой кластер K8s внутри виртуальной машины или Docker-контейнера.

### Шаг 1: Установка kubectl
`kubectl` — это CLI для управления кластером.

**Для Linux:**
```bash
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
chmod +x kubectl
sudo mv kubectl /usr/local/bin/
```

### Шаг 2: Установка minikube
**Для Linux:**
```bash
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube
```

### Шаг 3: Запуск кластера
```bash
# Запуск с драйвером docker (рекомендуется)
minikube start --driver=docker --cpus=2 --memory=4g

# Проверка статуса
kubectl cluster-info
kubectl get nodes
```

## Базовые объекты Kubernetes

1. **Pod (Под):** Минимальная единица развертывания. Под может содержать один или несколько контейнеров, которые делят общую сеть и хранилище. Контейнеры внутри одного пода всегда запускаются на одной ноде.
2. **Node (Нода):** Виртуальная или физическая машина в кластере.
3. **Namespace (Пространство имен):** Виртуальный кластер внутри физического. Позволяет логически разделять ресурсы (например, `dev`, `staging`, `prod`).

## Первый запуск: Работа с Pod

Запустим первый контейнер с Nginx:

```bash
# Создание пода напрямую (не рекомендуется для продакшена, но полезно для тестов)
kubectl run nginx --image=nginx:alpine

# Просмотр списка подов
kubectl get pods

# Подробная информация о поде (события, IP, статус)
kubectl describe pod nginx

# Просмотр логов контейнера
kubectl logs nginx

# Выполнение команды внутри контейнера
kubectl exec -it nginx -- sh
# (внутри sh) ls /usr/share/nginx/html
```

## KPI: Таблица основных команд kubectl

| Команда | Описание | Пример |
|---------|----------|--------|
| `get` | Список ресурсов | `kubectl get pods`, `kubectl get ns` |
| `describe` | Детальная информация о ресурсе | `kubectl describe node minikube` |
| `logs` | Вывод логов контейнера | `kubectl logs my-pod` |
| `exec` | Выполнение команды в контейнере | `kubectl exec -it my-pod -- bash` |
| `delete` | Удаление ресурса | `kubectl delete pod nginx` |
| `apply` | Создание/обновление из YAML | `kubectl apply -f config.yaml` |
| `edit` | Редактирование ресурса "на лету" | `kubectl edit pod nginx` |

## Антипаттерны (на заметку)
*   **Запуск Pod напрямую:** В реальных проектах поды запускаются через `Deployment` или `StatefulSet`. Одиночный под не будет перезапущен на другой ноде при сбое текущей.
*   **Все в namespace default:** Со временем кластер превращается в "свалку". Используйте неймспейсы для разделения проектов.
*   **Игнорирование лимитов:** Без указания `resources.limits` один "прожорливый" под может занять все ресурсы ноды, убив соседей.
