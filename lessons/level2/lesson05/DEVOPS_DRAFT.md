# Deployments и Services: деплоим приложение в K8s

В прошлом уроке мы разобрали архитектуру Kubernetes и узнали, из каких компонентов состоит кластер. Теперь пришло время перейти к практике и научиться запускать приложения так, как это делают в продакшене. Мы разберем, почему нельзя просто запустить Pod, как Deployment управляет жизненным циклом приложения и как Service обеспечивает доступ к вашим микросервисам.

---

## Почему не запускать Pod напрямую?

**Pod** — это минимальная единица развертывания в Kubernetes. Это один или несколько контейнеров, которые делят общие ресурсы (сеть, хранилище). Казалось бы, если нам нужно запустить контейнер, достаточно создать объект типа Pod. Но на практике так почти никогда не делают.

**Основные проблемы "голых" Pods:**
1. **Отсутствие самолечения (Self-healing):** Если узел (Node), на котором запущен Pod, выйдет из строя, Pod не будет пересоздан автоматически на другом узле. Он просто умрет.
2. **Сложность масштабирования:** Чтобы запустить 10 копий приложения, вам пришлось бы вручную создавать 10 разных Pods.
3. **Трудности обновления:** Обновление версии образа превращается в ручной процесс удаления старых Pods и создания новых.

Для решения этих задач Kubernetes предлагает уровни абстракции: **ReplicaSet** и **Deployment**.

---

## Deployment — декларативное управление репликами

**Deployment** — это высокоуровневый контроллер, который позволяет декларативно описывать желаемое состояние приложения. Вы говорите: "Я хочу, чтобы в кластере всегда работало 3 реплики образа `nginx:1.25`", и Deployment берет на себя всю работу по достижению этого состояния.

### ReplicaSet — страж количества реплик
Под капотом Deployment создает **ReplicaSet**. Его единственная задача — следить за тем, чтобы количество запущенных Pods соответствовало числу `replicas`, указанному в манифесте. Если какой-то Pod упадет или будет удален, ReplicaSet мгновенно заметит расхождение и запустит новый экземпляр.

### Разбор Deployment YAML

Рассмотрим стандартный манифест для деплоя веб-приложения:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
      - name: webapp
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "64Mi"
            cpu: "250m"
          limits:
            memory: "128Mi"
            cpu: "500m"
        readinessProbe:
          httpGet:
            path: /healthz
            port: 80
```

**Объяснение ключевых полей:**

*   **`apiVersion` и `kind`**: Определяют тип ресурса. Для Deployment это всегда `apps/v1`.
*   **`metadata.name`**: Уникальное имя нашего деплоймента в рамках namespace.
*   **`spec.replicas`**: Количество идентичных копий Pods, которые должны быть запущены.
*   **`spec.selector`**: Важнейшая часть. Определяет, какими Pod-ами управляет этот Deployment. Он ищет Pod-ы с метками (labels), указанными в `matchLabels`.
*   **`spec.template`**: Шаблон для создания новых Pods.
    *   **`metadata.labels`**: Метки, которые будут присвоены каждому созданному Pod-у. Они **должны совпадать** с `matchLabels` выше.
    *   **`spec.containers`**: Описание контейнера (образ, порты).
    *   **`resources`**: Лимиты и запросы ресурсов (см. ниже).
    *   **`readinessProbe`**: Проверка готовности приложения.

---

## Probes: Liveness и Readiness

В Kubernetes контейнер считается "живым", пока процесс внутри него запущен. Но это не значит, что приложение работает корректно. Например, Java-приложение может запуститься, но еще 30 секунд "прогреваться" и не принимать трафик.

### Readiness Probe (Проба готовности)
Определяет, готово ли приложение принимать сетевой трафик. Если проба провалена, Kubernetes временно исключает этот Pod из списка адресов сервиса (Service endpoints).
*   **Зачем:** Чтобы пользователи не получали ошибку 502/504 во время запуска или временной перегрузки приложения.

### Liveness Probe (Проба живучести)
Проверяет, не зависло ли приложение окончательно (например, deadlock). Если проба провалена, Kubernetes **убивает и перезапускает** контейнер.
*   **Зачем:** Для автоматического восстановления после критических сбоев.

> **Важно:** Всегда указывайте `readinessProbe` и `resources.limits`. Запуск приложения без лимитов в продакшене — прямой путь к "падению" всего узла из-за утечки памяти в одном контейнере.

## Service: открываем доступ к приложению

Pod-ы эфемерны: они умирают и воскресают с новыми IP-адресами. Чтобы другие компоненты (или пользователи) могли стабильно обращаться к приложению, нужен **Service**.

Service — это абстракция, которая определяет логический набор Pod-ов и политику доступа к ним.

### Типы сервисов:
1.  **ClusterIP (по умолчанию):** Сервис доступен только внутри кластера. Идеально для баз данных или внутренних API.
2.  **NodePort:** Открывает статический порт на каждой ноде кластера. Приложение становится доступным снаружи по адресу `<NodeIP>:<NodePort>`.
    *   *Антипаттерн:* Не используйте NodePort в продакшене для веб-трафика. Это небезопасно и неудобно в масштабировании. Используйте Ingress или LoadBalancer.
3.  **LoadBalancer:** Создает облачный балансировщик (в AWS, GCP, Azure) и направляет трафик в кластер.

### Связка Deployment ↔ Service через Labels и Selectors

Service находит Pod-ы по меткам. Если в Deployment указано `app: webapp`, то и в Service в поле `selector` должно быть `app: webapp`.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-service
spec:
  selector:
    app: webapp
  ports:
    - protocol: TCP
      port: 80
      targetPort: 80
  type: ClusterIP
```

## Rolling Update и Rollback

Одно из главных преимуществ Deployment — механизм **Rolling Update** (плавное обновление). Он позволяет обновить версию приложения без простоя (zero-downtime).

### Как это работает:
1. Вы меняете версию образа в манифесте и применяете его.
2. Deployment создает новый ReplicaSet с новым образом.
3. Постепенно запускаются новые Pod-ы. Как только новый Pod проходит `readinessProbe`, один старый Pod удаляется.
4. Процесс продолжается, пока все Pod-ы не будут заменены на новые.

### Команды для управления:

1. **Применить изменения:**
   ```bash
   kubectl apply -f deployment.yaml
   ```

2. **Обновить образ "на лету":**
   ```bash
   kubectl set image deployment/webapp webapp=nginx:1.26
   ```

3. **Следить за статусом обновления:**
   ```bash
   kubectl rollout status deployment/webapp
   ```

4. **Откат (Rollback), если что-то пошло не так:**
   Если новая версия образа оказалась битой (например, падает при запуске), вы можете мгновенно вернуться к предыдущему стабильному состоянию:
   ```bash
   kubectl rollout undo deployment/webapp
   ```

## Заключение

Мы разобрали "святую троицу" Kubernetes:
*   **Pod** — где живет контейнер.
*   **Deployment** — как мы масштабируем и обновляем Pod-ы.
*   **Service** — как мы даем доступ к этим Pod-ам.

В реальных проектах вы всегда будете описывать эти объекты в YAML-файлах, версионировать их в Git и деплоить через CI/CD пайплайны. В следующем уроке мы разберем ConfigMaps и Secrets — как передавать настройки в приложения, не меняя код и образы.
