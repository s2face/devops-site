# Ansible Roles: пишем переиспользуемый код

Автоматизация инфраструктуры — это не только написание скриптов, которые «просто работают». Это создание системы, которую легко поддерживать, масштабировать и передавать другим участникам команды. Если в начале пути с Ansible один длинный Playbook кажется удобным решением, то с ростом проекта он неизбежно превращается в «простыню» кода, в которой страшно что-то менять. 

В этом уроке мы перейдем на следующий уровень владения инструментом — изучим **Ansible Roles (Роли)**. Мы разберем, как превратить хаотичный набор задач в элегантную модульную структуру, научимся использовать мощь шаблонизатора Jinja2 и узнаем, как не изобретать велосипед, используя готовые решения из сообщества Ansible Galaxy.

## Когда один playbook перестаёт справляться

На начальных этапах обучения мы привыкаем к структуре, где все задачи (tasks), переменные (vars) и обработчики (handlers) живут в одном YAML-файле. Для настройки одного сервера с парой сервисов это приемлемо. Но давайте посмотрим на типичный путь развития проекта и те «боли», которые возникают при использовании монолитного подхода.

### Проблема 1: «Спагетти-код» и радиус поражения
В монолитном плейбуке переменные для базы данных могут находиться рядом с переменными для веб-сервера. Если вы случайно опечатаетесь в имени переменной или переопределите её в середине файла, это может привести к непредсказуемым последствиям в совершенно другой части автоматизации. Чем длиннее файл, тем сложнее отследить зависимости между задачами. В итоге внесение небольшого изменения превращается в игру «Сапер»: вы не знаете, где «рванет» в следующий раз. Отсутствие четких границ между компонентами системы приводит к тому, что «радиус поражения» любой ошибки становится неоправданно велик. Это усложняет аудит безопасности и соответствие стандартам (compliance).

### Проблема 2: Рост количества сервисов
Сначала вы ставили только Nginx. Потом добавили Docker, затем PostgreSQL, Redis, систему мониторинга Prometheus и экспортеры. Playbook вырос до 1000 строк. Читать такой файл становится физически тяжело. Поиск нужной задачи превращается в бесконечный скроллинг. Инженеры тратят больше времени на навигацию по файлу, чем на саму автоматизацию. Постепенно такой плейбук становится «священным знанием», которое понимает только его автор, что недопустимо в современной командной разработке. Потеря такого «автора» становится критическим риском для бизнеса (так называемый Bus Factor).

### Проблема 3: Разные окружения и нарушение принципа DRY
Вам нужно развертывать приложение на `dev`, `staging` и `production`. Конфигурации везде чуть-чуть отличаются. Вы начинаете плодить условия `when` внутри задач, усложняя логику до предела. Или, что еще хуже, создаете три копии одного и того же файла, что нарушает принцип DRY (Don't Repeat Yourself). Если вам нужно будет обновить версию ПО, придется менять её во всех копиях, что неминуемо приведет к ошибкам и рассинхрону окружений. Инфраструктура становится хрупкой, «уникальной» для каждого сервера (snowflake servers), что противоречит самой идее автоматизации.

### Проблема 4: Сложность командной работы
Когда над одним файлом работают несколько инженеров, конфликты при слиянии веток (merge conflicts) становятся ежедневной реальностью. Git не всегда может корректно объединить изменения в одном огромном файле. Роли позволяют разделить зоны ответственности: один инженер может работать над ролью базы данных, другой — над ролью веб-сервера, не мешая друг другу. Каждая роль живет в своей директории, имеет свою историю изменений и свои тесты.

**Ansible Roles** решают эти проблемы, предлагая стандарт де-факто для структурирования кода. Роль — это независимый модуль автоматизации, который фокусируется на одной конкретной задаче. Это своего рода «черный ящик»: вы подаете на вход переменные (интерфейс), а на выходе получаете настроенный сервис (реализация).

## Структура роли Ansible

Роль — это строго определенная иерархия директорий. Ansible автоматически знает, где искать задачи, где лежат шаблоны, а где переменные. Это магия соглашений над конфигурацией (convention over configuration), которая делает Ansible таким популярным.

### Глубокий разбор папок роли:

1. **`tasks/`**: Основной список задач. Файл `main.yml` — точка входа. Если задач много, их стоит разделять по смыслу. Например, `install.yml` для установки пакетов, `configure.yml` для настройки конфигов и `service.yml` для управления состоянием служб.
   
2. **`handlers/`**: Задачи, вызываемые через `notify`. Срабатывают один раз в конце. Идеально для перезагрузки сервисов после изменения конфигурационных файлов. Это предотвращает лишние рестарты, если вы правите несколько конфигов за один прогон.

3. **`defaults/`**: Переменные по умолчанию. Самый низкий приоритет. Идеально для портов, версий и путей к логам. Эти переменные служат основной документацией для пользователя роли. Открыв этот файл, инженер должен сразу понять, что он может переопределить.

4. **`vars/`**: Константы с высоким приоритетом. Сюда выносятся системные параметры, которые пользователь не должен менять вручную. Например, названия пакетов для разных дистрибутивов или внутренние флаги логики.

5. **`templates/`**: Шаблоны Jinja2 (с расширением `.j2`). Это динамические конфигурации. Ansible считывает их, подставляет значения переменных и сохраняет результат на целевом хосте. Шаблоны позволяют генерировать уникальные файлы для каждого сервера на основе одного образца.

6. **`files/`**: Статические файлы (сертификаты, скрипты, картинки). Всё, что нужно скопировать на сервер «как есть», без обработки шаблонизатором.

7. **`meta/`**: Метаданные и зависимости. Файл `main.yml` здесь описывает автора, поддерживаемые платформы и, самое главное, зависимости от других ролей.

8. **`library/` & `filter_plugins/`**: Сюда можно положить свои собственные модули на Python или кастомные фильтры Jinja2, расширяя стандартные возможности Ansible.

### Разбор meta/main.yml
Этот файл — паспорт вашей роли. Вот пример того, как он должен выглядеть:
```yaml
galaxy_info:
  author: devops_team
  description: Role for professional deployment of Nginx with SSL support
  company: MySuperCorp
  license: MIT
  min_ansible_version: 2.15
  platforms:
    - name: Ubuntu
      versions: [20.04, 22.04]
    - name: CentOS
      versions: [7, 8]

dependencies:
  - role: geerlingguy.docker
    vars:
      docker_install_compose: true
```
Наличие зависимостей позволяет строить сложные системы из простых, надежных и протестированных кирпичиков.

## Создание роли через ansible-galaxy init

Используйте встроенную утилиту для генерации скелета роли. Это сэкономит время и предотвратит опечатки в именах папок.

```bash
mkdir -p roles
ansible-galaxy init roles/web_server
```

Типичный вывод команды инициализации выглядит так:
```text
- Role roles/web_server was created successfully
```

Это создаст все папки и файлы `main.yml`. Даже если некоторые из них останутся пустыми, они служат «дорожной картой» для развития роли. Например, папка `tests/` напомнит вам о необходимости написать проверочные тесты.

**Совет:** В профессиональной разработке неиспользуемые папки часто удаляют перед коммитом, чтобы не засорять проект. Однако для новичков лучше оставить их, чтобы привыкнуть к структуре.

## Динамические задачи: import vs include

В Ansible есть два способа подключения внешних файлов с задачами:

*   **`import_tasks`**: Статическое подключение. Файлы парсятся в момент загрузки плейбука. Ошибки синтаксиса будут видны сразу. Это надежнее, но менее гибко.
*   **`include_tasks`**: Динамическое подключение. Происходит во время выполнения. Это позволяет загружать файлы на основе фактов, полученных в середине работы:
    ```yaml
    - include_tasks: "{{ ansible_os_family }}.yml"
    ```
    Это загрузит `Debian.yml` или `RedHat.yml` на лету, что идеально для мультиплатформенных ролей.

## Иерархия переменных: глубокое погружение

Приоритеты в Ansible — это сложная тема (всего 22 уровня), но для работы с ролями важна следующая упрощенная цепочка (от самого слабого к самому сильному):

1.  **Role Defaults**: `role/defaults/main.yml`. Здесь мы объявляем всё, что может быть изменено.
2.  **Inventory Group Vars**: Переменные для групп хостов из инвентаря.
3.  **Inventory Host Vars**: Переменные для конкретных хостов.
4.  **Playbook Vars**: Переменные, заданные в секции `vars` самого плейбука.
5.  **Role Vars**: `role/vars/main.yml`. Эти переменные перебьют переменные из плейбука. Используйте их только для внутренних констант.
6.  **Extra Vars**: Параметр `-e` в командной строке. Это «ядерная кнопка», которая перебивает абсолютно всё.

**Золотое правило:** Всегда используйте уникальные префиксы для переменных. Вместо `port` пишите `web_port` или `nginx_listen_port`. Это гарантирует, что переменные разных ролей не будут конфликтовать друг с другом.

## Jinja2 шаблоны — динамические конфиги

Шаблоны — это сердце гибкости Ansible. Они позволяют адаптировать конфигурацию под конкретный сервер, окружение или даже время суток.

### Основные элементы:
*   `{{ variable }}` — вставка значения переменной. Можно использовать фильтры: `{{ domain | lower }}`.
*   `{% if condition %}` — условная логика. Позволяет включать или выключать блоки конфига.
*   `{% for item in list %}` — циклы. Незаменимы для генерации списков апстримов или виртуальных хостов. **Апстрим (Upstream)** в контексте Nginx — это группа серверов, на которые проксируется входящий трафик для балансировки нагрузки.
*   `{{ value | default(80) }}` — установка значения по умолчанию, если переменная не задана.

Пример сложного шаблона Nginx:
```nginx
# templates/nginx.conf.j2
worker_processes {{ ansible_processor_vcpus | default(1) }};

server {
    listen {{ web_port | default(80) }};
    server_name {{ domain_name }};

    access_log /var/log/nginx/{{ domain_name }}.access.log;

    location / {
        proxy_pass http://{{ backend_host }}:{{ backend_port }};
        {% if enable_compression %}
        gzip on;
        {% endif %}
    }
}
```

## Условия, циклы и теги в tasks

*   **`when`**: Условие выполнения задачи. Позволяет делать роль адаптивной (например, ставить разные пакеты на Ubuntu и CentOS).
*   **`loop`**: Современный способ перебора элементов (вместо устаревшего `with_items`).
*   **`tags`**: Позволяют запускать только часть кода: `ansible-playbook site.yml --tags deploy`. Это экономит массу времени при отладке.
*   **`register`**: Позволяет сохранить результат выполнения задачи в переменную. Это критически важно, когда вам нужно использовать вывод одной команды для принятия решения в другой. Например, вы можете проверить наличие файла или захватить ID созданного контейнера:
    ```yaml
    - name: Check if backup directory exists
      stat:
        path: /var/backups/myapp
      register: backup_dir

    - name: Create backup directory if it is missing
      file:
        path: /var/backups/myapp
        state: directory
        mode: '0755'
      when: not backup_dir.stat.exists
    ```
*   **`block`**: Группировка задач для совместной обработки ошибок. С помощью `rescue` можно прописать логику отката (rollback), если деплой пошел не по плану.

## Ansible Galaxy и коллекции

[Ansible Galaxy](https://galaxy.ansible.com/) — это «App Store» для DevOps-инженеров. Зачем писать роль для установки PostgreSQL с нуля, если Jeff Geerling (`geerlingguy`) уже сделал это идеально?

### Работа с коллекциями:
С версии 2.9 Ansible перешел на **Collections**. Коллекция объединяет в себе роли, модули (на Python) и плагины. Это более современный способ дистрибуции контента.

Установка коллекции:
```bash
ansible-galaxy collection install community.docker
```

При успешной установке вы увидите примерно такой вывод:
```text
Process dependency 'community.docker'
Starting galaxy collection install process
Installing 'community.docker:3.4.10' to '/home/user/.ansible/collections/ansible_collections/community/docker'
community.docker:3.4.10 was installed successfully
```

Для работы с Docker Compose в современных ролях часто используют модуль `community.docker.docker_compose_v2`. Вот пример его использования:
```yaml
- name: Deploy application with Docker Compose
  community.docker.docker_compose_v2:
    project_src: "/opt/myapp"  # Путь к директории, где лежит docker-compose.yml
    state: present             # Убедиться, что контейнеры запущены
    pull: always               # Всегда скачивать свежие образы перед запуском
```
Параметр `project_src` является ключевым — он указывает Ansible, в какой директории на целевом хосте искать файл конфигурации Docker Compose.

Управление зависимостями через `requirements.yml`:
```yaml
# requirements.yml
roles:
  - name: geerlingguy.nginx
    version: 3.1.1
collections:
  - name: community.general
    version: 8.0.0
```

## Антипаттерны при работе с ролями

Чтобы ваши роли были по-настоящему качественными, избегайте следующих распространенных ошибок:

1.  **❌ Всё в `tasks/main.yml`**: Не сваливайте все задачи в один файл. Разделяйте их на логические блоки (`install.yml`, `configure.yml`, `service.yml`). Это значительно упрощает чтение и поддержку кода.
2.  **❌ Хардкод значений**: Никогда не пишите пути, порты или версии прямо в задачах. Выносите их в `defaults/main.yml`. Если завтра порт изменится, вам придется искать его по всему коду, вместо того чтобы поменять одну строку в конфиге.
3.  **❌ Роли без `meta/main.yml`**: Если ваша роль требует наличия Docker или базы данных, обязательно укажите это в зависимостях. Иначе другой инженер запустит вашу роль на пустом сервере, и она упадет с непонятной ошибкой.
4.  **❌ Использование `shell` вместо модулей**: Ansible — это декларативный инструмент. Если есть модуль `apt`, `yum` или `template`, используйте его. Команды `shell` должны быть последним средством, когда стандартных модулей не существует.
5.  **❌ Отсутствие префиксов у переменных**: Если вы назовете переменную просто `port`, она может быть случайно перезаписана другой ролью. Используйте `nginx_port`, `mysql_port` и так далее.

## Секреты в ролях: Ansible Vault

Никогда не храните пароли, API-ключи или сертификаты в открытом виде в Git!
1. Создайте зашифрованный файл: `ansible-vault create vars/secrets.yml`.
2. Используйте переменные из этого файла в задачах.
3. При запуске добавьте флаг `--ask-vault-pass` или укажите путь к файлу с паролем в `ansible.cfg`.
Это критически важный стандарт безопасности в любой IT-компании.

## Тестирование: Molecule и Ansible Lint

На профессиональном уровне вы не можете просто «надеяться», что ваша автоматизация работает. Вам нужны автоматические проверки.

*   **Ansible Lint**: Проверяет ваш код на соответствие лучшим практикам (например, наличие имен у задач, отсутствие хардкода паролей, правильное использование модулей).
*   **Molecule**: Позволяет поднять временный Docker-контейнер, прогнать вашу роль «на чистую», проверить результат (например, с помощью Python-тестов Testinfra) и убедиться в идемпотентности. Идемпотентность означает, что повторный запуск роли не вносит изменений, если система уже находится в нужном состоянии.

**Важное дополнение:** В современных версиях Molecule плагины для драйверов (например, Docker) вынесены в отдельный пакет. Рекомендуется устанавливать их через `pip install molecule-plugins[docker]`.

Команда для запуска полного цикла тестов: `molecule test`.

## CI/CD: Pipeline для вашей инфраструктуры

Роли должны проверяться автоматически при каждом пуше в Git. Пример простого пайплайна (GitHub Actions или GitLab CI):
```yaml
test_role:
  image: python:3.10
  script:
    - pip install ansible molecule[docker] ansible-lint
    - ansible-lint roles/
    - molecule test
```
Это гарантирует, что в вашу основную ветку не попадет код, который ломает сервера.

## Лучшие практики (Best Practices)

Чтобы ваши роли были по-настоящему профессиональными, следуйте этим советам:

1.  **Идемпотентность**: Это религия Ansible. Второй прогон роли не должен ничего менять.
2.  **Префиксы**: Все переменные должны начинаться с имени роли (например, `myapp_port`).
3.  **Документация**: Обязательно заполняйте `README.md`. Опишите, какие переменные принимает роль и какие зависимости ей нужны.
4.  **Одна роль — одна функция**: Не пытайтесь создать «ультра-роль», которая ставит и базу данных, и веб-сервер. Разбейте их на две независимые роли.
5.  **Используйте теги**: Это сделает вашу жизнь проще при отладке.
6.  **Стандарты именования**: Используйте `snake_case` для переменных и понятные английские предложения для имен задач.

## Заключение и Практика

Роли — это путь от «системного администратора со скриптами» к «DevOps-инженеру с архитектурным мышлением». Они позволяют масштабировать инфраструктуру до тысяч серверов, сохраняя при этом чистоту кода и легкость поддержки.

### Практическое задание
Создайте роль `deploy_docker_app`, выполнив следующие шаги. 

**Важное примечание (Prerequisites):** Перед выполнением убедитесь, что на целевом хосте установлены Docker, Docker Compose и библиотека `python3-docker`. Для установки Docker вы можете использовать стороннюю роль (например, `geerlingguy.docker`), указав её в зависимостях вашей роли в `meta/main.yml`.

1. Инициализируйте структуру через `ansible-galaxy init roles/deploy_docker_app`.
2. В `defaults/main.yml` определите порт (`deploy_docker_app_port: 8080`) и имя образа. Использование префикса `deploy_docker_app_` гарантирует отсутствие конфликтов с другими ролями.
3. Создайте шаблон `docker-compose.yml.j2`, использующий эти переменные.
4. В `tasks/main.yml` напишите задачи: создание папки проекта, копирование шаблона и запуск приложения через модуль `community.docker.docker_compose_v2`.
5. Добавьте обработчик (`handler`) для перезапуска приложения при изменении шаблона.
6. Вызовите эту роль в своем основном плейбуке, используя ключевое слово `roles:`:
```yaml
- name: Deploy application
  hosts: app_servers
  roles:
    - role: deploy_docker_app
      vars:
        deploy_docker_app_port: 80
```
7. Убедитесь, что роль работает корректно.

#### Справочная реализация (Reference Implementation)

Если у вас возникли сложности, используйте этот код как образец.

**`defaults/main.yml`**
```yaml
---
deploy_docker_app_port: 8080
deploy_docker_app_image: nginx:latest
deploy_docker_app_project_name: my_app
deploy_docker_app_dest: "/opt/{{ deploy_docker_app_project_name }}"
```

**`templates/docker-compose.yml.j2`**
```yaml
version: '3.8'
services:
  web:
    image: "{{ deploy_docker_app_image }}"
    ports:
      - "{{ deploy_docker_app_port }}:80"
    restart: always
```

**`tasks/main.yml`**
```yaml
---
- name: Create project directory
  file:
    path: "{{ deploy_docker_app_dest }}"
    state: directory
    mode: '0755'

- name: Copy docker-compose template
  template:
    src: docker-compose.yml.j2
    dest: "{{ deploy_docker_app_dest }}/docker-compose.yml"
    mode: '0644'
  notify: Restart application

- name: Deploy application with Docker Compose
  community.docker.docker_compose_v2:
    project_src: "{{ deploy_docker_app_dest }}"
    state: present
    pull: always
```

**`handlers/main.yml`**
```yaml
---
- name: Restart application
  community.docker.docker_compose_v2:
    project_src: "{{ deploy_docker_app_dest }}"
    state: restarted
```
