# Приватный Docker Registry: храним образы безопасно

В мире современной контейнеризации Docker стал стандартом де-факто для упаковки и доставки приложений. Однако создание Docker-образа — это только половина дела. Нам нужно место, где эти образы будут храниться, индексироваться и откуда они будут доставляться на серверы продакшена, стейджинга или компьютеры разработчиков. Это место называется **Docker Registry** (реестр образов).

Для многих путь начинается с Docker Hub — публичного облачного хранилища. Но по мере роста проекта, усложнения требований к безопасности и производительности, стандартных возможностей Docker Hub становится недостаточно. В этой статье мы разберем, зачем нужен собственный реестр, как развернуть его с нуля, настроить безопасность и интегрировать в CI/CD пайплайны.

---

## Почему не хватает Docker Hub

Docker Hub — отличный инструмент для open-source проектов и быстрого старта. Однако для бизнеса он несет ряд ограничений, которые со временем становятся критическими.

### 1. Безопасность и сетевая изоляция
Использование публичного реестра означает, что ваши образы (даже если они помечены как `private`) находятся на сторонней платформе. В некоторых отраслях (финтех, медицина, госсектор) это недопустимо по соображениям комплаенса. Приватный реестр позволяет хранить образы внутри защищенного периметра компании (VPC/VPN), исключая доступ из внешнего интернета. Это минимизирует риски утечки проприетарного кода или конфигураций, которые могут случайно оказаться внутри образа.

### 2. Производительность и пропускная способность
Каждый `docker pull` из Docker Hub расходует внешний трафик и зависит от скорости вашего интернет-канала. Если у вас кластер из сотен нод, которые одновременно пытаются обновить тяжелый образ (например, на базе Java или Python), это может привести к серьезной задержке деплоя. Локальный реестр в той же сети обеспечивает гигабитную скорость загрузки, что особенно важно для автоскейлинга и быстрого восстановления после сбоев.

### 3. Лимиты и гарантированная доступность
Docker Hub ввел строгие лимиты на количество скачиваний для бесплатных и даже некоторых платных аккаунтов (Rate Limiting). Если ваш CI/CD пайплайн внезапно встанет из-за ошибки `Too Many Requests`, это может парализовать разработку на несколько часов. Собственный реестр дает вам полный контроль: вы сами решаете, сколько образов хранить и как часто их скачивать.

### 4. Контроль жизненного цикла
В Docker Hub сложно настроить сложные правила удаления старых образов. В своем реестре вы можете реализовать любые политики очистки: например, оставлять только последние 10 версий для каждой ветки или удалять образы, которые не скачивались более месяца.

## Разворачиваем registry:2 на своём сервере

Самый простой способ запустить собственный реестр — использовать официальный образ от Docker под названием `registry`. Это легковесное приложение на Go, которое реализует спецификацию Docker Registry HTTP API V2.

### Внутреннее устройство реестра
Прежде чем переходить к практике, важно понимать, как реестр хранит данные. Образ в Docker состоит из **манифеста** (JSON-файл с описанием слоев и конфигурации) и самих **слоев** (тарболы с файлами). Реестр хранит их как "блубы" (blobs). Когда вы делаете `push`, Docker сначала отправляет слои, которых еще нет в реестре, а затем — манифест.

### Подготовка и структура проекта
Для работы нам понадобится Linux-сервер с Docker и Docker Compose. Создадим рабочую директорию:

```bash
mkdir -p ~/docker-registry/{data,auth,certs}
cd ~/docker-registry
```

### Конфигурация через Docker Compose
Файл `docker-compose.yml` — это фундамент нашего сервиса. Мы будем использовать версию API 2.0.

```yaml
version: '3.8'

services:
  registry:
    image: registry:2
    container_name: private-registry
    restart: always
    ports:
      - "443:5000"
    environment:
      # Настройка аутентификации через htpasswd
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: "Registry Realm"
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd
      
      # Настройка TLS (HTTPS) — обязательна для безопасности
      REGISTRY_HTTP_TLS_CERTIFICATE: /certs/domain.crt
      REGISTRY_HTTP_TLS_KEY: /certs/domain.key
      
      # Параметры хранилища
      REGISTRY_STORAGE_FILESYSTEM_ROOTDIRECTORY: /var/lib/registry
      # Включаем возможность удаления образов через API
      REGISTRY_STORAGE_DELETE_ENABLED: "true"

      # Настройка CORS для работы Web UI
      REGISTRY_HTTP_CORS_ORIGINS: "['http://localhost:8080', 'http://myregistry.local:8080']"
      REGISTRY_HTTP_CORS_METHODS: "['GET', 'OPTIONS', 'DELETE']"
      REGISTRY_HTTP_CORS_HEADERS: "['Authorization', 'Content-Type']"
    volumes:
      - ./data:/var/lib/registry
      - ./auth:/auth
      - ./certs:/certs
```

В этой конфигурации мы пробрасываем порт 443. Параметр `REGISTRY_STORAGE_DELETE_ENABLED` критически важен, если вы планируете в будущем чистить реестр от старых образов.

## Настройка TLS и аутентификации

Docker-клиент по умолчанию требует HTTPS. Работа по HTTP (insecure-registry) возможна, но требует правки конфигов на каждой машине-клиенте, что неудобно и небезопасно.

### Шаг 1: Настройка Basic Auth
Мы будем использовать механизм `htpasswd`. Это простой способ защитить доступ логином и паролем.

```bash
# Используем образ httpd (Apache) для генерации файла
docker run --entrypoint htpasswd httpd:2.4 -Bbn admin my_secure_password > auth/htpasswd
```

### Шаг 2: TLS-сертификаты
Для продакшена лучше всего использовать сертификаты от **Let's Encrypt** (через Certbot) или корпоративного CA. Для демонстрации создадим самоподписанный сертификат:

```bash
openssl req -newkey rsa:4096 -nodes -sha256 -keyout certs/domain.key \
  -x509 -days 365 -out certs/domain.crt \
  -subj "/CN=myregistry.local"
```

**Нюанс с доверием и DNS:**

1.  **DNS:** Так как мы используем вымышленный домен `myregistry.local`, добавьте его в файл `/etc/hosts` на клиентской машине (если вы запускаете Docker на том же хосте, укажите `127.0.0.1`):
    ```bash
    echo "127.0.0.1 myregistry.local" | sudo tee -a /etc/hosts
    ```
2.  **Доверие в Linux:** Если вы используете самоподписанный сертификат, Docker-клиент выдаст ошибку `x509: certificate signed by unknown authority`. Чтобы это исправить, создайте директорию и скопируйте сертификат:
    ```bash
    sudo mkdir -p /etc/docker/certs.d/myregistry.local:443
    sudo cp certs/domain.crt /etc/docker/certs.d/myregistry.local:443/ca.crt
    ```
    После этого перезапустите Docker (`sudo systemctl restart docker`).

3.  **Для Windows и macOS (Docker Desktop):**
    В Docker Desktop процесс отличается. Вам нужно добавить `domain.crt` в системное хранилище доверенных корневых сертификатов:
    - **macOS:** Откройте «Связку ключей» (Keychain Access), перетащите `domain.crt` в раздел «Система» и в свойствах выберите «Всегда доверять» (Always Trust).
    - **Windows:** Нажмите правой кнопкой на `domain.crt` -> «Установить сертификат» -> «Локальный компьютер» -> Поместить в хранилище «Доверенные корневые центры сертификации».

### Запуск и проверка
```bash
docker compose up -d
docker ps
```
Теперь реестр слушает порт 443 и готов принимать запросы.

## Push и pull из приватного реестра

Процесс взаимодействия с приватным реестром немного отличается от работы с Docker Hub из-за необходимости указывать адрес хоста в имени образа.

### 1. Авторизация
```bash
docker login myregistry.local:443 -u admin
```
Система предложит ввести пароль. После успешного входа Docker создаст запись в `~/.docker/config.json`, которая будет использоваться при последующих запросах.

### 2. Тегирование (Naming Convention)
Имя образа должно явно указывать на реестр. Формат: `<REGISTRYHOST>:<PORT>/<REPOSITORY>:<TAG>`.

```bash
# Собираем образ
docker build -t backend-api:1.2.0 .

# Добавляем тег для нашего реестра
docker tag backend-api:1.2.0 myregistry.local:443/backend-api:1.2.0
```

### 3. Загрузка и скачивание
```bash
# Push в реестр
docker push myregistry.local:443/backend-api:1.2.0

# Pull на другом сервере
docker pull myregistry.local:443/backend-api:1.2.0
```

### Добавляем графический интерфейс (Web UI)
Сам по себе `registry:2` не имеет веб-интерфейса, что может быть непривычно после Docker Hub. Чтобы просматривать список образов и тегов через браузер, можно добавить легковесный UI, например **joxit/docker-registry-ui**. Просто добавьте еще один сервис в ваш `docker-compose.yml`:

```yaml
  ui:
    image: joxit/docker-registry-ui:latest
    ports:
      - "8080:80"
    environment:
      - REGISTRY_URL=https://myregistry.local
      - DELETE_IMAGES=true
```

### Как это выглядит "под капотом"?
Когда вы делаете `push`, Docker отправляет HEAD-запросы для каждого слоя. Если реестр отвечает `200 OK`, слой пропускается. Если `404 Not Found`, слой сжимается и передается по HTTP. Манифест отправляется последним, фиксируя целостность версии.

## Harbor — enterprise-grade реестр

Стандартный `registry:2` отлично подходит для хранения, но в нем нет графического интерфейса, аудита и продвинутой безопасности. Здесь на сцену выходит **Harbor**.

Harbor — это Open Source проект (CNCF Graduated), который надстраивается над обычным Docker Registry и добавляет слой управления.

### Почему Harbor лучше для компаний?
1.  **Ролевая модель (RBAC):** Вы можете интегрировать Harbor с LDAP или Active Directory. Разработчики получают доступ только к своим проектам.
2.  **Сканирование на уязвимости:** Harbor включает в себя сканеры (Trivy, Clair). Каждый раз, когда образ пушится, он проверяется на наличие известных CVE. Вы можете запретить `pull` образа, если уровень его уязвимостей выше "Medium".
3.  **Content Trust (Notary):** Позволяет подписывать образы цифровой подписью. Это гарантирует, что в продакшн попадет именно тот код, который прошел тесты, и его никто не подменил по пути.
4.  **Репликация:** Вы можете настроить автоматическую синхронизацию между дата-центрами. Например, образы пушатся в московский офис, а затем сами "переезжают" в филиал в Новосибирске.
5.  **Helm Chart Repository:** Harbor умеет хранить не только контейнеры, но и Helm-чарты, становясь единой точкой правды для Kubernetes-инфраструктуры.

### Архитектура Harbor
Harbor состоит из нескольких компонентов (Core, Job Service, Log, DB, Redis, Registry). Его установка сложнее и обычно требует Docker Compose или Helm-чарта в K8s.

## Облачные реестры: когда что выбирать

Если вы не хотите тратить время на администрирование патчей и бэкапов, Managed-решения — ваш выбор.

### 1. AWS Elastic Container Registry (ECR)
**Плюсы:** Бесшовная интеграция с IAM-ролями. Если ваш сервис запущен в ECS или EKS, ему не нужен пароль — он авторизуется через права роли сервера.
**Фишка:** Lifecycle Policies позволяют очень гибко настраивать удаление старых образов (например, "удалять все теги `*-dev`, если их больше 50").

### 2. GitHub Container Registry (GHCR)
**Плюсы:** Идеально, если ваш код живет на GitHub. Образы привязываются к репозиториям и отображаются прямо в интерфейсе GitHub.
**Цена:** Бесплатно для публичных проектов, для приватных — оплата за хранение.

### 3. GitLab Container Registry
**Плюсы:** Если вы используете self-hosted GitLab, реестр уже там есть (его нужно просто включить в конфиге).
**Интеграция:** В пайплайнах GitLab CI переменные `$CI_REGISTRY_IMAGE` и `$CI_JOB_TOKEN` позволяют пушить образы без настройки секретов.

### Сводная таблица сравнения

| Характеристика | registry:2 | Harbor | AWS ECR | GHCR / GitLab |
| :--- | :--- | :--- | :--- | :--- |
| **Сложность** | Низкая | Высокая | Средняя | Низкая |
| **Цена** | Свой сервер | Свой сервер | По потреблению | По потреблению |
| **Сканирование** | Нет | Да (встроено) | Да (платное) | Да |
| **UI** | Нет | Отличный | AWS Console | GitHub/GitLab UI |
| **RBAC** | Базовый | Продвинутый | Очень гибкий | Хороший |

## Интеграция в CI/CD (GitHub Actions, GitLab CI)

Реестр бесполезен, если он не включен в процесс автоматической сборки.

### GitHub Actions: Полный цикл
Ниже пример рабочего процесса, который собирает образ, тегирует его по SHA коммита и отправляет в GitHub Packages.

```yaml
name: CI/CD Pipeline
on:
  push:
    branches: [ "main" ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v3

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v2

      - name: Login to GHCR
        uses: docker/login-action@v2
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and Push
        uses: docker/build-push-action@v4
        with:
          context: .
          push: true
          # Используем несколько тегов для удобства
          tags: |
            ghcr.io/${{ github.repository }}/app:latest
            ghcr.io/${{ github.repository }}/app:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

### GitLab CI: Лаконичность
В GitLab всё еще проще благодаря встроенным переменным.

```yaml
build:
  image: docker:24.0.5
  services:
    - docker:dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
```

## Лучшие практики и антипаттерны

За годы работы с Docker сообщество выработало набор правил, которые уберегут вас от потери данных и проблем с безопасностью.

### 1. Неизменяемость (Immutability)
Никогда не используйте тег `latest` в Dockerfile или Kubernetes-манифестах для продакшена. `latest` — это указатель, который меняется. Если вы задеплоили `latest`, и через час кто-то запушил новую версию с тем же тегом, вы не сможете гарантировать, что новые ноды в кластере запустят ту же версию кода. Всегда используйте теги с версией или Git SHA.

### 2. Принцип минимальных привилегий
Для CI/CD создавайте отдельные учетные записи (Robot Accounts) с правами только на `push`, а для серверов продакшена — только на `pull`. Никогда не используйте основной пароль администратора в пайплайнах.

### 3. Очистка слоев (Garbage Collection)
Docker Registry не удаляет данные с диска сразу после того, как вы удалили манифест образа. Удаляются только ссылки. Чтобы реально освободить место, нужно запустить процесс сборки мусора:
```bash
docker exec private-registry bin/registry garbage-collect /etc/docker/registry/config.yml
```
**Внимание:** Во время GC реестр лучше перевести в режим `read-only`, чтобы избежать повреждения данных.

### 4. Многоархитектурные образы (Multi-arch)
Если ваши разработчики сидят на MacBook с чипами M1/M2 (ARM), а серверы работают на Intel (x86_64), используйте `docker buildx` для сборки манифестов, поддерживающих обе архитектуры. Реестр умеет хранить один тег, за которым скрываются разные бинарники для разных процессоров.

### 5. Резервное копирование
Регулярно делайте бэкап директорий `./data` (сами образы и их метаданные) и `./auth` (файлы с паролями). Если вы потеряете содержимое `./auth`, вы не сможете авторизоваться, а без манифестов из `./data` восстановить образы будет невозможно, даже если слои физически остались на диске.

### Антипаттерны
*   ❌ **Отсутствие бэкапа метаданных:** Даже если вы бэкапите файлы слоев, без манифестов вы не сможете их восстановить.
*   ❌ **Bloat Images:** Включать в образы исходный код, компиляторы и кэши менеджеров пакетов. Используйте Multi-stage сборку, чтобы в реестр попадал только готовый артефакт.
*   ❌ **Игнорирование CVE:** Хранить образы годами без пересборки. Даже если ваш код не менялся, зависимости в базовом образе (OpenSSL, libc) могут содержать критические уязвимости.

## Заключение

Приватный Docker Registry — это не просто "файлопомойка" для контейнеров, а центральный узел вашей системы поставки ПО. 

Для старта и небольших команд идеальным выбором будут **Managed Cloud Registries** (ECR, GHCR). Они снимают головную боль с обслуживанием. 
Если вам важен полный суверенитет над данными и сетевая изоляция — разворачивайте **registry:2**. 
Для крупного бизнеса, где вопросы безопасности, аудита и ролевого доступа стоят на первом месте, золотым стандартом является **Harbor**.

Главное — помните: автоматизация сборки, использование уникальных тегов и регулярная очистка хранилища сделают вашу работу с Docker приятной и предсказуемой.
