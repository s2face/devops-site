# Docker Compose: управляем несколькими контейнерами

## Проблема ручного запуска контейнерного стека

В начале пути каждого DevOps-инженера или разработчика, осваивающего контейнеризацию, стоит Docker. Вы учитесь собирать образы через `Dockerfile`, запускать их командой `docker run`, пробрасывать порты и монтировать тома. Это кажется простым и понятным, пока ваше приложение состоит из одного исполняемого файла.

Однако современная микросервисная архитектура диктует свои правила. Типичное веб-приложение сегодня — это не просто монолит. Это сложная экосистема, включающая:
- Веб-сервер (Nginx или Apache).
- Бэкенд на Python (Flask/Django), Node.js или Go.
- Основную реляционную базу данных (PostgreSQL, MySQL).
- Быстрое кеширующее хранилище (Redis, Memcached).
- Очереди сообщений (RabbitMQ, Kafka).
- Системы мониторинга и логирования (Prometheus, Grafana, ELK).

Попробуйте запустить всё это вручную. Вам придется:
1. Создать общую сеть Docker (`docker network create`).
2. Запустить контейнер с базой данных, не забыв про переменные окружения и тома для данных.
3. Дождаться, пока БД инициализируется.
4. Запустить Redis.
5. Запустить ваше приложение, передав ему адреса БД и Redis.
6. Запустить Nginx с правильным конфигурационным файлом, указывающим на ваше приложение.

Если вы ошибетесь в одном параметре или забудете порядок запуска, вся система развалится. А если нужно передать проект коллеге? Ему придется повторить все эти магические пассы в консоли. Именно здесь на сцену выходит **Docker Compose** — инструмент для определения и запуска многоконтейнерных Docker-приложений. С его помощью вы описываете всю инфраструктуру в одном YAML-файле и управляете ею одной командой.

Это не просто "скрипт запуска". Это декларативный подход: вы описываете *желаемое состояние* системы, а Docker Compose берет на себя всю грязную работу по его достижению.

## Анатомия docker-compose.yml

Файл `docker-compose.yml` — это манифест вашей инфраструктуры. Он позволяет декларативно описать, какие сервисы должны быть запущены, как они связаны между собой и какими ресурсами они обладают. Давайте разберем ключевые блоки и директивы, которые составляют костяк любого проекта.

### version
Хотя в современных версиях Docker Compose (V2, который встроен в Docker Desktop и устанавливается как плагин в Linux) указание версии стало необязательным, в старых проектах вы часто встретите `version: '3.8'`. Это определяет схему синтаксиса. Если вы используете последнюю версию Docker, можете опустить эту строку, но для совместимости со старыми инструментами её часто оставляют.

### services
Это основной раздел, где описываются ваши контейнеры. Каждый подраздел внутри `services` — это отдельный контейнер, который Compose превращает в рабочий инстанс.

#### build и image
Существует два основных способа запустить сервис:
1. **image**: Использовать готовый образ из реестра (Docker Hub). Например, `image: postgres:15-alpine`. Это идеально для стандартных инструментов (БД, кеши, прокси), где вам не нужно менять внутреннюю логику.
2. **build**: Собрать собственный образ из локальных исходников.
   ```yaml
   flask:
     build:
       context: ./app
       dockerfile: Dockerfile
       args:
         - APP_ENV=production
   ```
   Здесь `context` — это папка с исходниками, а `args` позволяют передавать переменные прямо в процесс сборки (в `ARG` внутри Dockerfile).

#### ports
Директива для проброса портов в формате `ХОСТ:КОНТЕЙНЕР`. Например, `8080:80` означает, что запросы на порт 8080 вашего компьютера будут перенаправлены на порт 80 внутри контейнера.
Важно понимать: если вы указываете только порт контейнера (например, `"80"`), Docker назначит случайный свободный порт на хосте.

#### volumes
Тома (Volumes) критически важны для сохранения данных. Контейнеры по своей природе эфемерны: если вы удалите контейнер с БД без тома, все данные пропадут.
- **Именованные тома** (`postgres_data:/var/lib/postgresql/data`): Управляются самим Docker. Они хранятся в специальной директории (в Linux это обычно `/var/lib/docker/volumes`). *Обратите внимание: в Docker Desktop (Windows/macOS) эти пути скрыты внутри виртуальной машины и недоступны напрямую через проводник.* Это лучший выбор для баз данных в продакшене.
- **Bind mounts** (`./app:/app`): Прямое монтирование папки с исходным кодом в контейнер. Любое изменение файла на хосте мгновенно отображается в контейнере. В связке с режимом отладки (например, Flask Debug Mode) это позволяет приложению автоматически перезапускаться при изменении кода — это и есть основа "горячей перезагрузки" (Hot Reload).

#### networks
Docker Compose автоматически создает сеть для вашего проекта (обычно с драйвером `bridge`). В этой сети контейнеры могут общаться друг с другом по именам сервисов. Например, Flask-приложение может просто обратиться к `http://db:5432`.
Вы можете создавать свои сети:
```yaml
networks:
  frontend:
  backend:
```
Это позволяет изолировать сервисы: например, база данных может быть доступна только в сети `backend`, а Nginx — и в `frontend`, и в `backend`.

#### environment
Здесь задаются переменные окружения. Это "пульт управления" приложением. Вместо того чтобы жестко прописывать пароли в коде, вы передаете их через этот блок.
- Прямое указание: `DEBUG: "true"`
- Из файла `.env`: `POSTGRES_PASSWORD: ${DB_PASS}`

#### depends_on
Определяет порядок запуска. Но будьте осторожны: по умолчанию `depends_on` гарантирует только то, что контейнер с БД *стартовал*, но не то, что база данных готова принимать подключения. Для реальной проверки готовности используется связка с `healthcheck`.

#### healthcheck
Это механизм самодиагностики контейнера. Контейнер может быть запущен, но "зависнуть" или еще не инициализироваться.
```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U postgres"]
  interval: 10s
  timeout: 5s
  retries: 5
```
Если проверка проваливается 5 раз подряд, контейнер помечается как `unhealthy`.

#### restart
Политика перезапуска определяет поведение Docker при падении контейнера:
- `no`: Не перезапускать.
- `always`: Всегда перезапускать (даже после перезагрузки ОС).
- `on-failure`: Перезапускать только при ошибке (код выхода не 0).
- `unless-stopped`: Перезапускать всегда, кроме случаев ручной остановки.

## Практика: Разбор полного стека (Nginx + Flask + PostgreSQL + Redis)

Перейдем от теории к реальному проекту. Представим, что нам нужно развернуть веб-приложение, которое считает количество посещений в Redis и хранит версию базы данных PostgreSQL. Перед ним стоит Nginx в качестве обратного прокси.

### Структура проекта

Для начала организуем файлы:
```text
.
├── app/
│   ├── app.py           # Логика приложения
│   ├── Dockerfile       # Как собрать образ приложения
│   └── requirements.txt # Зависимости Python
├── nginx/
│   └── nginx.conf       # Настройки прокси
├── .env                 # Секретные данные
├── docker-compose.yml   # Главный оркестратор
└── docker-compose.override.yml # Настройки для разработки
```

### 1. Бэкенд: Python Flask

Само приложение (`app/app.py`) взаимодействует с двумя базами. Мы используем библиотеку `redis` для инкремента счетчика и `psycopg2-binary` для работы с PostgreSQL.

```python
import os
import time
import redis
import psycopg2
from flask import Flask, jsonify

app = Flask(__name__)

# Настройка Redis: используем имя сервиса 'redis' как хост
# Docker Compose сам разрешит имя 'redis' в IP-адрес нужного контейнера
cache = redis.Redis(host='redis', port=6379)

def get_hit_count():
    retries = 5
    while True:
        try:
            return cache.incr('hits')
        except redis.exceptions.ConnectionError as exc:
            if retries == 0:
                raise exc
            retries -= 1
            time.sleep(0.5)

# Настройка Postgres: данные берем из переменных окружения
def get_db_connection():
    conn = psycopg2.connect(
        host='db', # имя сервиса в Docker Compose
        database=os.getenv('POSTGRES_DB'),
        user=os.getenv('POSTGRES_USER'),
        password=os.getenv('POSTGRES_PASSWORD')
    )
    return conn

@app.route('/')
def hello():
    count = get_hit_count()
    conn = get_db_connection()
    cur = conn.cursor()
    cur.execute('SELECT version();')
    db_version = cur.fetchone()
    cur.close()
    conn.close()
    return jsonify({
        "message": "Hello from Flask!",
        "hits": count,
        "db_version": db_version[0]
    })

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
```

Чтобы упаковать это в контейнер, используем `app/Dockerfile`:
```dockerfile
FROM python:3.9-slim
ARG APP_ENV
WORKDIR /app
COPY requirements.txt .
# Устанавливаем зависимости. Библиотека psycopg2-binary нужна для работы с Postgres
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
# Запускаем приложение
CMD ["python", "app.py"]
```

Содержимое `app/requirements.txt`:
```text
flask
redis
psycopg2-binary
```

### 2. Веб-сервер: Nginx

Nginx будет принимать внешние запросы на порту 80 и перенаправлять их на Flask (порт 5000). Файл `nginx/nginx.conf`:
```nginx
events {}
http {
    upstream flask_app {
        server flask:5000; # Обращение по имени сервиса в сети Docker
    }
    server {
        listen 80;
        location / {
            proxy_pass http://flask_app;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
}
```

### 3. Главный файл: docker-compose.yml

Теперь соберем всё воедино. Мы добавим проверки здоровья, чтобы система была отказоустойчивой.

```yaml
version: '3.8'

services:
  # База данных PostgreSQL
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 5s
      timeout: 5s
      retries: 5
    restart: always

  # Кеш Redis
  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5
    restart: always

  # Наше Python приложение
  flask:
    build:
      context: ./app
      args:
        - APP_ENV=production
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    # Ждем не просто запуска, а успешного прохождения проверок здоровья БД и Redis
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    restart: on-failure

  # Прокси-сервер
  nginx:
    image: nginx:alpine
    ports:
      - "8080:80"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - flask
    restart: always

volumes:
  postgres_data:
```

### Почему такая конфигурация важна?

Использование `depends_on` с `condition: service_healthy` решает одну из самых частых проблем многоконтейнерных систем — гонку состояний. Если Flask попытается подключиться к базе, которая еще только создает свои таблицы и индексы, он упадет с ошибкой "Connection Refused". Благодаря нашей настройке, Docker Compose придержит запуск Flask до тех пор, пока `pg_isready` внутри контейнера `db` не вернет статус "готов". Это делает запуск системы предсказуемым и надежным.

## Управление стеком через CLI

Знание командной строки Docker Compose — это 50% успеха. Все команды должны выполняться из директории, где лежит `docker-compose.yml`.

### up — запуск системы
Команда `docker compose up` читает файл конфигурации, скачивает образы, собирает ваши файлы Dockerfile и запускает контейнеры.
- `-d` (detached): Запускает всё в фоновом режиме. Консоль остается свободной.
- `--build`: Заставляет Compose пересобрать образы. Полезно, если вы внесли изменения в код.
- `docker compose up -d --scale flask=3`: Позволяет запустить 3 экземпляра сервиса Flask. Docker использует внутренний DNS для балансировки запросов (round-robin) между несколькими экземплярами сервиса. Это требует правильной настройки балансировщика в Nginx.

### down — полная остановка
Команда `docker compose down` останавливает и **удаляет** контейнеры и сети, созданные проектом.
- `-v`: Удаляет также и тома (volumes). Осторожно: это удалит все данные в PostgreSQL!
- `--rmi all`: Удаляет также и собранные образы.

### logs — диагностика
Если что-то не работает, используйте `docker compose logs`.
- `-f` (follow): Позволяет смотреть логи в реальном времени (аналог `tail -f`).
- `--tail=100`: Показать только последние 100 строк.

### ps — состояние сервисов
Команда `docker compose ps` показывает список контейнеров проекта, их текущий статус, используемые порты и связанные сервисы.

### exec и run — выполнение команд
- `docker compose exec [service] [command]`: Выполняет команду в уже **запущенном** контейнере.
  Пример: `docker compose exec db psql -U admin -d dev_db`.
- `docker compose run [service] [command]`: Создает **новый временный** контейнер для выполнения команды. Часто используется для миграций БД или запуска тестов.

### Другие полезные команды
- `docker compose config`: Проверяет синтаксис файла и показывает итоговую конфигурацию со всеми подставленными переменными.
- `docker compose top`: Показывает запущенные процессы внутри контейнеров (аналог `htop`).
- `docker compose images`: Список образов, используемых проектом, с указанием их размера.

## .env файл — внешние переменные

Переменные окружения не стоит хранить прямо в `docker-compose.yml`, особенно если это пароли. Создайте файл `.env` в корне проекта:
```env
POSTGRES_USER=admin
POSTGRES_PASSWORD=secret_pass
POSTGRES_DB=dev_db
```
Compose автоматически подставит эти значения в переменные типа `${VARIABLE_NAME}`. Это позволяет безопасно хранить `docker-compose.yml` в Git (без секретов), а `.env` добавлять в `.gitignore`.

## Override файлы: dev vs production

Если в директории рядом с `docker-compose.yml` лежит файл `docker-compose.override.yml`, Compose автоматически объединит их при запуске. Это стандартный механизм для переопределения настроек под локальную разработку.

Например, в `docker-compose.override.yml` можно добавить проброс локального кода для Hot Reload:
```yaml
services:
  flask:
    volumes:
      - ./app:/app
    environment:
      FLASK_ENV: development
      DEBUG: "true"
```
В продакшене вы просто не копируете этот файл, и приложение работает в изолированном режиме. Вы также можете создавать специфичные файлы, например `docker-compose.prod.yml`, и запускать их командой:
`docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`

## Глубокое погружение: Сети и Тома

### Сети (Networking)
Когда вы запускаете проект, Docker Compose создает сеть типа `bridge` под названием `projectname_default`. 
- **DNS**: Внутри этой сети работает встроенный DNS-сервер. Имена сервисов (`db`, `redis`, `flask`) становятся именами хостов.
- **Изоляция**: Если у вас запущено два разных проекта Docker Compose, они будут находиться в разных сетях и не увидят друг друга, даже если у обоих сервисы называются `db`. Это предотвращает конфликты портов и данных.

### Тома (Volumes)
Тома в Docker бывают трех типов:
1. **Anonymous Volumes**: Создаются, если вы просто указали путь в контейнере (например, `- /var/lib/mysql`). Docker даст им случайное имя. Они удаляются при `down -v`.
2. **Named Volumes**: Имеют четкое имя (`postgres_data`). Docker сохраняет их даже после удаления проекта. Чтобы удалить их, нужно явно выполнить `docker volume rm`.
3. **Bind Mounts**: Прямая ссылка на папку на хосте. Они не управляются Docker, и если вы удалите папку на компьютере, контейнер сломается.

## Лучшие практики использования Docker Compose

1. **Используйте легкие образы**: Выбирайте теги с суффиксом `-alpine` (например, `node:18-alpine` или `python:3.9-slim`). Они весят в 5-10 раз меньше обычных образов, что ускоряет сборку и деплой.
2. **Явно указывайте версии**: Никогда не используйте тег `latest` в продакшене. Завтра `latest` может обновиться и сломать совместимость. Всегда пишите `postgres:15.3`.
3. **Не запускайте контейнеры от root**: В `Dockerfile` создавайте пользователя и переключайтесь на него через директиву `USER`. Это значительно повышает безопасность.
4. **Используйте .dockerignore**: Не копируйте в образ папку `.git`, `node_modules` или `__pycache__`. Это сделает образ чище и легче.
5. **Разделяйте конфигурации**: Держите базовый `docker-compose.yml` простым, а специфичные настройки (порты для разработки, мониторинг) выносите в файлы переопределения.

## Заключение и выводы

Docker Compose — это мощный инструмент, который превращает хаос из десятков команд `docker run` в стройный, воспроизводимый и понятный код. Он позволяет:
1. **Стандартизировать среду**: У всех разработчиков в команде проект запускается одинаково.
2. **Упростить CI/CD**: Тесты в Jenkins или GitHub Actions запускаются одной командой `up`.
3. **Управлять зависимостями**: Забыть о конфликтах версий баз данных или библиотек на хост-машине.

Однако стоит помнить, что Docker Compose предназначен в первую очередь для локальной разработки, тестирования и небольших односерверных инсталляций. Если ваше приложение начинает требовать масштабирования на сотни серверов, автоматического восстановления после падения оборудования и сложной оркестрации ресурсов — пора смотреть в сторону Kubernetes.

Но даже в мире Kubernetes Docker Compose остается незаменимым инструментом для локальной отладки. Начните с описания своего стека сегодня, и вы больше никогда не захотите возвращаться к ручному управлению контейнерами. Docker Compose — это мост между простым запуском кода и настоящим DevOps-подходом к инфраструктуре.
