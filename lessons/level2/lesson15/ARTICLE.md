# ELK Stack: агрегация и анализ логов для DevOps

В современной микросервисной архитектуре, где количество контейнеров может исчисляться сотнями, а запросы проходят через десятки узлов, вопрос «что пошло не так?» становится критическим. Логи — это «черный ящик» вашей системы. Если они разбросаны по разным серверам, поиск причины сбоя превращается в детективное расследование с SSH, `grep` и `tail -f`, которое может длиться часами.

В этой статье мы разберем, как построить надежную систему централизованного логирования на базе **ELK Stack** (Elasticsearch, Logstash, Kibana) с использованием **Filebeat**, научимся правильно готовить логи и обсудим, когда стоит выбрать альтернативу в виде Grafana Loki.

## Проблема «где искать ошибки?»

Представьте типичную ситуацию: пользователь сообщает об ошибке «500 Internal Server Error» при попытке оформить заказ. Ваше приложение запущено в Kubernetes или Docker Swarm на 10 разных нодах. 

Без централизованного логирования ваш алгоритм действий выглядит так:
1. Зайти по SSH на первую ноду.
2. Найти нужный контейнер.
3. Посмотреть логи через `docker logs` или прочитать файл в `/var/log`.
4. Не найти ничего подозрительного.
5. Повторить для остальных 9 нод.

Этот подход называется **«SSH-driven debugging»**, и это главный антипаттерн современной эксплуатации. Он не масштабируется, он медленный, и он не позволяет коррелировать события. Если ошибка вызвана цепочкой вызовов между тремя сервисами, собрать полную картину вручную практически невозможно.

**Централизованное логирование решает три основные задачи:**
*   **Агрегация:** Сбор логов изо всех источников (файлы, базы данных, сетевые устройства) в единое хранилище.
*   **Индексация и поиск:** Возможность мгновенно найти записи по ID пользователя, Trace ID или коду ошибки среди терабайтов данных.
*   **Визуализация и мониторинг:** Построение графиков частоты ошибок, времени ответа и дашбордов для оперативного контроля состояния системы.

## Архитектура ELK Stack

Классический стек ELK (Elasticsearch, Logstash, Kibana) сегодня чаще называют **Elastic Stack**, так как в цепочку добавился четвертый важный компонент — **Beats**.

Стандартный путь лога в современном DevOps-пайплайне выглядит следующим образом:
**Приложение → Filebeat → Logstash → Elasticsearch → Kibana**

Разберем роль каждого компонента подробнее:

### 1. Filebeat (Lightweight Shipper)
Это легкий агент, написанный на Go, который устанавливается на каждый сервер (или запускается как Sidecar-контейнер). Его задача — следить за файлами логов, считывать новые строки и максимально быстро отправлять их дальше. 
**Преимущества:**
*   Потребляет минимум ресурсов (в отличие от тяжелого Logstash на Java).
*   Обладает механизмом *Backpressure sensitivity*: если Logstash занят, Filebeat замедляет чтение, чтобы не перегрузить сеть и приемник.
*   Умеет запоминать позицию чтения (регистр), поэтому при перезагрузке он не будет читать файл заново и не потеряет данные.

### 2. Logstash (Processing Pipeline)
«Тяжелая артиллерия» для обработки данных. Он принимает логи от Filebeat, фильтрует их, парсит и обогащает.
**Ключевые возможности:**
*   **Grok-фильтры:** превращение неструктурированного текста в JSON.
*   **Мутации:** переименование полей, приведение типов (строка в число), удаление конфиденциальной информации.
*   **Обогащение:** добавление GeoIP данных, поиск информации в базах данных или кэше Redis в процессе обработки лога.

### 3. Elasticsearch (Storage & Search Engine)
Сердце системы. Это распределенная NoSQL база данных на базе движка Lucene. Она хранит логи в виде JSON-документов и строит по ним полнотекстовый индекс. Именно благодаря Elasticsearch мы можем искать по огромным массивам данных за доли секунды.

### 4. Kibana (Visualization Layer)
Веб-интерфейс для работы с данными. Здесь инженеры пишут запросы на языке KQL, создают графики, дашборды и настраивают алерты.

## Подготовка системы

Перед запуском стека необходимо настроить хостовую машину, иначе Elasticsearch может не стартовать или работать нестабильно.

1.  **Увеличение лимитов памяти (Linux):** Elasticsearch использует `mmapfs` для хранения индексов. По умолчанию лимиты в Linux слишком малы.
    Выполните команду:
    ```bash
    sudo sysctl -w vm.max_map_count=262144
    ```
    Чтобы настройка сохранилась после перезагрузки, добавьте строку `vm.max_map_count=262144` в файл `/etc/sysctl.conf`.

2.  **Требования к ресурсам:**
    *   **RAM:** Минимум 4 ГБ свободной оперативной памяти для всего стека (ES + Kibana + Logstash). На машинах с 4 ГБ общего объема RAM возможны зависания системы.
    *   **CPU:** Минимум 2 ядра.
    *   **Docker Desktop (Windows/Mac):** Убедитесь, что в настройках Docker выделено не менее 4 ГБ RAM.

## Разворачиваем ELK через Docker Compose

Для обучения и локальной разработки удобнее всего использовать Docker Compose. Мы развернем минимальный стек версии 8.10.0. 

### Шаг 0: Подготовка конфигурационных файлов

Перед запуском Docker Compose **необходимо** создать файлы конфигурации в той же директории, где будет лежать `docker-compose.yml`. Если этого не сделать, Docker создаст пустые *директории* вместо файлов, и контейнеры не запустятся.

Создайте `filebeat.yml` и `logstash.conf` (содержимое возьмите из разделов ниже).

### Конфигурация `docker-compose.yml`

```yaml
version: '3.8'

services:
  elasticsearch:
    image: elasticsearch:8.10.0
    container_name: elasticsearch
    environment:
      - discovery.type=single-node      # Режим одного узла
      - xpack.security.enabled=false    # Отключаем TLS и авторизацию для тестов
      - "ES_JAVA_OPTS=-Xms1g -Xmx1g"    # Ограничение Heap Size — КРИТИЧЕСКИ ВАЖНО
    ulimits:
      memlock:
        soft: -1
        hard: -1
    volumes:
      - es_data:/usr/share/elasticsearch/data
    ports:
      - "9200:9200"
    networks:
      - elk

  kibana:
    image: kibana:8.10.0
    container_name: kibana
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    ports:
      - "5601:5601"
    depends_on:
      - elasticsearch
    networks:
      - elk

  logstash:
    image: logstash:8.10.0
    container_name: logstash
    volumes:
      - ./logstash.conf:/usr/share/logstash/pipeline/logstash.conf:ro
    environment:
      - "LS_JAVA_OPTS=-Xms512m -Xmx512m"
    ports:
      - "5044:5044"
    depends_on:
      - elasticsearch
    networks:
      - elk

  filebeat:
    image: elastic/filebeat:8.10.0
    container_name: filebeat
    user: root # Нужно для доступа к логам Docker
    volumes:
      - ./filebeat.yml:/usr/share/filebeat/filebeat.yml:ro
      - /var/lib/docker/containers:/var/lib/docker/containers:ro
      - /var/run/docker.sock:/var/run/docker.sock:ro
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    depends_on:
      - logstash
    networks:
      - elk

  logger:
    image: busybox
    container_name: logger-test
    command: sh -c 'while true; do echo "{\"message\": \"Test log\", \"level\": \"info\", \"user_id\": 123}"; sleep 5; done'
    networks:
      - elk

volumes:
  es_data:
    driver: local

networks:
  elk:
    driver: bridge
```

### Запуск стека

Когда файлы `docker-compose.yml`, `filebeat.yml` и `logstash.conf` созданы, выполните команду:

```bash
docker compose up -d
```

Проверьте, что все контейнеры запущены: `docker compose ps`.

### Разбор ключевых параметров:
*   **`ES_JAVA_OPTS="-Xms1g -Xmx1g"`**: Это ограничение объема памяти (heap), которую может занять Java Virtual Machine. Если этого не сделать, Elasticsearch попытается забрать всю доступную память хоста.
*   **`ulimits.memlock`**: Запрещает ОС выгружать память Elasticsearch в своп на диск. Это критично для производительности поиска.
*   **Монтирование `/var/lib/docker/containers`**: Стандартный путь, куда Docker пишет stdout/stderr логи контейнеров.

## Filebeat: собираем логи Docker-контейнеров

Filebeat настраивается через файл `filebeat.yml`. Нам нужно указать ему, какие файлы читать и куда отправлять.

### Пример `filebeat.yml`:

```yaml
filebeat.inputs:
- type: container
  paths:
    - '/var/lib/docker/containers/*/*.log' # Маска для логов всех контейнеров

processors:
  - add_docker_metadata: ~ # Магия: добавляет имя контейнера, образ и лейблы
  - add_host_metadata: ~   # Добавляет имя хоста и IP

output.logstash:
  hosts: ["logstash:5044"] # Отправляем в Logstash для обработки
```

**Зачем нам `add_docker_metadata`?**
Docker пишет логи в файлы с именами вроде `9a2b3c...-json.log`. Без этого процессора в Kibana вы не поймете, какой именно сервис выдал ошибку. Процессор сопоставляет ID контейнера с данными из Docker API и добавляет поле `container.name`.

## Structured Logging: почему JSON — это стандарт

Огромная ошибка при построении систем логирования — продолжать писать логи в формате «простого текста».

**Почему Plain Text — это проблема?**
Представьте лог: `[INFO] 2023-10-27 12:00:01 User 555 created order 1023 in 150ms`.
Для человека это понятно. Для машины — это просто строка. Чтобы извлечь из нее `user_id` или `duration`, системе нужно применить регулярное выражение (Grok-фильтр).

Grok-фильтры потребляют много CPU и ломаются при малейшем изменении формата лога.

**Решение: Structured Logging (JSON)**
Современные приложения должны сразу выводить логи в JSON-формате.

```json
{
  "@timestamp": "2023-10-27T12:00:01.456Z",
  "level": "info",
  "service": "order-service",
  "user_id": 555,
  "duration_ms": 150,
  "message": "Order created"
}
```

Когда вы отправляете JSON, Elasticsearch автоматически создает поля с нужными типами данных. Поле `duration_ms` станет числовым, и вы сможете мгновенно найти «все заказы со временем выполнения > 500мс».

## Logstash: фильтрация и обогащение

Logstash позволяет гибко обрабатывать входящий поток. В файле `logstash.conf` мы опишем логику приема, парсинга и отправки данных.

```conf
input {
  # Принимаем данные от Filebeat по протоколу Beats
  beats {
    port => 5044
  }
}

filter {
  # 1. Распознаем JSON из приложения
  if [message] =~ /^\{.*/ {
    json {
      source => "message"
      target => "app"
    }
  }

  # 2. Использование Grok-фильтров для системных логов (например, Nginx)
  if [container][image][name] =~ /nginx/ {
    grok {
      match => { "message" => "%{COMBINEDAPACHELOG}" }
    }
    date {
      match => [ "timestamp" , "dd/MMM/yyyy:HH:mm:ss Z" ]
    }
  }

  # 3. Обогащение данными User-Agent
  if [app][user_agent] {
    useragent {
      source => "[app][user_agent]"
      target => "browser"
    }
  }
}

output {
  # Отправляем результат в Elasticsearch
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    # Создаем динамические индексы для удобной ротации
    index => "logs-%{+YYYY.MM.dd}"
  }
}
```

Благодаря такой обработке, в Kibana вы сможете построить карту мира, показывающую, из каких стран приходят пользователи с ошибками, или узнать, на какой версии iOS ваше приложение падает чаще всего.

## Визуализация: Создаем дашборды в Kibana

Поиск через Discover удобен для дебага, но для мониторинга здоровья системы нужны дашборды. Kibana позволяет создавать сложные визуализации на основе ваших логов.

**Популярные типы визуализаций:**
1.  **Metric:** Простое число. Например, «Количество ошибок за последние 15 минут».
2.  **Lens:** Самый удобный инструмент «drag-and-drop» для создания графиков. Вы просто перетаскиваете поле `app.duration_ms` и выбираете «Average», чтобы увидеть график среднего времени ответа.
3.  **Pie Chart:** Диаграмма кодов ответов. Сразу видно, какую долю занимают 200, 404 и 500 ошибки.
4.  **Data Table:** Список самых «популярных» ошибок с их количеством.

**Как собрать дашборд:**
*   Перейдите в раздел **Analytics -> Dashboard**.
*   Нажмите **Create new dashboard**.
*   Добавляйте визуализации через **Create visualization**.
*   Настройте фильтры (например, исключите тестовые окружения) и сохраните.

Хороший дашборд позволяет DevOps-инженеру за один взгляд понять: «Система жива или пора открывать ноутбук?».

## Управление индексами и Retention Policy (ILM)

Логи имеют свойство накапливаться. Если их не удалять, диск заполнится, и Elasticsearch перестанет принимать данные.

**Best Practice: Index Lifecycle Management (ILM)**
В Elasticsearch есть встроенный механизм ILM. Вместо ручного удаления индексов, вы создаете политику. Например, через Dev Tools в Kibana:

```json
PUT _ilm/policy/logs_retention_policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": { "max_size": "50gb", "max_age": "30d" }
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": { "delete": {} }
      }
    }
  }
}
```

Эта политика говорит: «Держи индекс в фазе Hot, пока ему не исполнится 30 дней или он не вырастет до 50 ГБ, а через 90 дней после этого — удали его совсем».

**Никогда не храните логи вечно на одном и том же узле без политики удаления.**

## Kibana Discover: ищем ошибки в логах

После запуска стека зайдите в Kibana (порт 5601). 
1.  Перейдите в **Stack Management -> Data Views**.
2.  Создайте Data View с паттерном `logs-*`.
3.  Откройте раздел **Discover**.

### Примеры запросов KQL (Kibana Query Language):
*   `app.level : "error"` — найти все ошибки.
*   `app.duration_ms > 1000` — найти долгие запросы.
*   `http.response.status_code : [500 TO 599]` — найти все 5xx ошибки.
*   `container.name : "nginx" AND message : "404"` — искать 404 ошибки только в логах Nginx.

## Устранение неполадок (Troubleshooting)

Даже при правильной конфигурации ELK может преподнести сюрпризы. Вот основные проблемы, с которыми вы можете столкнуться:

1.  **Elasticsearch не запускается (Exit code 137):**
    *   **Причина:** Нехватка оперативной памяти. Код 137 означает, что контейнер был убит OOM Killer.
    *   **Решение:** Проверьте `ES_JAVA_OPTS`. Если на сервере всего 4 ГБ RAM, не выделяйте ES больше 2 ГБ. Также убедитесь, что в Docker Desktop (если вы на Windows/Mac) выделено достаточно ресурсов.

2.  **Логи не доходят до Kibana:**
    *   **Причина:** Ошибка в цепочке Filebeat -> Logstash.
    *   **Решение:** Проверьте логи Filebeat. Если вы видите `Connection refused`, значит Logstash не доступен по сети или порт 5044 закрыт. Если Filebeat пишет `Publishing 0 events`, значит он не может найти файлы логов по указанному пути.

3.  **Kibana пишет «Elasticsearch is not ready yet»:**
    *   **Причина:** Kibana запустилась быстрее, чем Elasticsearch успел проинициализироваться.
    *   **Решение:** Просто подождите 1-2 минуты. В production-средах рекомендуется использовать `healthcheck` в Docker Compose для управления порядком запуска.

## Loki — когда ELK избыточен

ELK — мощный, но прожорливый. Если у вас небольшой проект или вы ограничены в ресурсах, посмотрите на **Grafana Loki**.

| Характеристика | ELK Stack | Grafana Loki |
| :--- | :--- | :--- |
| **Ресурсы (RAM)** | Высокие (от 4ГБ+) | Низкие (от 256МБ) |
| **Индексация** | Полный текст (быстрый поиск везде) | Только метаданные (лейблы) |
| **Хранение** | Дорого (индексы занимают много места) | Дешево (сжатые блоки в S3/облаке) |
| **Сложность** | Высокая | Низкая |

**Выбирайте ELK**, если вам нужен сложный полнотекстовый поиск и глубокая аналитика.
**Выбирайте Loki**, если вам нужно просто смотреть логи контейнеров в Grafana и вы хотите сэкономить на серверах.

## Что изучать дальше?

Освоение сбора логов — это только начало. Elastic Stack предлагает огромный набор инструментов для Observability:

*   **Alerting:** Настройте уведомления в Slack или Telegram, если количество ошибок в секунду превышает порог.
*   **APM (Application Performance Monitoring):** Позволяет видеть «трассировку» запроса — какие функции внутри кода работали медленно и на каких SQL-запросах тормозила база.
*   **Uptime:** Мониторинг доступности ваших сервисов (HTTP, TCP, ICMP) из разных точек сети.
*   **Fleet:** Централизованное управление сотнями агентов Filebeat и Metricbeat из интерфейса Kibana.

## Антипаттерны

1.  **❌ Elasticsearch без heap limits:** Самая частая причина «падения» серверов у новичков.
2.  **❌ Логи в формате printf:** Текстовые логи — это долги, которые вы отдадите позже при попытке их парсить.
3.  **❌ Отсутствие ротации на диске:** Даже если ELK работает, Docker продолжает писать в `/var/lib/docker/containers`. Настройте лимиты размера файлов в `daemon.json`.
4.  **❌ Логирование конфиденциальных данных:** Пароли, токены и персональные данные (PII) никогда не должны попадать в ELK в открытом виде. Используйте фильтры Logstash для их маскирования.

## Заключение

ELK Stack — это промышленный стандарт. Несмотря на свою ресурсоемкость, он дает непревзойденные возможности по поиску и анализу данных. Правильно настроенный пайплайн сборки логов — это залог быстрого решения инцидентов и спокойного сна DevOps-инженера.
