# Terraform в команде: state, backend и модули

Инфраструктура как код (IaC) совершила революцию в управлении ресурсами, позволив описывать серверы, сети и базы данных в текстовых файлах. Однако переход от запуска `terraform apply` на личном ноутбуке к полноценной командной разработке — это не просто перенос кода в Git. Это смена парадигмы, где «состояние» (state) становится центральным узлом взаимодействия, а модульность — залогом выживания и масштабируемости системы.

В этой статье мы разберем, почему локальный state — это яд для команды, как настроить надежный Remote Backend на базе AWS S3 и DynamoDB, и как правильно структурировать код с помощью модулей и workspaces, чтобы избежать хаоса и конфликтов.

---

## Подготовка к работе (Prerequisites)

Прежде чем переходить к командной работе, необходимо убедиться, что ваша рабочая среда соответствует современным стандартам:

1.  **Terraform CLI**: У вас должна быть установлена версия 1.6 или выше. Проверить текущую версию можно командой `terraform -v`. Версия 1.6+ важна для поддержки новых функций, таких как декларативный импорт ресурсов.
2.  **AWS CLI и учетные данные**: У вас должен быть настроен доступ к облаку AWS. Terraform ищет учетные данные в стандартных местах: переменных окружения (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`) или в конфигурационном файле `~/.aws/credentials`.
3.  **Минимальные IAM права**: Для выполнения начального этапа («bootstrap») вашему пользователю потребуются права на:
    *   `s3:CreateBucket`, `s3:PutBucketVersioning`, `s3:PutEncryptionConfiguration`, `s3:PutBucketPublicAccessBlock`
    *   `dynamodb:CreateTable`, `dynamodb:DescribeTable`, `dynamodb:UpdateTable`

## Проблема локального state в команде

Файл `terraform.tfstate` — это «мозг» вашей инфраструктуры. В нем в формате JSON хранится детальная карта соответствия между вашими манифестами и реальными объектами в облаке (ID инстансов, ARN бакетов, IP-адреса).

Когда вы работаете над проектом в одиночку, локальное хранение файла кажется удобным. Но как только в проект приходит второй инженер, локальный state превращается в мину замедленного действия.

### 1. Проблема «Разделенного мозга» (Split Brain)
Представьте ситуацию: Инженер А добавил правило в Security Group и запустил `terraform apply`. Его локальный state обновился. Инженер Б в это же время решил изменить тип инстанса. Поскольку Инженер Б не имеет доступа к обновленному файлу Инженера А, его копия Terraform «думает», что нового правила не существует. При запуске `apply` Terraform удалит правило, созданное коллегой. Результат — непредсказуемое поведение инфраструктуры и возможные простои.

### 2. Секреты в открытом виде
Terraform state хранит значения всех атрибутов ресурсов. Если вы создаете базу данных RDS и передаете пароль через переменную, этот пароль будет записан в `.tfstate` в открытом виде. Хранение такого файла в Git (даже в приватном репозитории) — критическая ошибка безопасности. Секреты остаются в истории коммитов навсегда, и любой, кто получит доступ к репозиторию, получит ключи от вашей инфраструктуры.

### 3. Состояние гонки (Race Conditions)
Если два инженера одновременно запустят `terraform apply` без механизма блокировки, они начнут модифицировать одни и те же облачные ресурсы параллельно. Это приведет к повреждению state-файла или к тому, что API облачного провайдера вернет ошибки о конфликтах. Без централизованного контроля вы никогда не узнаете, чьи изменения были применены последними.

### 4. Отсутствие аудита и отказоустойчивости
Локальный файл легко случайно удалить или испортить. Без централизованного и версионируемого хранилища вы не можете узнать, кто внес изменения 15 минут назад, и не сможете быстро откатиться к рабочему состоянию, если что-то пошло не так.

## Remote backend: S3 + DynamoDB

Чтобы решить эти проблемы, Terraform предлагает концепцию **Remote Backend**. Это удаленное хранилище, куда Terraform автоматически выгружает состояние после каждой операции.

В экосистеме AWS «золотым стандартом» является связка **Amazon S3** (хранение файла) и **Amazon DynamoDB** (распределенная блокировка).

### Почему именно эта связка?
*   **S3**: Объектное хранилище с высочайшей доступностью и поддержкой версионности. Если state-файл будет поврежден, вы сможете восстановить его из предыдущей версии. С 2021 года S3 обеспечивает строгую согласованность после записи (strong read-after-write consistency), что критично для state.
*   **DynamoDB**: Terraform использует ее для создания «флага блокировки». Пока один процесс выполняет `apply`, в таблицу вносится запись, запрещающая другим пользователям вносить изменения.

### Глубокое погружение: Механизм блокировок и конкурентный доступ

Когда вы запускаете `terraform apply` в команде, возникает риск того, что два человека одновременно нажмут «Enter». Без блокировок это привело бы к состоянию гонки (race condition).

**Как работает блокировка через DynamoDB:**
1.  При запуске любой команды, изменяющей состояние (`apply`, `destroy`, `import`), Terraform сначала обращается к таблице DynamoDB.
2.  Он пытается создать запись с ключом `LockID` (значение которого соответствует пути к state-файлу в S3).
3.  Если запись успешно создана, Terraform получает «эксклюзивное право» на изменение инфраструктуры. В записи фиксируется ID операции, имя пользователя и время начала.
4.  Если другой инженер в это время запустит `terraform apply`, его CLI получит ошибку: `Error: Error acquiring the state lock`. В сообщении будет указано, кто именно и когда заблокировал состояние.
5.  После завершения операции (`Apply complete!`), Terraform автоматически удаляет запись из DynamoDB, освобождая путь для коллег.

**Что если блокировка «зависла»?**
Иногда процесс Terraform может аварийно завершиться (например, при потере интернет-соединения или сбое CI-сервера), не успев удалить запись из DynamoDB. В этом случае вы не сможете запустить команду повторно.
Решение — команда `terraform force-unlock <LOCK_ID>`, где ID берется из сообщения об ошибке. Будьте крайне осторожны: убедитесь, что ваш коллега действительно не выполняет изменения в данный момент.

**Согласованность данных в S3:**
Важно понимать, что S3 хранит сам JSON-файл состояния. Благодаря Strong Consistency, как только Terraform завершает запись обновленного state в S3, любой последующий `read` (от любого другого инженера) гарантированно получит самую свежую версию. Это исключает ситуацию, когда один инженер видит «старую» картину мира из-за задержек репликации в облаке.

### Шаг 0: Подготовка инфраструктуры (Bootstrapping)
Прежде чем прописать backend в конфигурации, нужно создать сами ресурсы S3 и DynamoDB. Создайте отдельную директорию `bootstrap/` внутри вашего репозитория.

**Пример `bootstrap/main.tf`:**
*Примечание: Имена S3 бакетов глобально уникальны. Используйте формат `company-project-env-state`.*

```hcl
provider "aws" { region = "us-east-1" }

resource "aws_s3_bucket" "terraform_state" {
  bucket = "my-org-project-prod-tfstate" # ЗАМЕНИТЕ НА СВОЕ
  lifecycle { prevent_destroy = true }
}

resource "aws_s3_bucket_versioning" "enabled" {
  bucket = aws_s3_bucket.terraform_state.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "default" {
  bucket = aws_s3_bucket.terraform_state.id
  rule {
    apply_server_side_encryption_by_default { sse_algorithm = "AES256" }
  }
}

resource "aws_dynamodb_table" "terraform_locks" {
  name         = "terraform-state-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  attribute { name = "LockID"; type = "S" }
}
```

**Запуск bootstrap:**
Перейдите в папку `bootstrap/` и выполните: `terraform init` -> `terraform plan` -> `terraform apply`.

### Шаг 1: Конфигурация Remote Backend
Добавьте блок `terraform` в файл `backend.tf` основного проекта:

```hcl
terraform {
  required_version = ">= 1.6.0"

  backend "s3" {
    bucket         = "my-org-project-prod-tfstate" # Имя из Шага 0
    key            = "global/s3/terraform.tfstate" # Путь в бакете
    region         = "us-east-1"
    dynamodb_table = "terraform-state-locks"
    encrypt        = true
  }
}
```

После добавления блока запустите `terraform init`. Terraform предложит мигрировать данные из локального state в S3.

## Workspaces — dev, staging, production

**Workspaces** позволяют использовать один и тот же код для разных сред (dev, staging, prod), разделяя их состояния в бакете S3.

### Основные команды
```bash
terraform workspace list          # Список окружений
terraform workspace new production # Создать новое
terraform workspace select dev    # Переключиться
```

### Использование в коде
С помощью переменной `${terraform.workspace}` можно динамически менять параметры ресурсов. Это позволяет избежать дублирования кода и сделать инфраструктуру гибкой.

#### Динамические теги и аллокация затрат (Cost Allocation)
В крупных компаниях важно понимать, сколько тратит каждое окружение. Это достигается за счет тегирования.

```hcl
locals {
  # Карта параметров для разных сред
  env_params = {
    default = { 
      instance_count = 1
      type           = "t3.micro"
      owner          = "platform-team"
      backup_enabled = false
    }
    dev = { 
      instance_count = 1
      type           = "t3.micro"
      owner          = "dev-team-a"
      backup_enabled = false
    }
    prod = { 
      instance_count = 5
      type           = "m5.large"
      owner          = "ops-team"
      backup_enabled = true
    }
  }

  # Получаем параметры для текущего воркспейса
  current = lookup(local.env_params, terraform.workspace, local.env_params["default"])

  # Общие теги для всех ресурсов проекта
  common_tags = {
    Project     = "SkyNet"
    Environment = terraform.workspace
    ManagedBy   = "Terraform"
    Owner       = local.current.owner
    CostCenter  = "department-101" # Для финансового биллинга
  }
}

resource "aws_instance" "web" {
  count         = local.current.instance_count
  instance_type = local.current.type
  ami           = data.aws_ami.ubuntu.id
  
  # Слияние базовых тегов с уникальными для ресурса
  tags = merge(local.common_tags, {
    Name = "Web-${terraform.workspace}-${count.index}"
    Backup = local.current.backup_enabled ? "True" : "False"
  })
}
```

#### Нюансы использования Workspaces
1.  **Изоляция**: Все состояния для разных воркспейсов хранятся в одном бакете S3, но под разными префиксами (`env:/dev/`, `env:/prod/`). Это логическая изоляция.
2.  **Видимость ресурсов**: Будьте осторожны — если вы удаляете ресурс в воркспейсе `dev`, он не затронет `prod`, но риск ошибки при переключении командной строкой сохраняется.
3.  **Переменные**: Вы не можете использовать разные файлы `.tfvars` автоматически для разных воркспейсов без дополнительных инструментов или оберток.

### Выбор архитектуры: Workspaces vs Папки
| Критерий | Terraform Workspaces | Структура по директориям |
| :--- | :--- | :--- |
| **Код** | Один набор файлов для всех сред | Разные вызовы модулей в папках |
| **Дублирование** | Минимальное | Требуется дублировать `backend.tf` |
| **Риск ошибки** | Выше (можно забыть переключить) | Ниже (переход в папку осознан) |
| **Изоляция** | Логическая (один бакет S3) | Физическая (можно разные бакеты) |

**Рекомендация:** Используйте Workspaces для feature-ветки. Для Production лучше использовать разделение по директориям (`envs/prod/` и `envs/dev/`), чтобы минимизировать риск случайного удаления данных.

## Модули — переиспользуем код

Модули в Terraform позволяют упаковать группу связанных ресурсов в логический блок с входными параметрами (`variables`) и выходами (`outputs`).

### Анатомия хорошего модуля
Модуль должен быть атомарным. Типичная структура папок:
```text
modules/
  s3-static-site/
    main.tf      # Ресурсы
    variables.tf # Входные данные
    outputs.tf   # Возвращаемые данные
    README.md    # Описание
```

### Создание комплексного reusable модуля (App Instance)
Рассмотрим создание модуля для типового веб-сервера, который требует не только инстанс, но и права доступа (IAM) и сетевые правила (Security Group).

**Структура модуля `modules/aws-app-instance/`:**
*   `main.tf`: Основной ресурс EC2.
*   `iam.tf`: Роль, политики и Instance Profile.
*   `sg.tf`: Правила входящего и исходящего трафика.
*   `variables.tf`: Настройки гибкости.

**Шаг 1: `iam.tf` — права доступа**
В облаке важно следовать принципу наименьших привилегий (Least Privilege).
```hcl
resource "aws_iam_role" "app_role" {
  name = "${var.app_name}-role"
  assume_role_policy = jsonencode({
    Version = "2012-10-17", Statement = [{
      Action = "sts:AssumeRole", Effect = "Allow", Principal = { Service = "ec2.amazonaws.com" }
    }]
  })
}

resource "aws_iam_instance_profile" "app_profile" {
  name = "${var.app_name}-profile"
  role = aws_iam_role.app_role.name
}
```

**Шаг 2: `sg.tf` — сетевая безопасность**
Вместо жестко заданных портов, позволим пользователю модуля передавать список разрешенных портов.
```hcl
resource "aws_security_group" "app_sg" {
  name        = "${var.app_name}-sg"
  description = "Security group for ${var.app_name}"

  dynamic "ingress" {
    for_each = var.allowed_ports
    content {
      from_port   = ingress.value
      to_port     = ingress.value
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  }

  egress {
    from_port   = 0; to_port = 0; protocol = "-1"; cidr_blocks = ["0.0.0.0/0"]
  }
}
```

**Шаг 3: `main.tf` — объединение**
```hcl
resource "aws_instance" "app" {
  ami                  = var.ami_id
  instance_type        = var.instance_type
  iam_instance_profile = aws_iam_instance_profile.app_profile.name
  vpc_security_group_ids = [aws_security_group.app_sg.id]

  tags = { Name = var.app_name }
}
```

**Шаг 4: `variables.tf` — входные данные**
```hcl
variable "app_name" { type = string; description = "Name of the application" }
variable "ami_id" { type = string }
variable "instance_type" { type = string; default = "t3.micro" }
variable "allowed_ports" { 
  type = list(number)
  default = [80, 443]
  description = "List of TCP ports to open in Security Group"
}
```

### Вызов модуля
Теперь любой инженер в вашей команде может развернуть защищенный инстанс одной командой:
```hcl
module "payment_gateway" {
  source        = "../../modules/aws-app-instance"
  app_name      = "payment-gw"
  ami_id        = "ami-0c55b159cbfafe1f0"
  allowed_ports = [80, 443, 8080]
}
```

При использовании внешних модулей из [Terraform Registry](https://registry.terraform.io/) всегда фиксируйте версию: `version = "5.1.0"`. Это защитит вас от внезапных изменений в API модуля, которые могут сломать ваш билд.

## `terraform import`: Берём под контроль существующее

Если ресурсы были созданы вручную через консоль или с помощью сторонних скриптов, вы можете импортировать их под управление Terraform. С версии 1.5 появился декларативный способ импорта, который значительно упрощает жизнь.

### Декларативный импорт (Terraform 1.5+)
Вместо того чтобы вручную писать `terraform import` в консоли для каждого ресурса, вы добавляете специальный блок в код:
```hcl
import {
  to = aws_instance.legacy_server
  id = "i-0abcd1234efgh5678" # ID ресурса в AWS
}

import {
  to = aws_security_group.legacy_sg
  id = "sg-0123456789abcdef"
}
```

После добавления блока запустите команду для автоматической генерации кода:
```bash
terraform plan -generate-config-out=generated_resources.tf
```
Terraform проанализирует текущее состояние облака и создаст файл `generated_resources.tf` с описанием ресурсов. Вам останется только проверить его, отредактировать (например, заменить hardcoded значения на переменные) и запустить `terraform apply`.

### Частые ловушки и подводные камни импорта
1.  **State Drift (Дрейф состояния)**: После импорта Terraform может показать, что ресурс требует изменений (например, добавления тегов, которых нет в реальности). Это нормально — приведите код в соответствие с желаемым состоянием.
2.  **Зависимости**: Terraform импортирует ресурс, но не всегда понимает связи между ними (например, какая подсеть используется). Эти связи нужно прописать вручную.
3.  **Атрибуты, доступные только для чтения**: Некоторые параметры нельзя импортировать или изменить после создания (например, Availability Zone для некоторых ресурсов). Если код в TF отличается от реальности в таких полях, Terraform может попытаться пересоздать ресурс.
4.  **Удаление блока import**: После успешного выполнения `apply` блок `import` можно (и нужно) удалить из кода, так как ресурс уже находится в `state`.

## Основные команды и утилиты для команды

### Таблица полезных команд Terraform

| Команда | Описание | Применение |
| :--- | :--- | :--- |
| `terraform init` | Инициализация проекта и бэкенда | При старте или смене бэкенда |
| `terraform fmt -recursive` | Форматирование всех файлов проекта | Перед каждым коммитом в Git |
| `terraform validate` | Синтаксическая проверка кода | В CI/CD пайплайнах |
| `terraform state list` | Список ресурсов в текущем состоянии | Для инвентаризации ресурсов |
| `terraform state show <addr>` | Детальные атрибуты ресурса из state | Для отладки и проверки ID |
| `terraform state rm <addr>` | Удаление из state (без удаления в облаке) | При выводе из-под управления TF |
| `terraform state mv <old> <new>`| Переименование ресурса в state | Рефакторинг без пересоздания |
| `terraform taint <addr>` | Пометка ресурса на пересоздание | Если инстанс работает неверно |
| `terraform apply -replace="<a.b>"`| Современный способ пересоздать ресурс | Замена taint в новых версиях |

### Утилиты качества кода (Must Have)
Работа в команде требует единого стандарта качества кода. Эти инструменты помогут автоматизировать проверки и избежать глупых ошибок.

1.  **TFLint**: "Умный" линтер. В отличие от `terraform validate`, он понимает специфику провайдеров. Например, он может предупредить вас, что выбранный тип инстанса `t2.micro` недоступен в регионе `us-east-1` или что вы используете устаревший синтаксис.
2.  **Безопасность: Checkov vs tfsec**:
    *   **Checkov**: Поддерживает сотни политик "из коробки" для AWS, Azure, GCP. Позволяет писать собственные правила на Python. Проверяет не только Terraform, но и CloudFormation, Kubernetes.
    *   **tfsec**: Очень быстрый сканер безопасности, написанный на Go. Основной упор сделан на поиск открытых портов, отсутствия шифрования на дисках и публично доступных бакетов S3. Рекомендуется использовать оба инструмента в связке.
3.  **Infracost**: Анализатор стоимости. Он подключается к пайплайну и для каждого `terraform plan` генерирует отчет: "Ваш бюджет вырастет на $150/мес". Это мощный аргумент при код-ревью.
4.  **terraform-docs**: Утилита, которая вычитывает ваши `variables.tf` и `outputs.tf` и генерирует красивую таблицу в `README.md`. Незаменима для тех, кто пишет модули для коллег.
5.  **Terragrunt (Бонус)**: Это тонкая обертка (wrapper) над Terraform. Она помогает реализовать принцип DRY (Don't Repeat Yourself) для конфигураций с множеством окружений и аккаунтов. Terragrunt автоматически настраивает backend, управляет зависимостями между модулями и делает структуру проекта чище.

## CI/CD и безопасность в команде

Все изменения в инфраструктуре должны проходить через автоматизированный пайплайн. Это гарантирует, что код отформатирован, не содержит синтаксических ошибок и прошел проверку безопасности.

### Пример GitHub Actions для Terraform

Ниже приведен пример рабочего процесса, который использует **OIDC** для безопасной аутентификации в AWS без хранения статических ключей (`ACCESS_KEY`).

```yaml
name: Terraform CI/CD

on:
  pull_request: # Запуск при создании PR
  push:
    branches: [ main ] # Запуск при слиянии в main

permissions: # Нужно для OIDC (OpenID Connect)
  id-token: write
  contents: read
  pull-requests: write

jobs:
  terraform:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          role-to-assume: arn:aws:iam::1234567890:role/github-actions-role
          aws-region: us-east-1

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v2

      - name: Terraform Format
        run: terraform fmt -check

      - name: Terraform Init
        run: terraform init

      - name: Terraform Plan
        if: github.event_name == 'pull_request'
        run: terraform plan -no-color
        # Здесь можно добавить скрипт для публикации Plan в комментарии к PR

      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply -auto-approve
```

### Разбор этапов пайплайна
1.  **Format & Validate**: Первым делом проверяем стиль кода. Если инженер забыл запустить `terraform fmt`, билд упадет.
2.  **Plan на Pull Request**: При создании PR пайплайн выводит результат `terraform plan`. Это позволяет коллегам при ревью видеть не просто код, а реальные изменения, которые произойдут в облаке.
3.  **Apply на Merge**: Мы никогда не запускаем `apply` вручную с ноутбуков для `production`. Применение изменений происходит только после того, как код попал в основную ветку (`main`) и прошел все проверки.

**Безопасность (OIDC):** Использование OpenID Connect позволяет GitHub Actions получать временные токены доступа от AWS. Вам не нужно создавать IAM пользователя и хранить его секреты в GitHub Secrets, которые могут быть скомпрометированы. Вы просто доверяете определенному репозиторию GitHub выпускать токены для конкретной IAM роли.

## Итого: Лучшие практики и антипаттерны

### ✅ Что нужно делать (Best Practices)
1.  **Remote State с первого дня**: Настраивайте S3 Backend сразу.
2.  **Versioning & Locks**: Всегда включайте версионность в S3 и блокировки в DynamoDB.
3.  **Используйте `.gitignore`**: Не допускайте попадания `.terraform/` и `*.tfstate` в Git.
4.  **Код-ревью**: Изменения в `prod` должны проверяться коллегами.

### ❌ Чего делать НЕ стоит (Антипаттерны)
1.  **❌ Один workspace для всех окружений**: Разделяйте среды для минимизации рисков.
2.  **❌ Ручное редактирование state**: Никогда не открывайте файл `.tfstate` в редакторе.
3.  **❌ Hardcoded secrets**: Пароли должны приходить из переменных или Vault.
4.  **❌ Модули без документации**: Описывайте переменные (поле `description`).

Terraform — мощный инструмент, который при правильном подходе становится фундаментом для масштабирования всей инфраструктуры компании.
