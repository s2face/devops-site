# Terraform: управляем инфраструктурой как кодом

## Что такое IaC и зачем он нужен

**Infrastructure as Code (IaC)** — это методология управления и подготовки IT-инфраструктуры через машиночитаемые файлы определений, а не через физическую настройку оборудования или интерактивные инструменты настройки.

### Мутабельная vs Иммутабельная инфраструктура

Одним из ключевых архитектурных решений при переходе к IaC является выбор между обновлением существующих ресурсов и их полной заменой.

*   **Мутабельная (изменяемая) инфраструктура:** Традиционный подход. Если вам нужно обновить PHP на сервере, вы заходите на него (через SSH или Ansible) и запускаете обновление. 
    *   *Проблема:* Со временем серверы "дрейфуют". Один обновился успешно, на другом возникла ошибка, на третьем кто-то вручную поправил конфиг. В итоге парк серверов становится неоднородным, что крайне затрудняет отладку.
*   **Иммутабельная (неизменяемая) инфраструктура:** Современный облачный подход. Вместо изменения существующего сервера вы собираете новый образ (AMI) или просто запускаете новый инстанс с обновленным скриптом инициализации, а старый удаляете.
    *   *Преимущество:* Вы всегда точно знаете, что запущено в вашей среде, потому что каждый инстанс создан "с нуля" по строгому шаблону. Terraform идеально подходит для этого подхода.

### Terraform vs Ansible: сравнение инструментов

Часто новички спрашивают: "Зачем мне Terraform, если я уже умею в Ansible?". Эти инструменты не конкуренты, а партнеры.

| Характеристика | Terraform | Ansible |
| :--- | :--- | :--- |
| **Специализация** | Оркестрация (Infrastructure Provisioning) | Управление конфигурациями (Config Management) |
| **Подход** | Декларативный (Описываем конечную цель) | Преимущественно процедурный (Описываем шаги) |
| **Управление состоянием** | Есть (State-файл) | Нет (Stateless) |
| **Жизненный цикл** | Создание, изменение, удаление всей инфраструктуры | Настройка ПО внутри уже созданных ОС |

**Пример совместной работы:** Terraform создает VPC, подсети и EC2-инстанс, а затем передает IP-адрес этого инстанса в Ansible, который устанавливает там Docker и запускает контейнеры.

## Установка Terraform

Terraform поставляется в виде одного исполняемого файла. Мы рекомендуем использовать официальный репозиторий HashiCorp для получения обновлений.

### Для Ubuntu/Debian:
```bash
# Установка зависимостей и ключа GPG
sudo apt-get update && sudo apt-get install -y gnupg software-properties-common curl
curl -fsSL https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

# Добавление репозитория
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
sudo tee /etc/apt/sources.list.d/hashicorp.list

# Установка
sudo apt-get update && sudo apt-get install terraform
```

Проверьте работоспособность: `terraform -version`. Вы должны увидеть версию 1.6.0 или выше.

## Структура Terraform-проекта

В Terraform файлы с расширением `.tf` в одной директории образуют "модуль". Хотя можно писать всё в одном файле, стандарт индустрии — разделение по ролям:

*   `versions.tf` — версии Terraform и провайдеров.
*   `providers.tf` — настройки подключения к облакам.
*   `variables.tf` — описание входных параметров (типы, описания, дефолты).
*   `main.tf` — описание ресурсов.
*   `outputs.tf` — данные, которые мы хотим получить "на выходе" (IP, DNS и т.д.).
*   `terraform.tfvars` — (опционально) реальные значения переменных (не путайте с `variables.tf`!).

## Провайдеры и ресурсы: основные блоки

### Провайдеры (Providers)
Это плагины, которые "учат" Terraform говорить на языке конкретного API (AWS, Azure, Google Cloud, Kubernetes).

```hcl
# providers.tf
provider "aws" {
  region = var.region
  # Credentials лучше передавать через переменные окружения AWS_ACCESS_KEY_ID
}
```

### Ресурсы (Resources)
Самый важный блок. Он описывает объект, который вы хотите создать.

```hcl
# main.tf
resource "aws_security_group" "web_sg" {
  name        = "allow_http"
  description = "Allow HTTP inbound traffic"

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

resource "aws_instance" "web" {
  ami                    = data.aws_ami.ubuntu.id
  instance_type          = var.instance_type
  vpc_security_group_ids = [aws_security_group.web_sg.id]

  tags = {
    Name = "DevOps-Lesson-Node"
  }
}
```

### Data Sources
Позволяют получать данные из существующей инфраструктуры или из API облака.

```hcl
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"] # Canonical

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }
}
```

## Рабочий цикл: Команды управления

1.  **`terraform init`**
    Скачивает плагины провайдеров и настраивает бэкенд для хранения стейта. Запускается первым в новом проекте.
2.  **`terraform plan`**
    **Критически важный шаг.** Terraform читает ваш код и текущее состояние облака, после чего выводит список действий: что будет создано (`+`), что изменено (`~`), а что удалено (`-`). Это ваша страховка от ошибок.
3.  **`terraform apply`**
    Исполняет план. По умолчанию требует подтверждения словом `yes`.
4.  **`terraform destroy`**
    Удаляет всё, что было создано этим проектом.

## Жизненный цикл ресурсов (Lifecycle)

Блок `lifecycle` позволяет управлять логикой обновления ресурсов:

```hcl
resource "aws_instance" "web" {
  # ...
  lifecycle {
    # Сначала создается новый инстанс, и только при успехе удаляется старый.
    # Полезно для минимизации Downtime.
    create_before_destroy = true

    # Попытка удалить этот ресурс через terraform завершится ошибкой.
    # Обязательно для баз данных и хранилищ!
    prevent_destroy = false 
  }
}
```

## State файл — сердце и память Terraform

`terraform.tfstate` — это JSON-файл, в котором Terraform хранит соответствие ваших имен в коде реальным ID в облаке.

### Почему это важно?
Если вы измените тег у сервера в консоли AWS вручную, Terraform при следующем `plan` увидит расхождение со стейтом и предложит вернуть всё "как в коде".

### ⚠️ Безопасность и хранение:
1.  **НИКОГДА не кладите `.tfstate` в Git.**
    *   В нем хранятся секреты (пароли от БД, сертификаты) в открытом тексте.
    *   Git не поддерживает блокировки: если два человека одновременно запустят `apply`, стейт может "развалиться".
2.  **Используйте Remote State.** На реальных проектах стейт хранится в S3 (AWS) или Terraform Cloud, с обязательной блокировкой через DynamoDB.

## Лучшие практики и антипаттерны

### ✅ Best Practices:
*   **Используйте переменные:** Никогда не хардкодьте ID образов или регионы.
*   **Разделяйте окружения:** Используйте разные стейты (и папки) для `dev`, `staging` и `prod`.
*   **Модули:** Если вы создаете что-то чаще одного раза — сделайте из этого модуль.
*   **Документируйте:** Используйте поле `description` в переменных и выводах.

### ❌ Anti-patterns:
*   **Игнорирование `plan`:** Запуск `apply -auto-approve` без предварительного просмотра плана — прямой путь к увольнению после удаления продакшн-базы.
*   **Ручные правки (ClickOps):** Если ресурс управляется Terraform, любые изменения в нем через UI-консоль облака — табу.
*   **Огромные файлы:** `main.tf` на 5000 строк невозможно поддерживать. Группируйте ресурсы логически.
*   **Секреты в коде:** Никогда не коммитьте ключи доступа. Используйте переменные окружения или Vault.

## Практическое задание

1.  Установите Terraform на свою рабочую станцию.
2.  Сконфигурируйте доступ к AWS через `aws configure`.
3.  Напишите конфигурацию для создания одного инстанса `t3.micro` с вашей Security Group.
4.  Запустите `terraform init` и `terraform plan`. Изучите вывод.
5.  Примените изменения через `terraform apply`.
6.  Убедитесь, что сервер появился в консоли AWS.
7.  Уничтожьте созданную инфраструктуру через `terraform destroy`.
