Представьте, что вы строите дом. У вас есть основной проект, по которому работают все строители. Но вдруг вам захотелось попробовать новый дизайн крыши. Если вы начнете менять крышу прямо на основном объекте, и идея окажется неудачной, вам придется потратить уйму времени, чтобы вернуть всё как было. А в это время другие рабочие не смогут продолжать отделку комнат, потому что всё перерыто.

В программировании ситуация аналогичная. До появления систем контроля версий совместная работа над одним проектом была сущим кошмаром. Разработчики пересылали друг другу архивы с кодом, вручную копировали файлы и постоянно затирали изменения друг друга. 

Git решил эту проблему с помощью **веток (branches)**. Ветки позволяют «разветвлять» историю проекта, работать над разными задачами изолированно и объединять результаты только тогда, когда они полностью готовы. В этом уроке мы разберем, как устроены ветки, как ими управлять и как не сойти с ума, когда возникают конфликты.

## Под капотом Git: почему ветки — это «дешево»?

Чтобы понять магию веток, нужно заглянуть во внутреннее устройство Git. Многие старые системы контроля версий (например, SVN) при создании ветки буквально копировали все файлы проекта в новую папку на сервере. Если проект весил 1 ГБ, ветка тоже весила 1 ГБ. Это было долго и накладно.

Git работает иначе. Он хранит данные как набор слепков (snapshots). Когда вы делаете коммит, Git сохраняет:
1.  **Blobs (Binary Large Objects):** Содержимое ваших файлов.
2.  **Trees:** Структуру каталогов (какой файл в какой папке лежит).
3.  **Commit object:** Метаданные (автор, дата, сообщение) и указатель на дерево проекта, а также указатель на предыдущий (родительский) коммит.

**Ветка в Git — это просто текстовый файл весом в 41 байт.** В этом файле записан SHA-1 хеш коммита, на который указывает ветка. 

Когда вы создаете ветку `git branch feature`, Git просто создает новый файл в папке `.git/refs/heads/` с именем `feature` и записывает туда хеш текущего коммита. Всё! Никакого копирования файлов, никаких задержек. Именно поэтому в Git говорят, что ветвление «дешевое» — вы можете создавать и удалять десятки веток в день без вреда для производительности.

### Указатель HEAD: где я?
Раз веток может быть много, Git нужно знать, в какой из них вы сейчас работаете. Для этого существует специальный указатель **HEAD**. 
Представьте себе HEAD как «вы находитесь здесь» на карте торгового центра. 
*   Обычно HEAD указывает на имя текущей ветки (например, `main`).
*   Ветка, в свою очередь, указывает на последний коммит.
*   Когда вы делаете новый коммит, ветка «переезжает» на него, а HEAD послушно следует за веткой.

## Зачем нужны ветки? (Реальные сценарии)

Если вы работаете один, может показаться, что ветки — это лишняя сложность. Но это не так. Вот основные причины их использовать:

### 1. Параллельная разработка и изоляция
Представьте команду из пяти человек. Один исправляет критическую ошибку в корзине интернет-магазина, второй добавляет темную тему, третий оптимизирует базу данных. 
Ветки позволяют каждому работать в своем «пузыре». Ошибка в коде одного разработчика не «сломает» работу других, пока он не решит объединить изменения.

### 2. Эксперименты без риска
Вы хотите попробовать новую библиотеку, которая может кардинально изменить архитектуру? Создайте ветку `experiment-lib`. Если через два дня вы поймете, что библиотека плохая — вы просто удаляете ветку и возвращаетесь к `main`, которая всё это время оставалась стабильной.

### 3. Культура Code Review (Pull Requests)
В профессиональной среде код почти никогда не попадает в `main` напрямую. Разработчик делает задачу в ветке, отправляет её на сервер и создает **Pull Request (PR)**. Коллеги смотрят код в браузере (на GitHub или GitLab), пишут замечания, и только после исправления всех замечаний код вливается в проект. Это «сито», которое отсеивает ошибки до того, как они попадут к пользователям.

## Создание и переключение между ветками

Давайте перейдем к практике. 

### Просмотр веток
```bash
git branch
```
Звездочкой `*` будет отмечена ветка, на которой вы находитесь. 
*   Флаг `-v` покажет последний коммит в каждой ветке.
*   Флаг `-a` покажет также удаленные ветки (те, что живут на GitHub).

### Создание новой ветки
```bash
git branch feature/login
```
Мы создали указатель, но всё еще стоим на месте. Чтобы начать работать, нужно переключиться.

### Переключение: checkout vs switch
В старых учебниках вы увидите команду `git checkout`. Но с 2019 года Git рекомендует использовать более понятную команду `git switch`:
```bash
git switch feature/login
```
**Лайфхак:** Создать ветку и сразу перейти на неё можно одной командой:
```bash
git switch -c feature/login
```
*(Флаг `-c` означает "create"). В старых версиях Git (и многих туториалах) для этого используется команда `git checkout -b feature/login`.*

### Что такое «Detached HEAD» (Отсоединенный HEAD)?
Иногда при попытке переключиться на конкретный коммит (`git checkout <hash>`) вы увидите страшное предупреждение. Это значит, что HEAD теперь указывает не на ветку, а на «голый» коммит. 
**Внимание:** Если вы сделаете коммиты в этом состоянии, они не будут принадлежать ни одной ветке. Как только вы переключитесь на `main`, найти эти коммиты будет очень сложно (они станут «сиротами»). Если вам нужно что-то исправить в этом состоянии — всегда создавайте новую ветку: `git switch -c fix-from-past`.

## Слияние веток: git merge

Когда работа в ветке завершена, её нужно «влить» обратно в основную линию. Этот процесс называется **слиянием (merge)**.

### Алгоритм действий:
1. Переходим в ветку, **куда** хотим влить (цель).
2. Запускаем команду слияния, указывая ветку, **откуда** берем изменения (источник).

```bash
git switch main
git merge feature/login
```

### Тип 1: Fast-forward (Перемотка)
Это идеальный случай. Если в ветке `main` не было новых коммитов с тех пор, как вы создали `feature/login`, Git просто переместит указатель `main` вперед. История останется идеально ровной линией.

### Тип 2: Three-way merge (Трехстороннее слияние)
Этот тип (также известный как **Recursive merge**) возникает, если история разошлась. Например, пока вы делали логин, ваш коллега успел влить в `main` свою задачу. В этом случае Git делает следующее:
1.  Находит **общего предка** (Common Ancestor) — коммит, от которого обе ветки начали свой путь.
2.  Смотрит изменения в `main` относительно предка.
3.  Смотрит изменения в `feature/login` относительно предка.
4.  Создает новый **коммит слияния (merge commit)**, у которого будет сразу два родителя.

**Схема трехстороннего слияния:**
```text
      A---B---C  (feature)
     /         \
D---E---F-------G  (main)
```
Здесь:
*   `E` — общий предок.
*   `F` — изменения в `main`.
*   `A, B, C` — изменения в ветке фичи.
*   `G` — коммит слияния, объединяющий `C` и `F`.

## Конфликты слияния — не паниковать!

Git — умная программа, но он не умеет читать мысли. Если вы в `main` изменили 10-ю строку файла `style.css`, а в ветке `feature` тоже изменили 10-ю строку того же файла, Git не знает, какой вариант правильный. Возникает **конфликт**.

### Как это выглядит в терминале:
```bash
Auto-merging style.css
CONFLICT (content): Merge conflict in style.css
Automatic merge failed; fix conflicts and then commit the result.
```

### Пошаговое разрешение конфликта:
1.  **Откройте файл.** Git пометит проблемные места маркерами:
    ```css
    <<<<<<< HEAD
    body { background: blue; }  /* То, что сейчас в main */
    =======
    body { background: green; } /* То, что вы принесли из ветки */
    >>>>>>> feature/login
    ```
2.  **Сделайте выбор.** Удалите маркеры и оставьте нужный код. Или напишите новый вариант, объединяющий оба.
3.  **Используйте IDE.** Современные редакторы (VS Code, IntelliJ IDEA) подсвечивают конфликты цветами и предлагают кнопки "Accept Current" или "Accept Incoming". Это гораздо удобнее, чем править вручную, но базовый принцип остается тем же.
4.  **Завершите процесс:**
    ```bash
    git add style.css
    git commit -m "Fix: resolve conflict in style.css"
    ```
    *Если вы поняли, что запутались и хотите всё отменить:* `git merge --abort`.

## git rebase — искусство чистой истории

Если `merge` создает «узлы» в истории (коммиты слияния), то `rebase` позволяет их избежать. `Rebase` буквально переносит вашу ветку на новый фундамент.

### Пример: Merge vs Rebase
Представьте, что вы пишете главу книги, а автор в это время обновил предисловие.
*   **Merge:** Вы просто вклеиваете страницу с обновленным предисловием в конец своей главы. История честная, но немного запутанная.
*   **Rebase:** Вы как бы «откатываете» свою главу, читаете новое предисловие и пишете свою главу заново поверх него. История выглядит так, будто вы начали писать уже после того, как предисловие было готово.

### Пошаговый процесс Rebase:
Допустим, вы в ветке `feature` и хотите подтянуть изменения из `main`.

```bash
git switch feature
git rebase main
```

**Что происходит внутри?**
1.  Git временно «откладывает» ваши коммиты из ветки `feature`.
2.  Он обновляет `feature` до состояния `main`.
3.  Он по очереди пытается применить ваши отложенные коммиты сверху.
4.  Если возникают конфликты, он останавливается на **каждом** проблемном коммите. Вы исправляете конфликт, делаете `git add` и говорите: `git rebase --continue`.

**ЗОЛОТОЕ ПРАВИЛО REBASE:** Никогда не делайте rebase веток, которые вы уже отправили (push) в общий репозиторий. Вы переписываете историю, и если кто-то другой уже начал работать на базе ваших старых коммитов, у него всё сломается. **Rebase — только для локальных правок!**

## Стратегии ветвления: как работать в команде

Чтобы репозиторий не превратился в свалку, команды договариваются о правилах игры.

### 1. GitHub Flow (Просто и быстро)
Используется в большинстве веб-проектов и стартапов.
*   Всё, что в ветке `main` — готово к деплою (выкатке на сайт).
*   Для любой задачи создается новая ветка с описательным именем.
*   Изменения вливаются через Pull Request.

### 2. Git Flow (Для сложных релизных циклов)
Если вы выпускаете мобильное приложение или софт, который нельзя обновлять каждые 5 минут, вам подойдет Git Flow.

**Схема Git Flow:**

```text
                                  GIT FLOW STRATEGY
    
      MAIN (Master)  -------------------------------------------● [v1.0] ---------------------------● [v1.1] ---
                                                               /   ^                               /   ^
                                         (branch for hotfix)  /    | (merge)                      /    | (merge)
      HOTFIX         ----------------------------------------/-----|-----------● [hotfix] -------/-----|---------
                                                            /      |          /   ^     \       /      |
                                                   (branch)/       |         /    |      \     /       |
      RELEASE        -------------------------------------/--------|--------/-----|-------● [v1.1-rc] -|---------
                                                         /         |       /      |      /   ^     \   |
                                        (branch)        /          |      /       |     /    |      \  | (merge)
      DEVELOP        ---●-------●-------●--------------●----------●------●-------●-----●----●-------V-----------
                       / ^     / ^                    /          /      /       /               /
               (branch)/  |    /  | (merge)          /          /      /       /         (branch)/
      FEATURE        -●---'---●---'-----------------/----------/------●-------'-----------------/---------------
                      [f1]    [f2]                 /          /       [f3]                     /
                                                  /          /                                /
                                                 '----------'--------------------------------'
```

1.  **main (master):** Хранит только официальные релизы (версии v1.0, v2.0). Здесь всегда стабильный код.
2.  **develop:** «Черновик» проекта. Все фичи вливаются сюда.
3.  **feature/name:** Ветки для новых функций. Создаются от `develop`, вливаются обратно в `develop`.
4.  **release/vX.X:** Создается от `develop`, когда мы готовы к релизу. Здесь только чинят баги. Потом вливается в `main` и `develop`.
5.  **hotfix/name:** Срочная заплатка. Создается от `main`, если на продакшене нашли баг, вливается в `main` и `develop`.

**Пример из жизни:**
Вы разрабатываете банковское приложение. В ветке `develop` идет работа над новым дизайном и чатом. Но внезапно в текущей версии приложения перестали проходить платежи. Вы создаете `hotfix`, чините платежи, вливаете в `main` (чтобы пользователи получили исправление) и в `develop` (чтобы в новой версии ошибка не вернулась).

### 3. Trunk-Based Development
Популярно в Google и Facebook. Разработчики делают очень маленькие коммиты прямо в `main` (или очень короткие ветки на пару часов). Чтобы проект не падал, функции прячут под «переключатели» (Feature Flags). Это требует высокого уровня автоматизации тестов.

## Работа с удаленными репозиториями (Lesson 8 Preview)

Хотя мы подробно разберем это в следующем уроке, важно понимать: ваши ветки изначально живут только на вашем компьютере. 
*   `origin` — это стандартное имя для вашего удаленного репозитория на GitHub.
*   `git push origin feature/login` — отправить вашу ветку в облако.
*   `git fetch origin` — узнать о новых ветках, которые создали ваши коллеги, но не скачивать их код в свои файлы.

Если вы видите ветку `origin/main` — это «фотография» того, в каком состоянии была ветка `main` на сервере в момент вашего последнего обращения к нему.

## git stash — временная «заначка»

Бывает так: вы в разгаре работы, файлы перекопаны, код не компилируется. И тут прилетает срочная задача в другой ветке. Что делать?
1.  Закоммитить мусор? Плохо, это портит историю.
2.  Стереть изменения? Жалко работу.
3.  **Git Stash!**

```bash
git stash             # Спрятать изменения в «карман»
git switch other-task # Сделать срочную задачу
git switch feature    # Вернуться назад
git stash pop         # Достать изменения из кармана
```
`stash` работает как стек: вы можете класть туда сколько угодно изменений, а потом забирать их.

## Типичные ошибки и как их избежать

1.  **Работа в main:** Никогда не пишите код прямо в `main`. Если вы ошибетесь, вы сломаете проект для всех. Всегда создавайте ветку.
2.  **Огромные ветки:** Ветка, которая живет месяц и содержит 1000 изменений — это гарантированные мучения при слиянии. Старайтесь делать задачи маленькими (на 1-3 дня).
3.  **Забытый rebase:** Если вы долго работаете в своей ветке, `main` уходит далеко вперед. Периодически вливайте `main` в свою ветку (или делайте rebase), чтобы конфликты не накапливались.
4.  **Непонятные имена:** Ветка `fix1` — это плохо. Ветка `fix/header-logo-alignment` — это хорошо.

## FAQ: Часто задаваемые вопросы

**В: Можно ли переименовать ветку?**
О: Да. Если вы на ней: `git branch -m new-name`.

**В: Я удалил ветку по ошибке. Код пропал навсегда?**
О: В Git почти ничего не пропадает бесследно. Есть команда `git reflog`, которая показывает историю перемещений HEAD. Там можно найти хеш последнего коммита удаленной ветки и восстановить её.

**В: Нужно ли удалять ветки после слияния?**
О: Да, это правило хорошего тона. Удалить локально: `git branch -d name`. Удалить на сервере: `git push origin --delete name`.

Главное правило новичка: **«Боишься — создай ветку»**. Это самый дешевый способ обезопасить свой код и свои нервы.

## Ключевые выводы урока

1.  **Ветки в Git — «дешёвые»:** ветка весит 41 байт (ссылка на коммит), а не копирует файлы — создавайте ветки часто.
2.  **HEAD — указатель текущей позиции:** обычно указывает на ветку, но может быть «отсоединён» (detached HEAD) при переключении на коммит.
3.  **Изоляция задач:** одна задача = одна ветка; не работайте напрямую в `main`.
4.  **Fast-forward vs Three-way merge:** если `main` не менялась — Git просто перемотает вперёд; если были изменения — создаст коммит слияния.
5.  **Конфликты — это нормально:** Git помечает конфликтные места маркерами (`<<<<<<<`, `=======`, `>>>>>>>`); выбирайте вариант и завершайте слияние.
6.  **Rebase переписывает историю:** используйте для локальных веток; никогда не делайте rebase отправленных на сервер веток.
7.  **GitHub Flow vs Git Flow:** GitHub Flow — просто (main + feature); Git Flow — сложно (main, develop, feature, release, hotfix).
8.  **`git stash` для временных изменений:** спрячьте недоделанную работу, переключитесь на другую задачу, вернитесь и восстановите.

## Глоссарий терминов урока

| Термин | Расшифровка | Простое объяснение |
|--------|-------------|-------------------|
| **Branch** | Ветка | Указатель на коммит (файл 41 байт); позволяет работать параллельно. |
| **Snapshot** | Снимок | Способ хранения данных в Git: каждый коммит — полная копия состояния файлов. |
| **Blob** | Binary Large Object | Объект Git, хранящий содержимое файла (без имени и пути). |
| **Tree** | Дерево | Объект Git, хранящий структуру каталогов (какой blob в какой папке). |
| **HEAD** | — | Указатель на текущую позицию пользователя (обычно ссылается на ветку). |
| **Detached HEAD** | Отсоединённый HEAD | Состояние, когда HEAD указывает на конкретный коммит, а не на ветку. |
| **Merge** | Слияние | Объединение изменений из двух веток в одну. |
| **Fast-forward** | Перемотка | Тип слияния, когда Git просто перемещает указатель ветки вперёд (без коммита слияния). |
| **Three-way merge** | Трёхстороннее слияние | Тип слияния через общего предка; создаёт коммит слияния с двумя родителями. |
| **Common Ancestor** | Общий предок | Коммит, от которого разошлись две ветки (используется при слиянии). |
| **Merge Commit** | Коммит слияния | Специальный коммит с двумя родителями, создаваемый при трёхстороннем слиянии. |
| **Rebase** | Перебазирование | Перенос ветки на новую основу; переписывает историю, делая её линейной. |
| **PR** | Pull Request | Запрос на слияние ветки в основной репозиторий (на GitHub/GitLab). |
| **Feature Flag** | Флаг функции | Переключатель в коде, позволяющий скрывать недоделанные функции от пользователей. |
| **Stash** | Заначка | Временное хранилище незакоммиченных изменений (работает как стек). |
| **Reflog** | Reference Log | Журнал перемещений HEAD; позволяет восстановить удалённые ветки и коммиты. |
| **Origin** | — | Имя удалённого репозитория по умолчанию (сервер, с которого клонировали). |
| **Trunk** | Ствол | Название основной ветки (main/master) в Trunk-Based Development. |
