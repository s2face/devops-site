# CI/CD с нуля: автоматизируем сборку и деплой

## Введение: Жизнь до автоматизации

Представьте классическую ситуацию: разработчик Вася закончил фичу. Он проверил её у себя на ноутбуке — всё работает. Затем он пишет в чат админу Пете: «Петя, деплой новую версию!». Петя заходит по SSH на сервер, делает `git pull`, правит конфиги вручную, перезапускает сервис и... всё падает. Оказывается, Вася забыл сказать, что обновил версию Python или добавил новую переменную окружения.

**Проблема ручного деплоя (Manual Deployment):**
1. **«Работает у меня» (It works on my machine):** Окружение разработчика и сервера различаются. У Васи macOS, а на сервере Debian. У Васи Python 3.11, а на сервере 3.9.
2. **Человеческий фактор:** Петя может опечататься в команде `rm -rf` или забыть выполнить `migration`.
3. **Отсутствие повторяемости:** Сложно воспроизвести процесс сборки и деплоя идентично каждый раз. Если Петя уйдет в отпуск, никто не будет знать, какие именно "магические заклинания" он вводил в терминале.
4. **Медленность:** Пока Петя занят другими задачами, новый код лежит мертвым грузом в репозитории. Бизнес не получает ценность, а пользователи — новые фичи.

Решение этих проблем — **CI/CD**. Это не просто инструменты, это культура автоматизации процессов сборки, тестирования и доставки кода.

## CI, CD и CD: Три понятия, одна аббревиатура

Аббревиатура CI/CD часто сбивает с толку, потому что «CD» может означать две разные вещи. Давайте разберемся в иерархии автоматизации.

### 1. Continuous Integration (Непрерывная интеграция)
Это практика частого слияния кода всех разработчиков в общую ветку (обычно `main` или `master`).
* **Суть:** Каждый раз, когда разработчик делает `push`, автоматически запускается процесс сборки и тестирования.
* **Цель:** Как можно раньше обнаружить ошибки интеграции. Если два разработчика изменили один и тот же модуль несовместимым образом, CI скажет об этом через 5 минут, а не через месяц перед релизом.

### 2. Continuous Delivery (Непрерывная доставка)
Это расширение CI. Код не только собирается и тестируется, но и полностью готов к деплою в любую секунду.
* **Суть:** После успешного прохождения всех тестов создается артефакт (например, Docker-образ или .jar файл), который автоматически отправляется в хранилище (Artifact Registry или Docker Hub).
* **Ключевой момент:** Решение о деплое на продакшн остается за человеком. Это «нажатие кнопки» в интерфейсе CI-системы. Это подходит компаниям с жестким циклом релизов или там, где требуется апрув от бизнеса.

### 3. Continuous Deployment (Непрерывное развертывание)
Это высшая степень DevOps-зрелости.
* **Суть:** Каждое изменение, прошедшее все этапы пайплайна (тесты, линтеры, сборку), автоматически деплоится на продакшн без участия человека.
* **Условие:** Требует исключительного качества тестов (Unit, Integration, E2E) и развитого мониторинга. Если что-то пойдет не так, система должна автоматически откатить изменения (Rollback).

| Понятие | Сборка | Тесты | Релиз (Артефакт) | Деплой на Prod |
|---------|:---:|:---:|:---:|:---:|
| **CI** | ✅ | ✅ | ❌ | ❌ |
| **Continuous Delivery** | ✅ | ✅ | ✅ | 🖱️ (Вручную) |
| **Continuous Deployment**| ✅ | ✅ | ✅ | ✅ (Автоматически) |

## Анатомия CI/CD пайплайна: Как движется код

**Пайплайн (Pipeline)** — это последовательность шагов (конвейер), через которые проходит код от момента `git push` до работающего приложения.

1. **Trigger (Триггер):** Искра, зажигающая процесс. Обычно это событие в Git: коммит в ветку, создание Pull Request или создание тега (v1.0.0).
2. **Checkout:** CI-сервер скачивает свежую копию вашего кода из репозитория.
3. **Linting & Static Analysis:** Проверка стиля кода (PEP8 для Python, ESLint для JS) и поиск уязвимостей без запуска кода. Это "первая линия обороны".
4. **Unit Testing:** Запуск быстрых тестов, проверяющих отдельные функции и модули.
5. **Build:** Сборка приложения. В современном DevOps это чаще всего создание Docker-образ (`docker build`).
6. **Integration Testing:** Запуск тестов, проверяющих взаимодействие вашего сервиса с базой данных или другими API. Обычно здесь используются временные контейнеры (через Docker Compose).
7. **Staging Deploy:** Деплой в окружение, идентичное продакшну, для финальной проверки.
8. **Production Deploy:** Выход в "большой мир" к реальным пользователям.

## GitHub Actions: Ваш первый инструмент автоматизации

GitHub Actions — это современная, мощная и (для многих случаев) бесплатная система CI/CD, встроенная прямо в GitHub. Вам не нужно настраивать свои сервера, GitHub предоставляет свои мощности (Runners).

### Структура Workflow-файла
Workflow описывается на языке YAML. Это формат "ключ-значение", где крайне важны отступы (как в Python).

* **Workflow:** Общий сценарий. Лежит в папке `.github/workflows/*.yml`.
* **Events (`on`):** Определяют, когда запускать сценарий. Например: `on: [push, pull_request]`.
* **Jobs:** Группы шагов. Джобы могут зависеть друг от друга или работать параллельно. Каждая джоба запускается на "свежей" виртуальной машине.
* **Steps:** Список конкретных команд. Могут быть простыми (`run: echo hello`) или использовать готовые **Actions** — переиспользуемые блоки от сообщества.

## Практика: Создаем рабочий workflow для Docker-проекта

Допустим, у нас есть Python-приложение. Мы хотим автоматизировать проверку качества и доставку образа на Docker Hub.

Создайте файл `.github/workflows/main.yml`:

```yaml
name: CI/CD Pipeline

on:
  push:
    branches: [ main ] # Запуск при пуше в main
  pull_request:
    branches: [ main ] # Запуск при создании PR в main

jobs:
  # Первая джоба: Проверка качества
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4 # Копируем код в Runner

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.10'
          cache: 'pip' # Оптимизация: кэшируем зависимости

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install flake8 pytest
          if [ -f requirements.txt ]; then pip install -r requirements.txt; fi

      - name: Lint with flake8
        run: flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics

      - name: Run unit tests
        run: pytest

  # Вторая джоба: Сборка и доставка (только если первая прошла успешно)
  build-and-push:
    needs: test # Зависимость! Если тесты упали, Docker не собираем.
    if: github.event_name == 'push' && github.ref == 'refs/heads/main' # Только для пуша в main
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ secrets.DOCKER_USERNAME }}/myapp:latest
            ${{ secrets.DOCKER_USERNAME }}/myapp:${{ github.sha }}
```

### Технические детали: Оптимизация и Кэширование
В примере выше мы добавили `cache: 'pip'`. Почему это важно?
Без кэша GitHub Actions будет скачивать все библиотеки (Flask, SQLAlchemy, Pandas и т.д.) заново при каждом запуске. Это занимает время и тратит сетевой трафик. С кэшем время сборки может сократиться с 3 минут до 40 секунд.

## Как читать логи и исправлять ошибки

Когда ваш пайплайн запускается, во вкладке **Actions** в GitHub появляется лог выполнения.
1. **Зеленая галочка:** Все шаги прошли успешно.
2. **Красный крестик:** На каком-то этапе произошла ошибка.
   * Нажмите на джобу, затем на конкретный шаг.
   * CI выведет стандартный поток вывода (stdout/stderr). Если тесты упали, вы увидите привычный Traceback от Python или Pytest.
   * Исправьте код, сделайте коммит и `push` — пайплайн запустится снова автоматически.

## Секреты (Secrets) и безопасность

Вы заметили конструкцию `${{ secrets.DOCKER_USERNAME }}`. Это способ передачи чувствительных данных (пароли, API-ключи, токены) в пайплайн.

**Золотое правило DevOps:** Никогда не храните пароли в коде (Hardcoding).
Даже если ваш репозиторий приватный, это плохая практика. Пароли могут попасть в историю коммитов, их могут увидеть коллеги, которым они не положены.

**Как использовать Secrets в GitHub:**
1. Перейдите в **Settings** репозитория.
2. Слева выберите **Secrets and variables** -> **Actions**.
3. Нажмите **New repository secret**.
4. Добавьте имя (например, `DOCKER_PASSWORD`) и значение (ваш токен).

В логах GitHub Actions эти значения будут автоматически заменены на `***`, даже если вы случайно попытаетесь их вывести через `echo`.

## Антипаттерны: Как не надо делать CI/CD

Даже автоматизацию можно сделать плохо. Избегайте этих ошибок:

1. **"Пропускать тесты, когда спешим":** Пайплайн без тестов — это просто автоматизированный способ сломать всё быстрее.
2. **Хардкод секретов:** Как мы уже обсудили, это путь к взлому.
3. **Слишком длинные пайплайны:** Если сборка идет 40 минут, разработчики будут реже пушить код. Оптимизируйте кэширование зависимостей.
4. **Отсутствие уведомлений:** Если пайплайн упал, а об этом никто не узнал (в Slack/Telegram), толку от него мало.
5. **Нестабильные (Flaky) тесты:** Тесты, которые падают случайным образом, убивают доверие к CI. Либо чините их, либо удаляйте.

## Обзор альтернативных систем

GitHub Actions — не единственный игрок на поле.

| Инструмент | Особенности | Идеально для... |
|------------|-------------|-----------------|
| **GitHub Actions** | Облачный, встроенный, огромный маркетплейс. | Быстрого старта и проектов на GitHub. |
| **GitLab CI** | Встроен в GitLab, очень гибкий (YAML includes, child pipelines). | Корпораций, использующих GitLab self-hosted. |
| **Jenkins** | Классика. Self-hosted, бесконечное количество плагинов. | Сложных Legacy-систем, где нужна кастомная логика. |
| **CircleCI** | Очень быстрые облачные билды, отличный UI. | Стартапов, которым важна скорость "time-to-market". |
