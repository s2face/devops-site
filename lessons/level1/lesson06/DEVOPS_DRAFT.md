# Урок 6 — Введение в Git: базовые команды

## 1. Зачем нужна система контроля версий (VCS)?
Представьте ситуацию: вы работаете над проектом и сохраняете копии папок:
* `project_v1`
* `project_v2`
* `project_v2_final`
* `project_v2_final_v2`
* `project_v2_final_v2_FIXED.zip`

Это путь к хаосу. Вы не помните, что изменилось, когда и зачем.
**Git** решает эту проблему, позволяя:
* Откатываться к любой точке в прошлом.
* Видеть историю изменений (кто, что и когда изменил).
* Работать над одной задачей командой, не затирая код друг друга.

---

## 2. Установка и базовая конфигурация
После установки Git (через `apt`, `brew` или скачивание с официального сайта), первое, что нужно сделать — представиться системе.

```bash
# Установка имени пользователя
git config --global user.name "Your Name"

# Установка почты
git config --global user.email "your_email@example.com"

# Проверка настроек
git config --list
```

---

## 3. Основные концепции: Области Git
Git разделяет рабочее пространство на три логические зоны:

1.  **Working Directory (Рабочая директория):** Ваши файлы "на диске", которые вы редактируете в данный момент.
2.  **Staging Area (Index / Область индексации):** "Черновик" для следующего коммита. Сюда попадают файлы после `git add`.
3.  **Repository (Репозиторий / .git):** Постоянное хранилище всех версий и истории.

---

## 4. Состояния файлов
Файлы в Git проходят через жизненный цикл:
*   **Untracked (Неотслеживаемый):** Git видит файл, но не следит за его изменениями.
*   **Modified (Изменен):** Файл уже был в репозитории, вы его изменили, но еще не добавили в индекс.
*   **Staged (Индексирован):** Изменения добавлены в Staging Area и готовы стать частью коммита.
*   **Committed (Зафиксирован):** Изменения надежно сохранены в базе данных Git.

## 5. Базовые команды

### Инициализация и клонирование
```bash
# Создать новый репозиторий в текущей папке
git init

# Склонировать существующий проект из сети
git clone https://github.com/user/repo.git
```

### Работа с изменениями
```bash
# Проверить статус файлов (в какой они зоне)
git status

# Добавить файл в Staging Area
git add filename.txt

# Добавить ВСЕ изменения
git add .

# Интерактивное добавление (позволяет выбрать конкретные куски кода — "hunks")
git add -p

# Фиксация изменений (создание слепка)
git commit -m "feat: add login functionality"

# Просмотр истории коммитов
git log --oneline --graph --all
```

### Анализ различий
```bash
# Посмотреть изменения в Working Directory по сравнению с индексом
git diff

# Посмотреть изменения, которые уже в индексе (готовы к коммиту)
git diff --staged
```

### Синхронизация с удаленным сервером
```bash
# Отправить свои коммиты на сервер
git push origin main

# Забрать чужие изменения и объединить со своими
git pull origin main
```

## 6. Игнорирование файлов (.gitignore)
Не все файлы должны попадать в репозиторий (логи, пароли, папки зависимостей вроде `node_modules`). Для этого используется файл `.gitignore`.

**Примеры паттернов:**
```text
# Игнорировать конкретный файл
config.json

# Игнорировать все файлы .log
*.log

# Игнорировать целую папку
/temp/
node_modules/

# Исключение: НЕ игнорировать этот файл, даже если папка в игноре
!/temp/important.txt
```

## 7. SSH-ключи для GitHub
Чтобы не вводить пароль при каждом `push`, используются SSH-ключи.

1.  **Генерация ключа:**
    ```bash
    ssh-keygen -t ed25519 -C "your_email@example.com"
    ```
2.  **Копирование публичного ключа:**
    `cat ~/.ssh/id_ed25519.pub`
3.  **Добавление в GitHub:**
    Settings -> SSH and GPG keys -> New SSH key.

Теперь вы можете клонировать через SSH: `git@github.com:user/repo.git`.
