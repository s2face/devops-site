## Что такое Dockerfile и зачем он нужен?

Представьте, что вы скачали новый кулинарный рецепт. В нем написано: «Возьмите кастрюлю, налейте воду, добавьте соль, доведите до кипения, положите макароны». Если вы точно следуете инструкции, результат будет предсказуемым.

**Dockerfile** — это именно такой рецепт для создания Docker-образа. Это обычный текстовый файл, содержащий последовательность команд, которые Docker выполняет одну за другой, чтобы собрать готовую «систему в коробке».

Почему нельзя просто настроить контейнер вручную и сохранить его?
1. **Воспроизводимость:** Dockerfile позволяет любому человеку в любой точке мира собрать точно такой же образ, как у вас.
2. **Автоматизация:** CI/CD системы (автоматическая сборка и деплой) используют Dockerfile для сборки новых версий приложения при каждом изменении кода.
3. **Версионирование:** Dockerfile хранится в Git вместе с кодом. Вы всегда видите, как менялось окружение вашего приложения.

## Подготовка приложения

Прежде чем писать Dockerfile, нам нужно что-то, что мы будем упаковывать. Давайте создадим простое веб-приложение на языке Python с использованием фреймворка Flask. Оно будет выводить приветствие и версию приложения.

Создайте папку для проекта, например `my-docker-app`, и внутри неё два файла.

> **Примечание:** Если вы захотите проверить код локально до сборки образа, вам нужно будет установить зависимости командой `pip install -r requirements.txt`. Но это не обязательно — Docker сделает всё сам внутри контейнера.

### 1. Файл `app.py`
Это основной код нашего приложения.

```python
from flask import Flask
import os

app = Flask(__name__)

@app.route('/')
def hello():
    # Мы берем версию из переменной окружения, которую позже зададим в Dockerfile
    version = os.getenv('APP_VERSION', '1.0.0')
    return f"""
    <html>
        <head><title>Docker Lesson 14</title></head>
        <body>
            <h1>Привет от Docker!</h1>
            <p>Это приложение работает внутри контейнера.</p>
            <p><b>Версия приложения:</b> {version}</p>
        </body>
    </html>
    """

if __name__ == '__main__':
    # Важно: внутри контейнера приложение должно слушать на 0.0.0.0, 
    # чтобы быть доступным извне. Порт выберем 5000.
    app.run(host='0.0.0.0', port=5000)
```

### 2. Файл `requirements.txt`
Здесь перечислены библиотеки, которые нужно установить для работы приложения.

```text
flask==3.0.0
```

Теперь у нас есть «сырье». Пора готовить инструкцию по сборке.

## Основные инструкции Dockerfile

Создайте в корне проекта файл с именем `Dockerfile` (без расширения, с заглавной буквы). Давайте напишем базовую версию и разберем каждую строку.

```dockerfile
# 1. Базовый образ
FROM python:3.11-slim

# 2. Рабочая директория
WORKDIR /app

# 3. Переменные окружения
ENV APP_VERSION=1.0.1
ENV PYTHONDONTWRITEBYTECODE 1
ENV PYTHONUNBUFFERED 1

# 4. Копируем список зависимостей
COPY requirements.txt .

# 5. Установка зависимостей
RUN pip install --no-cache-dir -r requirements.txt

# 6. Копируем весь остальной код приложения
COPY . .

# 7. Документируем порт
EXPOSE 5000

# 8. Команда запуска
CMD ["python", "app.py"]
```

Теперь подробно разберем каждую магическую команду.

### FROM — фундамент образа
Любой Dockerfile **обязательно** начинается с инструкции `FROM`. Она указывает, какой образ взять за основу. 
*   **Плохо:** `FROM ubuntu:latest`. Образ Ubuntu тяжелый (около 70-100 МБ), и вам придется вручную устанавливать Python, pip и кучу библиотек. Это увеличивает время сборки и вероятность ошибок.
*   **Хорошо:** `FROM python:3.11-slim`. Это официальный образ, где Python уже установлен. Префикс `-slim` означает «облегченная версия» — в ней удалены лишние файлы, документация и инструменты сборки, которые не нужны для работы приложения. Это экономит место.

**Slim vs Alpine: что выбрать?**
Вы часто можете встретить образы с тегом `alpine`, например `python:3.11-alpine`. Образы Alpine базируются на сверхлегком дистрибутиве Alpine Linux и весят всего по 5-10 МБ. Однако для Python-разработки с Alpine нужно быть осторожным. Он использует библиотеку `musl` вместо стандартной `glibc`, из-за чего некоторые популярные библиотеки (например, `pandas` или `numpy`) могут не устанавливаться или работать медленнее. Поэтому для новичков `slim` — идеальный баланс между размером и стабильностью.

**Совет:** Никогда не используйте тег `:latest` в реальных проектах. Если сегодня `latest` — это Python 3.11, а завтра выйдет 3.12, ваша сборка может сломаться из-за несовместимости библиотек. Всегда фиксируйте версию, например `3.11-slim`.

### WORKDIR — рабочее пространство
Инструкция `WORKDIR` создает папку внутри образа и делает её текущей для всех последующих команд (`RUN`, `COPY`, `CMD`). 
Аналог в Linux: `mkdir /app && cd /app`.
Использовать `WORKDIR` гораздо лучше, чем писать `RUN mkdir /app` и потом везде использовать абсолютные пути. Это делает Dockerfile чище и понятнее. Если вы не укажете `WORKDIR`, Docker будет выполнять команды в корневой директории `/`, что считается плохим тоном и может привести к конфликтам с системными файлами.

### ENV — переменные окружения
`ENV` задает переменные, которые будут доступны вашему приложению во время работы. 
В нашем примере:
*   `APP_VERSION`: используется в коде Python.
*   `PYTHONDONTWRITEBYTECODE 1`: запрещает Python создавать файлы `.pyc` (кэш компиляции), которые только занимают место в контейнере.
*   `PYTHONUNBUFFERED 1`: заставляет Python выводить логи в консоль сразу, не буферизируя их. Это критически важно для отладки — вы будете видеть ошибки в `docker logs` в реальном времени.

### COPY — перенос файлов и концепция Build Context
Команда `COPY <откуда_на_хосте> <куда_в_образе>` копирует файлы с вашего компьютера внутрь будущего образа.

Заметьте, мы сначала копируем только `requirements.txt`. Почему не всё сразу? Это связано с механизмом кэширования слоев, о котором мы поговорим чуть позже. Точка `.` в конце означает «копировать в текущую рабочую директорию» (которую мы задали через `WORKDIR`).

**Важно понимать, что такое Build Context (Контекст сборки).** Когда вы запускаете `docker build .`, точка в конце указывает на текущую папку. Docker архивирует всё содержимое этой папки и отправляет его «демону» Docker (фоновому процессу, который занимается сборкой). Если в этой папке лежат лишние тяжелые файлы (фильмы, бэкапы баз данных), сборка будет тормозить, даже если вы не копируете эти файлы в Dockerfile. Поэтому держите в папке с Dockerfile только то, что действительно относится к проекту.

### RUN — выполнение команд
`RUN` запускает команды внутри образа на этапе сборки. Обычно это установка пакетов, компиляция кода или создание папок. 
Здесь мы запускаем `pip install`. Флаг `--no-cache-dir` нужен, чтобы pip не сохранял скачанные архивы внутри образа — они нам там не понадобятся после установки, а место занимают.

### EXPOSE — документация портов vs реальный проброс
`EXPOSE 5000` сообщает Docker, что приложение внутри контейнера слушает порт 5000. 
**Важно:** Эта команда носит чисто информационный характер! Она не пробрасывает порты на ваш компьютер автоматически. Это «записка» для других инженеров и для самого Docker, чтобы он понимал, какой порт главный.
Чтобы реально получить доступ к приложению из браузера на вашем хосте, вы должны использовать флаг `-p` при запуске: `docker run -p 5000:5000 ...`. Это свяжет порт 5000 вашего компьютера с портом 5000 внутри контейнера.

### CMD и ENTRYPOINT — как запускать приложение?
Это самые важные и часто путаемые инструкции. Они определяют, какая команда выполнится, когда вы запустите контейнер командой `docker run`.

#### CMD (Command)
`CMD` задает команду по умолчанию.
```dockerfile
CMD ["python", "app.py"]
```
Если пользователь при запуске контейнера напишет что-то в конце, например:
`docker run myapp bash`
То `CMD` будет полностью проигнорирован, и запустится `bash`. Это удобно для отладки — вы можете зайти внутрь контейнера и проверить файлы.

#### ENTRYPOINT (Точка входа)
`ENTRYPOINT` делает контейнер похожим на исполняемый файл. Его сложнее переопределить.
Если вы напишете:
```dockerfile
ENTRYPOINT ["python"]
CMD ["app.py"]
```
То при запуске `docker run myapp` выполнится `python app.py`. 
А если вы напишете `docker run myapp test.py`, то выполнится `python test.py`. 
В этой связке `ENTRYPOINT` — это неизменяемая часть (бинарник), а `CMD` — аргументы по умолчанию, которые легко заменить.

**Shell vs Exec form:**
Существует две формы записи этих команд:
1. `CMD python app.py` (Shell form) — Docker запустит команду через `/bin/sh -c`. Это плохо, потому что ваше приложение не будет получать сигналы завершения (SIGTERM) напрямую, и `docker stop` будет работать долго.
2. `CMD ["python", "app.py"]` (Exec form) — Рекомендуемый способ. Приложение запускается напрямую как процесс №1. Это позволяет Docker корректно и быстро завершать работу контейнера.

## Слои и кэш — ускоряем сборку

Одна из самых крутых фишек Docker — это послойная структура образа. Каждая команда в Dockerfile (RUN, COPY, FROM) создает новый **слой**.

Представьте образ как стопку прозрачных листов. Каждый слой — это изменения в файловой системе. Когда вы собираете образ повторно, Docker проверяет: «Менялся ли файл `requirements.txt`? Нет? Значит, я могу не запускать `pip install` заново, а просто взять готовый слой из кэша!».

### Почему порядок имеет значение?
Docker кэширует слои сверху вниз. Как только один из слоев изменился (например, вы поменяли строчку в коде), **все последующие слои аннулируются** и пересобираются заново.

Именно поэтому мы в Dockerfile сделали так:
1. Скопировали `requirements.txt`.
2. Установили зависимости (`RUN pip install`).
3. И только в самом конце скопировали код приложения (`COPY . .`).

Зачем? Код приложения вы меняете постоянно. Если бы мы сначала копировали весь код, то при любом исправлении опечатки Docker пришлось бы заново скачивать и устанавливать все библиотеки из `pip install`. А так — библиотеки берутся из кэша за секунду, и обновляется только последний слой с кодом. 

**Золотое правило:** Сначала копируйте то, что меняется редко (зависимости, конфиги), и в самом конце — то, что меняется часто (код).

## .dockerignore — исключаем лишнее

Когда вы пишете `COPY . .`, Docker берет всё содержимое вашей папки и отправляет его «демону» сборки. Но вам точно не нужно копировать в образ:
*   Папку `.git` (она может весить сотни мегабайт).
*   Локальные виртуальные окружения (`venv`, `.env`).
*   Кэши Python (`__pycache__`).
*   Секретные ключи в файлах `.env`.
*   Документацию и README (внутри образа они не нужны).

Для этого создайте файл `.dockerignore` (по аналогии с `.gitignore`):

```text
# .dockerignore
__pycache__/
*.pyc
.git
.env
venv/
.venv/
Dockerfile
.dockerignore
```

Использование этого файла не только уменьшает размер образа, но и защищает вас от случайной утечки секретных данных в публичный образ.

## Multi-stage build: уменьшаем финальный образ

Иногда для сборки приложения нужны инструменты, которые не нужны для его работы. Например, для компиляции C++ библиотек в Python вам может понадобиться `gcc` и `build-essential`. Если вы просто установите их через `RUN apt install`, они останутся в образе навсегда, увеличивая его размер на 200-300 МБ.

**Multi-stage build** (многоэтапная сборка) позволяет использовать один образ для «стройки», а другой — для «жизни» приложения.

Давайте перепишем наш Dockerfile:

```dockerfile
# --- Этап 1: Сборка (Stage 1: Builder) ---
FROM python:3.11-slim as builder

WORKDIR /app

# Устанавливаем системные пакеты для сборки (если они нужны)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
# Устанавливаем зависимости в специальную папку
RUN pip install --no-cache-dir --user -r requirements.txt


# --- Этап 2: Финальный образ (Stage 2: Final) ---
FROM python:3.11-slim

WORKDIR /app

# Важно: переносим настройки окружения в финальный образ
ENV PYTHONDONTWRITEBYTECODE 1
ENV PYTHONUNBUFFERED 1
ENV APP_VERSION=1.0.1-multi

# Копируем только результат сборки из первого этапа
# Мы забираем установленные библиотеки из папки /root/.local первого образа
COPY --from=builder /root/.local /root/.local
# Копируем код приложения
COPY . .

# Настраиваем переменные окружения, чтобы Python видел скопированные библиотеки
ENV PATH=/root/.local/bin:$PATH

EXPOSE 5000

CMD ["python", "app.py"]
```

### В чем магия?
1. В первом этапе (`as builder`) мы можем устанавливать любые тяжелые инструменты, компиляторы и скачивать гигабайты исходников.
2. Во втором этапе мы начинаем с чистого листа (с чистого `python:3.11-slim`). **Обратите внимание:** настройки `ENV` из первого этапа не переносятся автоматически, поэтому мы задаем их заново.
3. С помощью команды `COPY --from=builder` мы переносим из первого образа только то, что нам реально нужно — готовые скомпилированные библиотеки.
4. Весь «мусор» (компиляторы, кэши) остается в первом образе и **не попадает** в финальный результат.

**Сравнение размеров:**
*   Обычная сборка с инструментами: ~450 МБ.
*   Multi-stage сборка: ~120 МБ.
Разница очевидна! Меньше образ — быстрее деплой, меньше расходов на хранилище и выше безопасность (меньше инструментов, которые может использовать хакер).

## Пошаговая сборка и запуск

Пора проверить наш Dockerfile в деле. Откройте терминал в папке проекта.

### 1. Сборка образа
```bash
docker build -t myapp:1.0 .
```
*   `-t myapp:1.0`: задает имя (тег) образу.
*   **`.` (точка в конце)**: это критически важная часть команды. Она указывает путь к контексту сборки (текущая папка). Без неё Docker не поймет, где искать Dockerfile и файлы вашего приложения.

### 2. Запуск контейнера
```bash
docker run -d -p 5000:5000 --name my-container myapp:1.0
```
*   `-d`: запуск в фоновом режиме.
*   `-p 5000:5000`: связываем порт 5000 вашего компьютера с портом 5000 внутри контейнера.
*   `--name`: даем контейнеру красивое имя.

Теперь откройте в браузере `http://localhost:5000`. Поздравляю! Ваше приложение работает внутри Docker.

## Публикуем образ на Docker Hub

Ваш образ готов, но пока он живет только на вашем компьютере. Чтобы развернуть его на сервере или поделиться с коллегами, нужно отправить его в реестр (Registry). Самый популярный — Docker Hub.

### Шаг 1: Регистрация и логин
Если у вас еще нет аккаунта, зарегистрируйтесь на [hub.docker.com](https://hub.docker.com/).
Затем выполните в терминале:
```bash
docker login
```
Введите свой логин и пароль.

### Шаг 2: Тегирование
Docker Hub ожидает, что имя образа будет содержать ваш логин. Если ваш логин на Docker Hub — `ВАШ_ЛОГИН`, а образ называется `myapp`, нужно создать тег с вашим логином:
```bash
# Замените <ВАШ_ЛОГИН> на ваш реальный логин Docker Hub
docker tag myapp:1.0 <ВАШ_ЛОГИН>/myapp:1.0
```

### Шаг 3: Push (Отправка)
Теперь отправляем образ в облако:
```bash
docker push <ВАШ_ЛОГИН>/myapp:1.0
```
После завершения вы увидите свой образ в личном кабинете на Docker Hub. Теперь любой человек в мире может запустить ваше приложение командой `docker run <ВАШ_ЛОГИН>/myapp:1.0`.

## Антипаттерны: как делать НЕ надо

Чтобы быть крутым специалистом, избегайте этих ошибок:
1. **Запуск от root:** По умолчанию Docker запускает процессы от имени суперпользователя. В реальных проектах это небезопасно. Лучше создавать отдельного пользователя через `RUN useradd`.
2. **Секреты в Dockerfile:** Никогда не пишите пароли в инструкции `ENV`. Их сможет увидеть любой, кто сделает `docker inspect`. Используйте `.env` файлы или секреты Kubernetes.
3. **Огромные слои:** Старайтесь объединять команды, которые логически связаны. Например:
    `RUN apt update && apt install -y curl && rm -rf /var/lib/apt/lists/*`. 
    Это создаст один слой вместо трех и сразу удалит временные файлы.
4. **Отсутствие .dockerignore:** Не тащите за собой гигабайты мусора из папки `.git`.

## Ключевые выводы урока

1.  **Dockerfile = рецепт образа:** текстовый файл с инструкциями для сборки Docker-образа; обеспечивает воспроизводимость и автоматизацию.
2.  **FROM задаёт базовый образ:** используйте конкретные версии (`python:3.11-slim`), а не `:latest`; `slim` стабильнее `alpine` для Python.
3.  **Порядок инструкций важен для кэша:** сначала копируйте `requirements.txt` и ставьте зависимости, потом — код приложения.
4.  **COPY vs RUN:** `COPY` копирует файлы с хоста в образ, `RUN` выполняет команды внутри образа при сборке.
5.  **CMD vs ENTRYPOINT:** `CMD` — команда по умолчанию (легко переопределить), `ENTRYPOINT` — неизменяемая часть (бинарник).
6.  **Exec form предпочтительнее:** `CMD ["python", "app.py"]` вместо `CMD python app.py` — корректная обработка SIGTERM.
7.  **`.dockerignore` защищает от мусора:** исключает `.git`, `__pycache__`, `.env`, `venv/` из контекста сборки.
8.  **Multi-stage build уменьшает образ:** сборка в одном этапе, запуск в другом; экономия 300+ МБ за счёт удаления компиляторов.

## Глоссарий терминов урока

| Термин | Расшифровка | Простое объяснение |
|--------|-------------|-------------------|
| **Dockerfile** | — | Текстовый файл с инструкциями для сборки Docker-образа. |
| **FROM** | — | Инструкция задания базового образа для сборки. |
| **WORKDIR** | Working Directory | Инструкция установки рабочей директории внутри образа. |
| **ENV** | Environment | Инструкция задания переменных окружения в образе. |
| **COPY** | — | Инструкция копирования файлов с хоста в образ. |
| **RUN** | — | Инструкция выполнения команд внутри образа при сборке. |
| **EXPOSE** | — | Инструкция документирования порта (не пробрасывает его!). |
| **CMD** | Command | Инструкция задания команды по умолчанию для запуска контейнера. |
| **ENTRYPOINT** | Точка входа | Инструкция задания неизменяемой части команды запуска. |
| **Build Context** | Контекст сборки | Папка, которую Docker отправляет демону для сборки образа. |
| **slim** | — | Облегчённая версия образа Debian (удалены лишние пакеты). |
| **Alpine** | — | Сверхлёгкий дистрибутив Linux на базе musl libc (~5 МБ). |
| **musl** | — | Альтернативная реализация стандартной библиотеки C (легче glibc). |
| **glibc** | GNU C Library | Стандартная библиотека C в большинстве дистрибутивов Linux. |
| **Shell form** | Форма оболочки | Запись CMD через `/bin/sh -c` (проблемы с SIGTERM). |
| **Exec form** | Исполняемая форма | Запись CMD в JSON-массиве (рекомендуемый способ). |
| **SIGTERM** | Signal Terminate | Сигнал процесса для корректного завершения работы. |
| **Multi-stage build** | Многоэтапная сборка | Использование нескольких FROM для уменьшения финального образа. |
| **Layers** | Слои | Уровни файловой системы образа; каждый кэшируется отдельно. |
| **docker build** | — | Команда сборки образа из Dockerfile. |
| **docker tag** | — | Команда присвоения образу нового имени/тега. |
| **docker push** | — | Команда отправки образа в реестр (Docker Hub). |
| **docker login** | — | Команда аутентификации в реестре образов. |
| **docker inspect** | — | Команда просмотра детальной информации об образе/контейнере. |
| **useradd** | — | Команда создания нового пользователя в Linux. |
| **Flask** | — | Лёгкий веб-фреймворк для Python. |
| **pip** | Pip Installs Packages | Менеджер пакетов для Python. |
| **--no-cache-dir** | — | Флаг pip для отказа от сохранения кэша загрузок. |
| **PYTHONDONTWRITEBYTECODE** | — | Переменная окружения, запрещающая создание `.pyc` файлов. |
| **PYTHONUNBUFFERED** | — | Переменная окружения для вывода логов в реальном времени. |
