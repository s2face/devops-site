# Урок 14: Первый Dockerfile — сборка и запуск приложения

В этом уроке мы научимся упаковывать приложение в Docker-образ, оптимизировать его размер и публиковать на Docker Hub. Мы разберем основные инструкции Dockerfile, механизм слоев и кэширования, а также продвинутую технику Multi-stage build.

## 1. Подготовка приложения (Flask)

Для примера создадим простое веб-приложение на Python с использованием фреймворка Flask.

### Файл `app.py`
```python
from flask import Flask
import os

app = Flask(__name__)

@app.route('/')
def hello():
    version = os.getenv('APP_VERSION', '1.0.0')
    return f"<h1>Hello from Docker!</h1><p>Version: {version}</p>"

if __name__ == '__main__':
    # Слушаем на всех интерфейсах (0.0.0.0) внутри контейнера
    app.run(host='0.0.0.0', port=5000)
```

### Файл `requirements.txt`
```text
flask==3.0.0
```

## 2. Написание Dockerfile

### Базовый Dockerfile (неоптимизированный)
Этот вариант прост в понимании, но не использует все возможности оптимизации.

```dockerfile
# 1. Базовый образ
FROM python:3.11-slim

# 2. Рабочая директория
WORKDIR /app

# 3. Переменные окружения
ENV APP_VERSION=1.0.1
ENV PYTHONDONTWRITEBYTECODE 1
ENV PYTHONUNBUFFERED 1

# 4. Копируем зависимости
COPY requirements.txt .

# 5. Установка зависимостей
RUN pip install --no-cache-dir -r requirements.txt

# 6. Копируем код приложения
COPY . .

# 7. Открываем порт
EXPOSE 5000

# 8. Команда запуска
CMD ["python", "app.py"]
```

### Оптимизированный Dockerfile (Multi-stage build)
Эта техника позволяет разделить процесс сборки (где нужны компиляторы и лишние инструменты) и финальный образ (где только приложение и зависимости).

```dockerfile
# --- Stage 1: Builder ---
FROM python:3.11-slim as builder

WORKDIR /app

# Установка системных зависимостей (если нужны для сборки пакетов)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
# Устанавливаем пакеты в локальную папку
RUN pip install --no-cache-dir --user -r requirements.txt


# --- Stage 2: Final ---
FROM python:3.11-slim

WORKDIR /app

# Копируем только установленные пакеты из первого этапа
COPY --from=builder /root/.local /root/.local
# Копируем код приложения
COPY . .

# Добавляем путь к установленным пакетам в PATH
ENV PATH=/root/.local/bin:$PATH
ENV APP_VERSION=1.0.1-multi

EXPOSE 5000

# Используем ENTRYPOINT + CMD для гибкости
ENTRYPOINT ["python"]
CMD ["app.py"]
```

Исключаем файлы, которые не должны попасть в образ (ускоряет сборку и уменьшает размер).


## 4. Пошаговые команды терминала

### Сборка образа
```bash
# Сборка из текущей директории с тегом 'myapp:1.0'
docker build -t myapp:1.0 .
```

### Запуск контейнера
```bash
# Запуск в фоновом режиме (-d) с пробросом портов 5000 -> 5000
docker run -d -p 5000:5000 --name my-flask-app myapp:1.0
```

### Публикация на Docker Hub
```bash
# 1. Логин в Docker Hub
docker login

# 2. Присвоение тега с вашим username
docker tag myapp:1.0 your_dockerhub_username/myapp:1.0

# 3. Отправка образа в репозиторий
docker push your_dockerhub_username/myapp:1.0
```

## 5. Технические пояснения

### Основные инструкции
*   **FROM**: Задает базовый образ. Всегда используйте специфичные теги (например, `3.11-slim` или `alpine`), чтобы избежать проблем с совместимостью в будущем.
*   **WORKDIR**: Устанавливает рабочую директорию. Если она не существует, Docker создаст её. Это чище, чем использовать `RUN mkdir`.
*   **COPY**: Копирует файлы с хоста в образ. Рекомендуется сначала копировать файлы зависимостей (например, `requirements.txt`), чтобы эффективно использовать кэш.
*   **RUN**: Выполняет команды на этапе сборки. Каждая инструкция `RUN` создает новый слой.
*   **ENV**: Устанавливает переменные окружения, доступные как во время сборки, так и при работе контейнера.
*   **EXPOSE**: Документирует порт, который приложение слушает внутри контейнера. Не пробрасывает порт автоматически!
*   **CMD**: Команда по умолчанию при запуске контейнера. Может быть переопределена при `docker run`.
*   **ENTRYPOINT**: Позволяет настроить контейнер как исполняемый файл. Обычно используется для указания основной команды, а `CMD` — для аргументов.

### Разница CMD vs ENTRYPOINT
*   **ENTRYPOINT** — это "что" запускается (например, `python`). Его сложно переопределить случайно.
*   **CMD** — это "аргументы" по умолчанию (например, `app.py`). Их легко заменить, просто добавив аргументы в конце команды `docker run`.
*   *Комбинация*: `ENTRYPOINT ["python"]` + `CMD ["app.py"]` позволяет запускать контейнер как `docker run myapp` (запустит `python app.py`) или `docker run myapp other_script.py` (запустит `python other_script.py`).

### Слои образа и кэширование
Каждая инструкция в Dockerfile (кроме метаданных) создает **слой**. Слои кэшируются.
*   Если вы изменили только `app.py`, но не `requirements.txt`, Docker пропустит шаг `pip install` и возьмет его из кэша.
*   **Важно:** Как только один слой изменился, все последующие слои будут пересобраны заново. Именно поэтому `COPY requirements.txt` стоит *перед* `COPY .`.

### Оптимизация и Multi-stage build
*   **Multi-stage** позволяет сократить размер финального образа в несколько раз. Мы используем тяжелый образ для сборки (с компиляторами и dev-пакетами), а затем копируем только результат работы в "чистый" легкий образ.
*   **Безопасность:** В финальном образе отсутствуют исходники сборки и лишние инструменты, что уменьшает поверхность атаки.
