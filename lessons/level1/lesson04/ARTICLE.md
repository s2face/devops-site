Linux — это многопользовательская система, изначально созданная для совместной работы. Десятки пользователей могут одновременно работать на одном сервере, не мешая друг другу и не повреждая чужие данные. Понимание системы разграничения прав — критический навык для любого системного администратора или DevOps-инженера. В этом уроке мы разберем, как Linux идентифицирует пользователей, управляет их доступом к ресурсам и обеспечивает безопасность через модель разрешений.

## Как Linux хранит информацию о пользователях

Для системы пользователь — это не имя, а цифровой идентификатор — **UID (User ID)**. Аналогично, группы идентифицируются через **GID (Group ID)**.

### Идентификаторы UID: Классификация
UID разделяются на диапазоны:
- **UID 0**: Суперпользователь `root`. В Linux используется модель **DAC** (избирательное управление доступом), но для `root` эти правила не действуют.
- **Системные пользователи (UID 1–999)**: Создаются для фоновых сервисов (например, `www-data`). Вход под ними обычно заблокирован оболочкой `/usr/sbin/nologin`.
- **Обычные пользователи (UID 1000+)**: Аккаунты для реальных людей.

### Основные конфигурационные файлы
Информация о пользователях хранится в текстовых файлах в `/etc/`:

1. **`/etc/passwd`**: Реестр пользователей. Формат: `логин:пароль:UID:GID:комментарий:дом_директория:оболочка`. Поле комментария (GECOS) часто используется для описания роли пользователя.
2. **`/etc/shadow`**: Хранит хеши паролей и информацию об их «старении». Файл доступен только root. Хеширование — это одностороннее преобразование, обеспечивающее безопасность при утечке файла.
3. **`/etc/group`**: Описывает группы и их участников. Формат: `имя:пароль:GID:список_пользователей`.

## Создание и управление пользователями

Для управления учетными записями в Linux существует набор мощных утилит. Все они требуют прав суперпользователя, поэтому мы предваряем их командой `sudo`.

### useradd vs adduser: Что выбрать?
Новички часто путаются между этими двумя командами.
- **`useradd`**: Это стандартная бинарная утилита, присутствующая практически в любом Linux-дистрибутиве. Она низкоуровневая, требует явного указания всех параметров через флаги и не задает лишних вопросов. **В DevOps-скриптах и автоматизации (Ansible, Terraform) используется именно она**, так как её поведение предсказуемо и не требует интерактивного ввода.
- **`adduser`**: Это высокоуровневый скрипт (обычно на Perl), который является «надстройкой» над `useradd` в системах Debian/Ubuntu. Он интерактивен: сам спросит пароль, полное имя и создаст домашнюю директорию по умолчанию. Это удобно для ручного управления на домашнем сервере, но плохо подходит для автоматизации.

Мы будем учиться работать с `useradd`, так как это универсальный инструмент профессионала.

### Добавление пользователя (`useradd`)
Чтобы создать полноценного пользователя одним махом, рекомендуется использовать флаги:
```bash
sudo useradd -m -c "John Doe" -s /bin/bash john
```
- `-m` (`--create-home`): Создает домашнюю директорию `/home/john` и копирует в нее начальные настройки из шаблона `/etc/skel` (скрытые файлы вроде `.bashrc`).
- `-c` (`--comment`): Позволяет задать описание (поле GECOS).
- `-s` (`--shell`): Назначает командную оболочку.

### Установка и управление паролем (`passwd` и `chage`)
После создания пользователя его аккаунт заблокирован. Установим пароль:
```bash
sudo passwd john
```
Чтобы заставить пользователя сменить пароль при первом входе, используйте:
```bash
sudo chage -d 0 john
```
Число `0` устанавливает дату последнего изменения на 1 января 1970 года. Система сочтет пароль устаревшим и потребует его обновления при логине. Это стандартный прием при выдаче доступов новым сотрудникам.

### Модификация пользователя (`usermod`)
Если нужно изменить параметры существующего пользователя, используется `usermod`. Например, заблокировать аккаунт можно так:
```bash
sudo usermod -L john  # Lock
```
А разблокировать так:
```bash
sudo usermod -U john  # Unlock
```

### Удаление пользователя (`userdel`)
При удалении важно решить, что делать с файлами пользователя:
```bash
sudo userdel -r john
```
Флаг `-r` удаляет домашнюю директорию и почтовый ящик. Без него файлы останутся в системе, но их владельцем будет числиться UID, которого больше не существует (в выводе `ls -l` вместо имени будет стоять число).

### Создание сервисного пользователя: практика DevOps
В мире DevOps мы постоянно создаем пользователей для приложений. Пример создания пользователя для базы данных:
```bash
sudo useradd -r -M -s /usr/sbin/nologin mongodb
```
- `-r`: создание системного пользователя.
- `-M`: не создавать домашнюю директорию (она не нужна базе данных).
- `-s /usr/sbin/nologin`: полная блокировка возможности входа.

## Группы и зачем они нужны

Механизм групп — это основной способ управления коллективным доступом. В Linux каждый пользователь обязательно принадлежит к одной **основной группе** (Primary Group, обычно с тем же именем, что и логин) и может принадлежать к неограниченному числу **дополнительных групп** (Supplementary Groups).

### Работа с группами
1. **Создание группы**:
   ```bash
   sudo groupadd devops
   ```
2. **Добавление пользователя в группу**:
   Это самая частая операция. Используйте `usermod` с флагами `-aG`:
   ```bash
   sudo usermod -aG devops john
   ```
   **Важно!** Флаг `-a` (append) критически важен. Если его пропустить, команда `usermod -G devops john` удалит пользователя из всех текущих дополнительных групп (например, из группы `docker` или `sudo`) и оставит только в `devops`. Это может привести к потере доступа к важным инструментам.

3. **Просмотр групп пользователя**:
   Чтобы проверить, в каких группах вы состоите, используйте команду `id` или `groups`:
   ```bash
   id john
   ```

## Права доступа: rwx и цифровая нотация

Теперь перейдем к самому важному — как именно Linux разрешает или запрещает доступ к файлам. Каждый объект в файловой системе (файл, папка, ссылка) имеет три набора прав:
- **u (user)**: Права для владельца файла.
- **g (group)**: Права для группы, связанной с файлом.
- **o (others)**: Права для всех остальных пользователей в системе.

### Значения r, w, x

Смысл этих букв различается для файлов и директорий, что часто путает новичков. Понимание этого различия — ключ к безопасности вашей системы.

**Для файлов:**
- **r (read)**: Можно читать содержимое (например, командой `cat` или `less`).
- **w (write)**: Можно изменять содержимое файла (но не удалять сам файл, если у вас нет прав на папку!).
- **x (execute)**: Файл можно запустить как программу или скрипт.

**Для директорий:**
- **r (read)**: Позволяет просмотреть список файлов в папке (команда `ls`). Но без права `x` вы увидите только имена файлов, без их атрибутов (прав, владельца, размера).
- **w (write)**: Позволяет создавать, удалять и переименовывать файлы внутри папки. 
  > **ВНИМАНИЕ!** Это самый опасный момент. Если у вас есть право `w` на папку, вы можете удалить любой файл внутри неё (даже если этот файл принадлежит `root` и у вас нет прав на его чтение!). Это часто шокирует новичков, но такова логика Unix: удаление файла — это изменение *содержимого папки*, а не самого файла.
- **x (execute)**: Это право на «проход» через директорию. Без него вы не сможете сделать `cd` в эту папку и не сможете обратиться к любому файлу внутри неё, даже если знаете точное имя.

#### Небольшой эксперимент: Капризный бит `x`
Представьте ситуацию: у вас есть папка `/data` и файл `/data/secret.txt`.
1. У папки права `r--` (только чтение, нет `x`). 
   - Вы делаете `ls /data`. Результат: вы видите имя `secret.txt`.
   - Вы делаете `cat /data/secret.txt`. Результат: **Ошибка «Permission denied»**. Несмотря на то, что у вас есть права на чтение самого файла, вы не смогли «пройти» сквозь папку `/data` к нему.
2. У папки права `--x` (только выполнение, нет `r`).
   - Вы делаете `ls /data`. Результат: **Ошибка «Permission denied»**. Вы не можете получить список файлов.
   - Вы делаете `cat /data/secret.txt`. Результат: **Файл прочитан успешно!** Если вы точно знаете путь, вы можете «проскочить» в папку и забрать файл, хотя «оглядеться» внутри папки вам не позволили.

### Разбор вывода `ls -la`
Команда `ls -la` — ваш главный инструмент диагностики прав.
```text
-rwxr-x--- 1 john developers 4096 Oct 12 14:30 deploy.sh
```
1. Первый символ `-`: Это тип объекта. `-` — обычный файл, `d` — директория, `l` — символическая ссылка.
2. Следующие 9 символов `rwxr-x---`: Это три тройки прав.
   - `rwx`: Владелец (User) может всё.
   - `r-x`: Группа (Group) может читать и запускать, но не менять.
   - `---`: Остальные (Others) не могут абсолютно ничего.
3. Число `1`: Количество жестких ссылок на файл.
4. `john`: Имя владельца.
5. `developers`: Группа-владелец.
6. `4096`: Размер файла в байтах.
7. `Oct 12 14:30`: Дата и время последнего изменения.
8. `deploy.sh`: Имя файла.

### Математика прав (Октальная нотация)
Системные администраторы чаще используют цифры. Каждому праву соответствует число (степень двойки):
- **Read (r) = 4**
- **Write (w) = 2**
- **Execute (x) = 1**

| Число | Описание | Буквы |
| :--- | :--- | :--- |
| **7** | Полный доступ (4+2+1) | `rwx` |
| **6** | Чтение и запись (4+2) | `rw-` |
| **5** | Чтение и выполнение (4+1) | `r-x` |
| **4** | Только чтение | `r--` |
| **3** | Запись и выполнение (2+1) | `-wx` |
| **2** | Только запись | `-w-` |
| **1** | Только выполнение | `--x` |
| **0** | Нет прав | `---` |

#### Топ-5 самых частых прав (запомните их!):
1. **644**: Стандарт для обычных файлов (владелец правит, все читают).
2. **755**: Стандарт для директорий и исполняемых скриптов (все могут войти и прочитать, владелец может менять).
3. **600**: Секретные файлы (только владелец видит и меняет, например, ключи SSH или конфиги с паролями).
4. **700**: Секретные директории (только владелец может войти).
5. **400**: «Только для чтения» для владельца (чтобы случайно не затереть важный конфиг).

## chmod и chown на практике

### Изменение прав (`chmod`)
Команда `chmod` (change mode) поддерживает два синтаксиса.
- **Символьный**: `chmod g+w file` (добавить запись группе), `chmod o-r file` (забрать чтение у остальных).
- **Числовой**: `chmod 755 script.sh` (стандарт для скриптов), `chmod 600 id_rsa` (стандарт для секретных ключей SSH).

### Изменение владельца (`chown`)
Команда `chown` (change owner) позволяет менять и пользователя, и группу:
```bash
sudo chown -R www-data:www-data /var/www/html
```
Если же вам нужно сменить **только** группу, можно воспользоваться командой `chgrp`:
```bash
sudo chgrp developers report.txt
```
Флаг `-R` (recursive) — самый часто используемый в DevOps. Он применяет изменения ко всей вложенной структуре папок и файлов.

### Что такое `umask`?
Новые файлы по умолчанию создаются с правами `644`, а папки — `755`. Это работа `umask` (user mask) — значения, которое «вычитается» из максимальных прав (666 для файлов и 777 для папок).

Проверить текущую маску можно командой `umask`. Если она равна `0022`, то новые файлы получат права `666 - 022 = 644`. Настроить маску можно в файлах `.bashrc` или `/etc/profile`.

### Важные нюансы команд
В Linux регистр флагов имеет критическое значение.
- **`chmod -R` и `chown -R`**: Флаг рекурсии должен быть **заглавным**.
- **`cp -r`**: Здесь чаще используется строчная `r`.
Новички часто путают их, что приводит к ошибкам «Unknown option». Просто запомните: для прав и владельцев — большая **-R**.

### Антипаттерн: `chmod 777`
Никогда не используйте `chmod 777` как способ «просто заставить это работать». Это признак лени и плохой квалификации. В продакшене это создает огромный риск: любой процесс, захваченный злоумышленником, сможет подменить ваш конфиг или внедрить вредоносный код в ваш скрипт. Всегда стремитесь к принципу наименьших привилегий.

## sudo — привилегированный доступ без root

Работать в терминале напрямую под пользователем `root` — это как ходить по минному полю в темноте. Ошибка в один лишний пробел в команде `rm -rf / tmp/my_app` (вместо `/tmp/my_app`) приведет к полному уничтожению системы.
Именно поэтому в Linux принята концепция `sudo` (Substitute User and Do).

### Механизм работы
Когда вы выполняете `sudo command`, происходит следующее:
1. Программа проверяет настройки в файле `/etc/sudoers`.
2. Она запрашивает **ваш** личный пароль (чтобы подтвердить, что за клавиатурой именно вы, а не кто-то отошедший попить кофе).
3. Если всё верно, команда выполняется с правами root (или другого указанного пользователя).

### Настройка `/etc/sudoers` и `visudo`
Файл `sudoers` имеет специфический синтаксис:
`john ALL=(ALL:ALL) NOPASSWD: ALL`
Эта строка позволила бы пользователю john выполнять любые команды без запроса пароля (удобно для автоматизации скриптов, но опасно для живых людей).

**Золотое правило**: Никогда не редактируйте этот файл командами `nano` или `vim` напрямую. Всегда используйте команду `sudo visudo`. Она открывает файл во временном буфере и перед сохранением проверяет его на синтаксические ошибки. Если вы допустите ошибку в `sudoers` обычным редактором, вы можете лишиться возможности использовать `sudo` и "запереть" себя снаружи административного доступа.

## Специальные биты (SUID, SGID, Sticky)

Когда стандартной модели `rwx` недостаточно, на помощь приходят три специальных бита.

### 1. SUID (Set User ID)
Позволяет запускать файл с правами его **владельца**, а не текущего пользователя.
**Пример**: `/usr/bin/passwd` принадлежит root. Обычный пользователь может сменить свой пароль только благодаря SUID, временно получая права root для записи в `/etc/shadow`.
- Вид: `rwsr-xr-x` (`s` у владельца).
- Установка: `chmod u+s file`.

### 2. SGID (Set Group ID)
- **Для файлов**: работает как SUID, но дает права группы файла.
- **Для директорий**: заставляет новые файлы наследовать группу папки, а не пользователя. Это идеально для совместной работы в общих папках.
- Вид: `rwxr-sr-x` (`s` у группы).
- Установка: `chmod g+s dir`.

### 3. Sticky Bit (Липкий бит)
Используется в общих папках (например, `/tmp`). Разрешает удалять файлы только их владельцам, даже если у всех есть права на запись в папку.
- Вид: `rwxrwxrwt` (`t` в конце).
- Установка: `chmod +t dir`.

## Заключение: Золотые правила безопасности Linux

1. **Принцип наименьших привилегий**: Всегда давайте пользователю или процессу ровно столько прав, сколько нужно для работы.
2. **Забудьте про вход под root**: Используйте `sudo` для разовых административных задач.
3. **Группы — это сила**: Используйте их для управления коллективным доступом, а не открывайте права «для всех».
4. **Контролируйте SUID**: Программы с SUID — потенциальные цели для атак. Регулярно проверяйте их список.
5. **Документируйте изменения**: Любые ручные правки прав системных папок должны быть зафиксированы, иначе система может стать неработоспособной после обновления.

Освоение прав доступа — это переход на уровень профессионального инженера. Концепции Unix-прав остаются неизменными десятилетиями. Практикуйтесь в «песочницах», ломайте и чините доступы — это лучший способ усвоить материал!

## Ключевые выводы урока

1.  **UID 0 = root.** Никогда не работайте под root постоянно — используйте `sudo`.
2.  **`useradd` для скриптов, `adduser` для людей.** Первый предсказуем в автоматизации.
3.  **`-aG` при добавлении в группы.** Без `-a` пользователь удаляется из других групп.
4.  **`w` на папку = можно удалять файлы.** Даже чужие, даже если нет прав на файл.
5.  **`x` на директорию = можно войти.** Без `x` вы не сможете `cd` в папку.
6.  **644 для файлов, 755 для папок.** Стандарт безопасности по умолчанию.
7.  **Никогда не используйте `chmod 777`.** Это дыра в безопасности.
8.  **`visudo` для правки sudoers.** Прямое редактирование может заблокировать `sudo`.

## Глоссарий терминов урока

| Термин | Расшифровка | Простое объяснение |
|--------|-------------|-------------------|
| **UID** | User ID | Цифровой идентификатор пользователя |
| **GID** | Group ID | Цифровой идентификатор группы |
| **DAC** | Discretionary Access Control | Избирательное управление доступом |
| **GECOS** | — | Поле с комментарием о пользователе (имя, телефон) |
| **SUID** | Set User ID | Бит, запускающий файл с правами владельца |
| **SGID** | Set Group ID | Бит, запускающий файл с правами группы |
| **Sticky Bit** | — | Бит, запрещающий удалять чужие файлы в общей папке |
| **Umask** | User Mask | Маска, определяющая права новых файлов |
| **NOPASSWD** | — | Разрешение выполнять команды без пароля в sudoers |
