# Bash-скрипты: автоматизация рутины для DevOps

Bash (Bourne Again SHell) — это стандартный командный интерпретатор в большинстве дистрибутивов Linux. Для DevOps-инженера Bash является "швейцарским ножом" для автоматизации рутины: от простых бэкапов до сложных CI/CD пайплайнов. В этой статье мы пройдем путь от создания первого скрипта "Hello World" до написания надежных инструментов мониторинга и резервного копирования, которые можно использовать в реальном продакшене.

## Введение: Почему Bash — это фундамент автоматизации

В мире, где существуют Ansible, Terraform и Kubernetes, новички часто задаются вопросом: "Зачем мне учить Bash?". Ответ прост: Bash — это клей, который соединяет все эти инструменты. Когда вы пишете Dockerfile, вы используете команды Bash. Когда вы настраиваете CI/CD пайплайны в GitLab или GitHub Actions, вы пишете скрипты на Bash. 

Bash доступен практически на любом Linux-сервере "из коробки", не требует установки тяжелых интерпретаторов (как Python или Ruby) и идеально подходит для задач, связанных с манипуляцией файлами, системными процессами и текстовыми потоками.

## Первый скрипт — Hello World и права доступа

Любой путь в программировании начинается с "Hello World". В Bash это выглядит максимально просто, но даже здесь есть важные нюансы.

**Важное замечание:** В этой статье мы используем возможности именно Bash (например, двойные скобки `[[ ]]` и массивы). Если вы попытаетесь запустить эти скрипты через стандартный командный интерпретатор `sh` (который часто является ссылкой на более простой Dash), они могут выдать ошибку. Всегда используйте `bash`.

Создайте файл `hello.sh`:
```bash
#!/bin/bash
echo "Hello, DevOps World!"
```

### Что такое Shebang (`#!`)?
Первая строка `#!/bin/bash` называется **shebang**. Она указывает операционной системе, какой именно интерпретатор должен выполнять этот файл. Если вы напишете `#!/usr/bin/python3`, система попытается выполнить скрипт с помощью Python. Для Bash-скриптов всегда используйте `/bin/bash` или более универсальный вариант `#!/usr/bin/env bash`.

### Права на выполнение
Просто создать файл недостаточно. По умолчанию у новых файлов нет прав на выполнение. Попробуйте запустить скрипт как `./hello.sh`, и вы получите ошибку:

```bash
$ ./hello.sh
bash: ./hello.sh: Permission denied
```

Чтобы сделать скрипт исполняемым, используйте команду `chmod`:
```bash
chmod +x hello.sh
./hello.sh
```
Теперь ваша система знает, что этот текстовый файл можно запускать как программу.

## Переменные и подстановка команд

Переменные позволяют нам хранить данные и использовать их многократно. В Bash нет строгой типизации, и объявление переменной выглядит так: `NAME="Значение"`.

**Важно:** Между именем переменной, знаком равенства и значением не должно быть пробелов!
*   `VAR = "value"` — Ошибка.
*   `VAR="value"` — Правильно.

### `$VAR` vs `${VAR}`: В чем разница?
Часто вы будете видеть оба варианта. В большинстве случаев они взаимозаменяемы, но фигурные скобки `${}` необходимы в следующих ситуациях:
1.  **Избежание двусмысленности:** Если вы хотите добавить текст сразу после переменной: `echo "File_${NAME}_backup.txt"`. Без скобок Bash будет искать переменную `NAME_backup`.
2.  **Работа с массивами:** `${SERVERS[0]}`.
3.  **Манипуляции со строками:** Например, задание значения по умолчанию: `${VAR:-"default_value"}`.

### Подстановка команд (Command Substitution)
Одна из самых мощных функций Bash — возможность сохранить результат выполнения команды в переменную. Для этого используется синтаксис `$(команда)`.

```bash
DISK_USAGE=$(df -h / | awk 'NR==2{print $5}')
echo "Диск занят на: $DISK_USAGE"
```

В этом примере:
1.  `df -h /` выводит информацию о свободном месте на корневом диске в человекочитаемом виде.
2.  `awk 'NR==2{print $5}'` анализирует вывод команды `df`:
    *   `NR==2` (Number of Row) означает "выбрать только вторую строку" (пропускаем заголовок).
    *   `{print $5}` указывает на пятый столбец (в данном случае — процент использования диска). 
    *   Столбцы в `awk` разделяются пробелами по умолчанию.
3.  Результат сохраняется в переменную `DISK_USAGE`. 

## Аргументы и ввод данных: Делаем скрипты гибкими

Скрипт становится по-настоящему полезным, когда он может принимать данные извне.

### Позиционные аргументы
Bash автоматически присваивает переданные параметры специавым переменным:
*   `$0` — Имя самого скрипта.
*   `$1, $2, $3...` — Первый, второй, третий аргументы.
*   `$#` — Общее количество аргументов.
*   `$@` — Все переданные аргументы списком.

Пример `deploy.sh`:
```bash
#!/bin/bash
echo "Деплоим проект $1 на сервер $2"
```
Запуск: `./deploy.sh my-app production`.

### Обработка флагов с помощью `getopts`
Для более сложных скриптов удобнее использовать флаги (например, `-u root -p password`). В Bash для этого есть встроенная команда `getopts`.

```bash
while getopts "u:p:" opt; do
  case $opt in
    u) USER="$OPTARG" ;;
    p) PORT="$OPTARG" ;;
    *) echo "Использование: $0 -u user -p port" && exit 1 ;;
  esac
done

echo "Подключение к $USER на порт $PORT"
```
Здесь `u:p:` означает, что флаги `-u` и `-p` ожидают аргументы (на это указывают двоеточия).

### Чтение ввода пользователя (`read`)
Иногда нужно спросить пользователя во время выполнения:
```bash
read -p "Вы уверены, что хотите удалить базу данных? (y/n): " confirm
if [[ "$confirm" == "y" ]]; then
    echo "Удаление..."
fi
```

## Условия и проверки: Логика в автоматизации

Условные операторы позволяют скрипту принимать решения. В Bash основным инструментом является `if`.

### `[[ ]]` vs `[ ]`: Что выбрать?
В старых скриптах вы можете встретить одинарные скобки `[ ]`. Однако в современном Bash рекомендуется использовать двойные скобки `[[ ]]`. Они мощнее и безопаснее:
*   Не нужно брать переменные в кавычки внутри скобок (Bash не "сломается" от пробелов).
*   Поддерживают регулярные выражения (оператор `=~`).
*   Более интуитивно понятная логика (например, операторы `&&` и `||` вместо `-a` и `-o`).

Как мы уже выяснили в разделе про подстановку команд, переменная `DISK_USAGE` может содержать строку вида `15%`. Однако операторы сравнения чисел в Bash (`-gt`, `-lt`) ожидают только целые числа. Попытка сравнить `15%` напрямую вызовет ошибку: `integer expression expected`.

Для решения этой проблемы мы "очистим" переменную с помощью утилиты `sed`:

```bash
# Исходное значение: "15%"
DISK_USAGE=$(df / | awk 'NR==2{print $5}')

# Удаляем символ % и сохраняем только число
DISK_USAGE_INT=$(echo "$DISK_USAGE" | sed 's/%//')

if [[ $DISK_USAGE_INT -gt 80 ]]; then
    echo "Внимание: Места почти нет! Использовано: $DISK_USAGE"
else
    echo "Все в порядке. Место на диске: $DISK_USAGE"
fi
```

### Что делает `sed`?
В данном примере `sed 's/%//'` расшифровывается как "найти символ `%` и заменить его на пустоту" (Substitute `%` with nothing). Это стандартный способ удаления или замены символов в Bash.

## Циклы — автоматизация повторений

Циклы позволяют выполнять одну и ту же задачу для множества объектов (файлов, серверов, строк).

### Цикл `for`
Идеально подходит для перебора списков или массивов:
```bash
SERVERS=("web-01" "web-02" "db-01")

for server in "${SERVERS[@]}"; do
    echo "Проверяем статус $server..."
    ssh "$server" "uptime"
done
```

**Разбор синтаксиса массива:**
*   `SERVERS=("a" "b")` — объявление массива через пробел.
*   `"${SERVERS[@]}"` — фигурные скобки `{}` и символ `@` нужны, чтобы обратиться ко всем элементам массива. Кавычки обязательны: они гарантируют, что если в имени сервера есть пробел, он не будет воспринят как два разных элемента.

### Цикл `while`
Выполняется, пока условие **истинно**. Удобен для бесконечных циклов или чтения файлов:
```bash
# Чтение файла построчно
while read -r line; do
    echo "Обработка строки: $line"
done < config.txt
```

### Цикл `until`
Работает наоборот: выполняется, пока условие **ложно** (то есть *пока не* станет истинным). Часто используется для ожидания доступности сервиса:

```bash
# Ждем, пока сервер начнет отвечать на пинг
until ping -c 1 8.8.8.8 > /dev/null 2>&1; do
    echo "Сервер недоступен, ждем 5 секунд..."
    sleep 5
done
echo "Связь установлена!"
```

## Функции — искусство переиспользования кода

Если вы копируете один и тот же кусок кода в скрипте трижды — вынесите его в функцию.

```bash
log_message() {
    local LEVEL=$1
    local MESSAGE=$2
    echo "[$(date +'%Y-%m-%d %H:%M:%S')] [$LEVEL] $MESSAGE"
}

log_message "INFO" "Скрипт запущен"
log_message "ERROR" "Файл не найден"
```
Использование ключевого слова `local` крайне важно: оно ограничивает область видимости переменной внутри функции, предотвращая случайное изменение глобальных переменных.

## Фундамент надежного скрипта: set -euo pipefail и trap

Начинающие часто пишут скрипты "на удачу". Но профессиональный DevOps-скрипт должен быть устойчивым к сбоям.

### Золотой стандарт: `set -euo pipefail`
Добавьте это в начало каждого скрипта сразу после shebang:

*   **`set -e` (errexit):** Скрипт немедленно завершится, если любая команда вернет ошибку (ненулевой код выхода). Это предотвращает "эффект домино", когда ошибка в первой команде приводит к катастрофе в десятой.
*   **`set -u` (nounset):** Ошибка при использовании необъявленной переменной. Помогает избежать опечаток. Представьте `rm -rf $DEL_DIR/`, если из-за опечатки `$DEL_DIR` пуста — вы можете удалить корень системы!
*   **`set -o pipefail`:** По умолчанию в цепочке `cmd1 | cmd2` ошибкой считается только сбой последней команды. `pipefail` заставляет всю цепочку возвращать ошибку, если хотя бы одно звено сломалось.

### Перехват сигналов с помощью `trap`
`trap` позволяет выполнить код при завершении скрипта (даже аварийном) или получении сигналов (например, Ctrl+C). Это идеально для очистки временных файлов.

```bash
# mktemp -d создает временную директорию с уникальным именем в /tmp
TEMP_DIR=$(mktemp -d)

cleanup() {
    rm -rf "$TEMP_DIR"
    echo "Временные файлы удалены."
}

# Выполнить cleanup при выходе (EXIT)
trap cleanup EXIT
```

## Логирование: tee и logger

Хороший скрипт оставляет следы.

*   **`tee`**: Позволяет одновременно выводить текст на экран и записывать в файл. Флаг `-a` (append) добавляет текст в конец файла, не затирая его.
    `echo "Action performed" | tee -a script.log`
*   **`logger`**: Отправляет сообщения в системный журнал (syslog/journald). Это стандарт для серверных приложений, позволяющий централизованно собирать логи.
    `logger -t "MY_SCRIPT" "Service restarted"`

## Реальный скрипт №1: Ротируемый бэкап

Давайте соберем все знания воедино. Этот скрипт делает бэкап папки, проверяет права и удаляет старые копии.

```bash
#!/bin/bash

# 1. Безопасный режим
set -euo pipefail

# 2. Конфигурация
BACKUP_SRC="${1:-/var/www/html}"    # Источник: аргумент $1 или значение по умолчанию
BACKUP_DEST="/mnt/backups"
LOG_FILE="/var/log/backup_script.log"

# 3. Функция логирования с динамической датой
log_message() {
    local MSG="$1"
    local TIMESTAMP=$(date +'%Y-%m-%d %H:%M:%S')
    echo "[$TIMESTAMP] $MSG" | tee -a "$LOG_FILE"
    logger -t "WEB_BACKUP" "$MSG"
}

# 4. Проверка прав (должен быть root)
# $EUID — это встроенная переменная Bash, которая хранит ID текущего пользователя.
# У суперпользователя (root) ID всегда равен 0.
if [[ $EUID -ne 0 ]]; then
   echo "Ошибка: Этот скрипт должен быть запущен от имени root или через sudo" >&2
   exit 1
fi

# 5. Подготовка
mkdir -p "$BACKUP_DEST"
log_message "Начало бэкапа директории $BACKUP_SRC"

# 6. Создание архива
# Используем 2>> для перенаправления ошибок в лог.
# В Linux '1' — это стандартный вывод, а '2' — это поток ошибок (stderr).
DATE=$(date +%Y-%m-%d_%H%M%S)
tar -czf "$BACKUP_DEST/backup_$DATE.tar.gz" "$BACKUP_SRC" 2>>"$LOG_FILE"

log_message "Бэкап успешно создан: $BACKUP_DEST/backup_$DATE.tar.gz"

# 7. Ротация: удаляем файлы старше 7 дней
find "$BACKUP_DEST" -type f -name "*.tar.gz" -mtime +7 -delete
log_message "Старые бэкапы удалены (ротация 7 дней)."
```

### Построчный разбор:
*   **Строка 7:** `${1:-...}` — если первый аргумент не передан, используется путь по умолчанию.
*   **Строка 21:** `$EUID` — позволяет скрипту убедиться, что у него достаточно прав для работы с системными папками.
*   **Строка 33:** `2>>` — гарантирует, что если `tar` столкнется с ошибкой (например, файл занят), описание этой ошибки попадет в лог-файл, а не просто пропадет.
*   **Строка 38:** `find ... -mtime +7 -delete` — поиск и немедленное удаление старых файлов.

## Реальный скрипт №2: Мониторинг ресурсов

Скрипт для проверки состояния сервера.

```bash
#!/bin/bash

set -euo pipefail

THRESHOLD_DISK=80  # Порог заполнения диска в %
THRESHOLD_MEM=90   # Порог использования памяти в %

check_disk() {
    local usage
    # Получаем % использования, убираем символ % с помощью sed
    usage=$(df / | awk 'NR==2 {print $5}' | sed 's/%//')
    if [[ "$usage" -gt "$THRESHOLD_DISK" ]]; then
        echo "WARNING: Диск заполнен на $usage%!" | logger -p user.warn
        return 1
    fi
    return 0
}

check_mem() {
    local mem_usage
    # Вычисляем процент занятой памяти через free и awk
    mem_usage=$(free | grep Mem | awk '{print $3/$2 * 100.0}' | cut -d. -f1)
    if [[ "$mem_usage" -gt "$THRESHOLD_MEM" ]]; then
        echo "CRITICAL: Память заполнена на $mem_usage%!" | logger -p user.crit
        return 1
    fi
    return 0
}

# Основная логика
echo "--- System Health Check ---"
DISK_STATUS=0
MEM_STATUS=0

check_disk || DISK_STATUS=1
check_mem || MEM_STATUS=1

if [[ $DISK_STATUS -eq 0 && $MEM_STATUS -eq 0 ]]; then
    echo "Все системы в норме."
else
    echo "Обнаружены проблемы! Проверьте системные логи."
    exit 1
fi
```

## Планирование задач через cron

Чтобы ваши скрипты работали без вашего участия, используйте планировщик `cron`.

**Внимание:** Если ваш скрипт требует прав root (как наш скрипт бэкапа), его нужно добавлять в crontab суперпользователя. Для этого используйте:
```bash
sudo crontab -e
```
Если вы просто запустите `crontab -e`, вы откроете расписание текущего (обычного) пользователя, и скрипт завершится с ошибкой прав доступа.

Добавьте строку:
```cron
# Запуск бэкапа каждый день в 3 часа ночи
0 3 * * * /usr/local/bin/backup.sh /var/www/my-site >> /var/log/cron_backup.log 2>&1
```

**Формат cron:**
`Минуты Часы ДеньМесяца Месяц ДеньНедели Команда`

**Что значит `2>&1`?**
Эта конструкция направляет стандартный поток ошибок (2) туда же, куда идет стандартный вывод (1). В итоге и лог работы, и ошибки попадут в один файл `cron_backup.log`.

Совет: Всегда указывайте полные пути к скриптам и файлам логов в crontab, так как окружение (PATH) у cron может отличаться от вашего терминала.

## Отладка и Troubleshooting

Даже опытные инженеры ошибаются. Вот как найти проблему в Bash:

1.  **Режим отладки (`-x`):**
    Запустите скрипт как `bash -x ./script.sh`. Вы увидите каждую выполняемую команду и то, во что превращаются переменные.
2.  **Проверка синтаксиса (`-n`):**
    `bash -n ./script.sh` проверит скрипт на ошибки, не запуская его.
3.  **ShellCheck:**
    Это обязательный инструмент (линтер) для любого DevOps. Он укажет на потенциальные ошибки, проблемы безопасности и плохой стиль кода. Можно использовать онлайн на [shellcheck.net](https://www.shellcheck.net/).

### Типичные проблемы:
*   **CRLF line endings:** Если вы писали скрипт в Windows, он может не запуститься в Linux из-за невидимых символов конца строки. Используйте `dos2unix`.
*   **PATH:** Команда работает в терминале, но не в скрипте? Проверьте, установлена ли переменная PATH или используйте полный путь к утилите.

## Лучшие практики и типичные ошибки (Anti-patterns)

### ✅ Делайте так:
*   **Кавычки:** Всегда пишите `"$VAR"` вместо `$VAR`. Без кавычек Bash разобьет значение по пробелам, что сломает скрипт, если в пути к файлу есть пробел.
*   **Длинные флаги:** В скриптах пишите `rm --recursive` вместо `rm -r`. Это делает код читаемым для коллег.
*   **Комментарии:** Пишите не *что* делает код (это и так видно), а *почему* он это делает.

### ❌ Не делайте так:
*   **Hardcoding:** Не зашивайте пароли и секреты прямо в код. Используйте переменные окружения или защищенные хранилища (Vault).
*   **Парсинг `ls`:** Никогда не используйте `for file in $(ls *.txt)`. Если в имени файла есть пробел или перенос строки, скрипт сломается. Используйте `for file in *.txt;` (globbing).
*   **Использование `sh` вместо `bash`:** Помните, что `sh` — это минималистичный стандарт. Многие функции, описанные в этой статье (например, `[[ ]]` или массивы), не будут работать в чистом `sh`.

## Заключение

Написание Bash-скриптов — это навык, который приходит с практикой. Начните с автоматизации самых простых действий: очистки логов, проверки статуса сервиса или создания архива. Постепенно внедряйте обработку ошибок, логирование и модульность. 

Помните, что лучший скрипт — это тот, который понятен не только вам сегодня, но и вашему коллеге через полгода. Соблюдайте стандарты, используйте `set -euo pipefail` и не бойтесь заглядывать в `man bash`. Удачной автоматизации!
