# Systemd: управляем сервисами как профи

Добро пожаловать на двенадцатый урок нашего курса для начинающих DevOps-инженеров. Сегодня мы разберем систему инициализации и управления службами **systemd**. Если вы когда-либо настраивали сервер на базе Ubuntu, Debian, CentOS или Fedora, вы уже использовали systemd, даже если не подозревали об этом. Это «первый гражданин» операционной системы, который берет на себя бразды правления сразу после загрузки ядра и не отпускает их до полной остановки питания. В этом уроке мы научимся не просто пользоваться базовыми командами, а понимать философию и внутреннее устройство этого мощного инструмента.

## Что такое init-система и зачем нужен systemd

Прежде чем мы начнем вводить команды в терминале, давайте разберемся, зачем Linux вообще нужна какая-то «система инициализации». Когда вы включаете компьютер, происходит сложная цепочка событий: BIOS/UEFI инициализирует оборудование, передает управление загрузчику (GRUB), тот загружает ядро Linux в память. Ядро монтирует корневую файловую систему, но оно само по себе не знает, как запустить веб-сервер, базу данных или графический интерфейс. Ядру нужен «менеджер», который займется организацией всего остального пространства пользователя (userspace).

Этот менеджер — самый первый процесс в системе. Он всегда имеет **PID 1** (Process ID). Именно он является «родителем» всех остальных процессов. Если этот процесс по какой-то причине перестанет работать, система немедленно остановится и выдаст критическую ошибку, известную как Kernel Panic.

### Наследие: Эпоха SysVinit и её закат

Долгие годы (с 1980-х до начала 2010-х) доминирующей системой в Linux была **SysVinit** (System V Initialization), заимствованная из классических UNIX-систем. Она была концептуально проста: существовали так называемые «уровни запуска» (runlevels) — от 0 до 6. Например, уровень 3 соответствовал многопользовательскому режиму без графики, а уровень 5 — с графической оболочкой.

Для каждого сервиса системный администратор должен был написать сложный скрипт на языке Bash в директории `/etc/init.d/`. Эти скрипты принимали стандартные команды `start`, `stop`, `restart`.

**Главные проблемы SysVinit, которые привели к её замене:**
1. **Последовательный запуск:** Сервисы запускались строго по очереди. Если сеть поднималась медленно, то база данных, веб-сервер и все остальные службы покорно ждали, пока сеть завершит настройку. Это делало время загрузки современных серверов неприемлемо долгим.
2. **Сложность поддержки скриптов:** Скрипты на Bash часто превращались в «спагетти-код», где было трудно отслеживать ошибки и корректно обрабатывать сценарии сбоев. Каждый разработчик писал скрипты по-своему, что создавало хаос.
3. **Отсутствие контроля зависимостей:** Было крайне сложно гарантировать, что приложение запустится *ровно тогда*, когда база данных действительно готова принимать соединения.

### Таблица соответствия команд: SysVinit vs systemd

Для тех, кто привык к старому синтаксису или работает с legacy-системами, ниже приведена таблица соответствия основных команд. Это поможет вам быстрее адаптироваться к `systemctl`, если вы раньше работали с `service` или `chkconfig`.

| Действие | Команда SysVinit / legacy | Команда systemd |
| :--- | :--- | :--- |
| Запуск сервиса | `sudo service nginx start` | `sudo systemctl start nginx` |
| Остановка сервиса | `sudo service nginx stop` | `sudo systemctl stop nginx` |
| Перезапуск сервиса | `sudo service nginx restart` | `sudo systemctl restart nginx` |
| Проверка статуса | `sudo service nginx status` | `sudo systemctl status nginx` |
| Перезагрузка конфигурации | `sudo service nginx reload` | `sudo systemctl reload nginx` |
| Добавление в автозагрузку | `sudo chkconfig nginx on` | `sudo systemctl enable nginx` |
| Удаление из автозагрузки | `sudo chkconfig nginx off` | `sudo systemctl disable nginx` |
| Проверка статуса автозагрузки | `chkconfig --list nginx` | `systemctl is-enabled nginx` |
| Список запущенных сервисов | `service --status-all` | `systemctl list-units --type=service` |

### Пришествие и триумф systemd

В 2010 году инженеры Леннарт Поттеринг и Кай Сиверс представили проект **systemd**. Это была амбициозная попытка не просто заменить init, а создать полноценный системный менеджер, объединяющий управление процессами, логами, сетью и монтированием дисков.

**Почему systemd победил?**
* **Параллелизм:** Systemd анализирует зависимости между юнитами и запускает всё, что можно, одновременно.
* **Декларативность:** Вместо императивных скриптов мы используем простые декларативные файлы (Unit-файлы) в формате `.ini`.
* **Сокетная активация:** Systemd может создать сетевой порт сервиса еще до его запуска.
* **Cgroups (Control Groups):** Позволяет системе четко отслеживать потребление ресурсов и гарантированно убивать все дочерние процессы при остановке службы.
* **Интегрированные инструменты:** В систему сразу встроены менеджер логов (`journald`), менеджер сети (`networkd`) и менеджер времени.

## Управление сервисами через systemctl

Основным инструментом администратора является утилита `systemctl`. Это «пульт управления», с помощью которого вы будете выполнять 99% задач.

### Базовые команды управления

Представим, что мы работаем с веб-сервером **Nginx**. 

> **Важное примечание:** Чтобы команды ниже сработали, Nginx должен быть установлен (`sudo apt install nginx`). Если вы не хотите ничего устанавливать, можете заменить `nginx` на `cron` или `dbus` — эти службы обычно уже есть в системе.

Вот команды, которые вы будете использовать ежедневно:

* **Просмотр состояния:** `sudo systemctl status nginx`.
* **Запуск и остановка:** `sudo systemctl start nginx` / `stop`.
* **Перезапуск:** `sudo systemctl restart nginx` (полная остановка и запуск) или `reload` (перечитать настройки без остановки процесса).

### Автозапуск (Enable vs Disable)

- `start` запускает сервис сейчас.
- `enable` гарантирует, что сервис запустится сам после старта системы.

```bash
sudo systemctl enable nginx   # Добавить в автозагрузку
sudo systemctl disable nginx  # Убрать из автозагрузки
```

Используйте флаг `--now`, чтобы выполнить действие и сразу применить его: `sudo systemctl enable --now nginx`.

## Анатомия unit-файла

Unit-файл — это декларативное описание ресурса. В systemd всё является «юнитом»: службы (`.service`), таймеры (`.timer`), точки монтирования (`.mount`).

### Локация файлов
1. `/lib/systemd/system/`: Юниты, установленные пакетным менеджером.
2. `/etc/systemd/system/`: Ваше рабочее пространство.

### Разбор структуры `.service` файла

```ini
[Unit]
Description=My Backend Application
After=network.target postgresql.service
Wants=redis.service

[Service]
Type=simple
User=webapp
Group=webapp
WorkingDirectory=/var/www/myapp
ExecStart=/usr/bin/python3 main.py
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```

**Кратко по параметрам:**
- **After vs Wants**: `After` задает *порядок* (запуститься строго *после* сети), а `Wants` задает *зависимость* (попытаться запустить `redis.service` вместе с нашим сервисом, но не падать, если он не запустится).
- **User / WorkingDirectory**: Эти параметры должны указывать на реально существующего пользователя и директорию в вашей системе. Если пользователя `webapp` не существует или у него нет прав доступа к `/var/www/myapp`, systemd выдаст ошибку `status=217/USER` или `Permission denied`.

## Создаём свой сервис — пример с Python-приложением (Flask API)

В реальной работе DevOps чаще всего приходится упаковывать в сервисы не Bash-скрипты, а полноценные приложения. Давайте создадим сервис для простого Python-приложения.

**Шаг 1: Код приложения (`/opt/myapp/app.py`)**
```python
from flask import Flask
app = Flask(__name__)

@app.route("/")
def hello():
    return "Hello from systemd-managed Python App!"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
```

**Шаг 2: Создание Unit-файла (`/etc/systemd/system/python-api.service`)**
Здесь мы используем `Environment` для передачи переменных окружения.

```ini
[Unit]
Description=Python Flask API Service
After=network.target

[Service]
Type=simple
User=webapp
WorkingDirectory=/opt/myapp
Environment="PYTHONPATH=/opt/myapp"
Environment="FLASK_ENV=production"
ExecStart=/usr/bin/python3 app.py
Restart=always
# Безопасность
ProtectHome=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```

**Шаг 3: Перезагрузка и запуск**
```bash
sudo systemctl daemon-reload
sudo systemctl enable --now python-api
```

Этот пример демонстрирует, как systemd может служить в качестве базового менеджера процессов для ваших микросервисов, если вы еще не перешли на Docker и Kubernetes.

## Глубокое погружение: Безопасность и Лимиты

DevOps-инженер должен уметь не просто запускать сервис, но и ограничивать его возможности для повышения стабильности и безопасности.

### Ограничение ресурсов (Resource Limits) и cgroup v2

Systemd позволяет легко задать лимиты потребления ресурсов прямо в Unit-файле. Это стало возможным благодаря механизму ядра Linux под названием **cgroups** (Control Groups). В современных дистрибутивах (Debian 11+, Ubuntu 22.04+, RHEL 9+) по умолчанию используется **cgroup v2**, которая предлагает единую иерархию ресурсов и более предсказуемое поведение.

> **Важно:** Раньше использовался параметр `MemoryLimit`, но в современных версиях systemd (начиная с v231) рекомендуется использовать `MemoryMax`. Если вы используете старый параметр на новой системе, вы можете увидеть предупреждения в логах.

Основные параметры ограничения ресурсов:

*   **MemoryMax=500M**: Жесткий лимит. Если сервис превысит это значение, ядро немедленно убьет его (OOM Killer).
*   **MemoryHigh=400M**: "Мягкий" лимит. При достижении этого порога система начнет замедлять выделение памяти процессу или агрессивно сбрасывать кэши, пытаясь удержать его в рамках. Это позволяет избежать внезапной смерти процесса.
*   **CPUQuota=20%**: Ограничивает время процессора. Если у вас 4 ядра, то 20% — это пятая часть времени одного ядра.
*   **TasksMax=100**: Лимит на количество процессов и потоков внутри юнита (защита от "fork bomb").

```ini
[Service]
CPUQuota=20%
MemoryMax=500M
MemoryHigh=450M
TasksMax=100
```

### Изоляция и Безопасность (Sandboxing)

Systemd предоставляет мощные инструменты для ограничения доступа процесса к системе:
* **ProtectSystem=full**: делает папки `/usr`, `/boot` и `/etc` доступными только для чтения.
* **ProtectHome=true**: запрещает сервису доступ к папкам `/home` и `/root`.
* **PrivateTmp=true**: создает для сервиса отдельную временную папку `/tmp`.
* **NoNewPrivileges=true**: гарантирует, что сервис никогда не сможет повысить свои привилегии.

## Systemd Timers: Современная альтернатива Cron

Долгое время стандартом для запуска задач по расписанию был `cron`. В systemd его заменили **Timers**.

**Преимущества таймеров:**
1. **Зависимости:** Можно связать запуск с другими событиями.
2. **Логирование:** Вывод скрипта попадает прямо в `journald`.
3. **Гибкость:** Поддержка монотонных таймеров (например, «через 5 минут после загрузки»).

Для работы таймера нужно два файла: `.service` (что запустить) и `.timer` (когда запустить).

**Пример файла `/etc/systemd/system/myapp-backup.timer`:**
```ini
[Unit]
Description=Run backup every 5 minutes

[Timer]
OnUnitActiveSec=5min
Unit=myapp-backup.service

[Install]
WantedBy=timers.target
```

## Экосистема systemd: Больше, чем просто Init

Systemd — это модульная система, включающая в себя множество вспомогательных демонов, которые заменяют старые утилиты.

1. **systemd-networkd**: Легковесный менеджер сетевых конфигураций. Он особенно популярен в контейнерах и облачных инстансах, так как потребляет минимум ресурсов и очень быстро настраивает интерфейсы.
2. **systemd-resolved**: Локальный DNS-резолвер. Он обеспечивает кеширование DNS-запросов и интеллектуальное управление DNS-серверами для разных интерфейсов (например, для VPN и основной сети).
3. **systemd-timesyncd**: Компактный NTP-клиент для синхронизации времени. Вам больше не нужно устанавливать тяжелый пакет `ntp` или `chrony` для простого поддержания точного времени на сервере.
4. **systemd-logind**: Управляет сессиями пользователей, отслеживает входы в систему и корректно завершает процессы пользователя при выходе.

## Цели (Targets) — Уровни запуска в новом формате

В старом добром SysVinit были "уровни запуска" (runlevels). В systemd их заменили более гибкие **Targets**. Таргет — это просто группа юнитов, которую система должна запустить.

### Основные таргеты:
1. **multi-user.target**: Соответствует классическому runlevel 3. Многопользовательский режим без графики (типично для серверов).
2. **graphical.target**: Соответствует runlevel 5. Включает в себя всё из `multi-user.target` плюс графическую оболочку (Gnome, KDE и т.д.).
3. **rescue.target**: Однопользовательский режим для восстановления системы.

### Как переключаться?
Вам больше не нужно помнить цифры уровней запуска. Команды интуитивно понятны:
*   **Сменить таргет "на лету"**: `sudo systemctl isolate graphical.target` (система запустит графику прямо сейчас).
*   **Узнать текущий таргет по умолчанию**: `systemctl get-default`.
*   **Сменить таргет навсегда**: `sudo systemctl set-default multi-user.target`.

Это крайне полезно, если вы установили сервер с графикой (зачем-то), а потом решили сэкономить ресурсы и оставить только консоль.

## Анализ загрузки системы: Ускоряем сервер

Одним из главных преимуществ systemd является возможность детально проанализировать, на что именно тратится время при старте ОС. Для этого используется утилита **systemd-analyze**.

### systemd-analyze blame
Эта команда выводит список всех запущенных сервисов, отсортированный по времени инициализации. 

```bash
systemd-analyze blame
```
*   **Пример из практики:** Вы видите, что `snapd.service` занимает 15 секунд. Если вы не используете Snap-пакеты, вы можете его отключить и сэкономить кучу времени.

### systemd-analyze critical-chain
Эта команда показывает "критическую цепочку" — последовательность сервисов, которые тормозят финальную загрузку системы.

```bash
systemd-analyze critical-chain
```
Она строит дерево зависимостей. Вы можете увидеть, что база данных ждет сеть, а приложение ждет базу. Если сеть поднимается слишком долго, вся цепочка будет висеть.

## Работа с логами: journalctl

Прошли времена, когда нужно было знать путь к каждому файлу в `/var/log/`. Теперь всё собирается в централизованном хранилище **journald**.

**Базовые приемы работы с логами:**

*   **Все логи в реальном времени**: `journalctl -f`.
*   **Логи конкретного сервиса**: `journalctl -u nginx.service`.
*   **Логи за определенный период**: 
    ```bash
    journalctl --since "2023-10-01 12:00:00"
    journalctl --since "1 hour ago"
    journalctl --since yesterday
    ```
*   **Только ошибки**: `journalctl -p err -b` (флаг `-b` означает "с момента последней загрузки").
*   **Объем занимаемого места**: `journalctl --disk-usage`. Если логи занимают слишком много места (например, 10 Гб), их можно ограничить в конфиге `/etc/systemd/journald.conf`.

## Устранение неполадок (Troubleshooting)

Как профессиональный DevOps, вы должны уметь быстро находить и исправлять ошибки в работе сервисов.

### Типичные ошибки и способы их решения:

1. **Ошибка: `ExecStart syntax error`**
   - **Причина:** В Unit-файле указан относительный путь к команде (например, `python3` вместо `/usr/bin/python3`).
   - **Решение:** Всегда используйте абсолютные пути. Проверить пути можно командой `command -v python3`. Помните, что `which` может выдать некорректный путь, если вы запускаете его не от того пользователя.

2. **Ошибка: `Permission denied` в логах**
   - **Причина:** Пользователь, указанный в `User=`, не имеет прав доступа к исполняемому файлу или рабочей директории.
   - **Решение:** Проверьте владельца файлов (`ls -l`) и права доступа. Не забудьте, что пользователю нужны права на выполнение (`+x`) для скриптов.

3. **Ошибка: Сервис постоянно перезапускается (Boot loop)**
   - **Причина:** Программа падает сразу после запуска, а `Restart=always` заставляет её запускаться снова и снова.
   - **Решение:** Посмотрите последние логи через `journalctl -u имя_сервиса -n 50`. Часто причина кроется в отсутствии переменных окружения или недоступности базы данных.

4. **Ошибка: `daemon-reload` не был выполнен**
   - **Причина:** Вы изменили Unit-файл на диске, но `systemctl start` запускает старую версию.
   - **Решение:** Всегда выполняйте **`sudo systemctl daemon-reload`** после любого редактирования файлов в `/etc/systemd/system/`. Это критически важный шаг!

## Вопросы для подготовки к собеседованию (DevOps Interview)

1. **В чем разница между `restart` и `reload`?**
   - `restart` перезапускает процесс (убивает и создает новый), `reload` просит процесс перечитать конфиг без остановки.

2. **Что такое `multi-user.target`?**
   - Это стандартный таргет для серверных систем, соответствующий многопользовательскому режиму работы без графической оболочки.

3. **Как ограничить потребление памяти сервисом?**
   - Использовать директиву `MemoryMax=X` (или устаревшую `MemoryLimit=X`) в секции `[Service]`.

4. **Как посмотреть логи системы с момента последней загрузки?**
   - Команда `journalctl -b`.

## Практический пример: Создаем сервис мониторинга

### Шаг 1: Скрипт
Создадим `/usr/local/bin/mem-check.sh`. Обратите внимание, что мы используем абсолютный путь к команде `free` и не перенаправляем вывод в файл — systemd сам перехватит `stdout` и отправит его в `journald`.

```bash
#!/bin/bash
while true; do
    /usr/bin/free -h
    sleep 60
done
```
Сделаем его исполняемым: `sudo chmod +x /usr/local/bin/mem-check.sh`.

### Шаг 2: Создание пользователя
Для безопасности сервис должен работать от отдельного пользователя с минимальными правами:
```bash
sudo useradd -r -s /bin/false membot
```

### Шаг 3: Юнит-файл
Создаем `/etc/systemd/system/mem-monitor.service`:
```ini
[Unit]
Description=Memory Usage Monitor
After=local-fs.target

[Service]
Type=simple
User=membot
ExecStart=/usr/local/bin/mem-check.sh
Restart=on-failure
ProtectSystem=full

[Install]
WantedBy=multi-user.target
```

### Шаг 4: Запуск и проверка
После создания или изменения любого юнит-файла необходимо обновить конфигурацию systemd и запустить сервис:

```bash
sudo systemctl daemon-reload      # Обновляем конфиг
sudo systemctl enable --now mem-monitor  # Включаем автозагрузку и запускаем
```

Проверить логи теперь можно стандартным способом через `journalctl`:
```bash
sudo journalctl -u mem-monitor -f
```
