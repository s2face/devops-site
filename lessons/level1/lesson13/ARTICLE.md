## Проблема «у меня работает» — и как Docker её решает

Прежде чем переходить к техническим деталям, давайте разберемся, какую именно боль лечит Docker. 

### Матрица ада (Matrix of Hell)

Представьте классическую ситуацию в IT-компании. У вас есть несколько приложений:
1.  Сайт на PHP 7.4 с базой данных MySQL.
2.  Микросервис на Python 3.9 с библиотекой обработки данных.
3.  Очередь задач на Redis.

Вам нужно запустить всё это на сервере разработчика, на тестовом стенде и, наконец, в продакшене. Однако выясняется, что на тестовом стенде стоит Ubuntu с предустановленным PHP 8.1, который ломает ваш сайт. А на продакшене используется другая версия системной библиотеки `glibc` (основная библиотека языка C, от которой зависят почти все программы в Linux), из-за чего Python-сервис отказывается стартовать.

Это и называется «матрицей ада»: когда количество комбинаций «приложение + окружение» растет в геометрической прогрессии. Разработчики тратят до 30–40% своего времени не на написание кода, а на борьбу с зависимостями и настройку окружения.

### Аналогия из мира логистики

До 1950-х годов в мировой торговле была похожая проблема. Товары (сахар, ткани, станки) грузили на корабли навалом. Каждый груз требовал особого обращения: мешки могли порваться, станки — погнуться, а сахар — отсыреть. Погрузка занимала дни, а перевалка с корабля на поезд была кошмаром.

Решение пришло в виде **стандартного морского контейнера**. Теперь неважно, что внутри — виски или электроника. Портовый кран знает, как захватить стандартный контейнер, а грузовик — как его закрепить. Контейнер защищает содержимое от внешней среды, а внешнюю среду — от содержимого.

Docker сделал то же самое для программного обеспечения. Он берет ваш код, библиотеки, настройки и переменные окружения, упаковывает их в «цифровой контейнер». Теперь операционной системе неважно, что внутри контейнера — Java, Go или Python. Она просто знает, как запустить этот контейнер.

### Как Docker решает конфликты зависимостей

Docker изолирует приложения друг от друга. Если одному приложению нужен Python 2.7, а другому Python 3.10, вы просто запускаете их в разных контейнерах на одном сервере. Они не будут знать о существовании друг друга и не смогут конфликтовать, так как у каждого будет своя собственная файловая система со своими библиотеками.

## VM vs Container: в чём разница?

Многие новички путают Docker с виртуальными машинами (такими как VirtualBox или VMware). На первый взгляд они похожи: и те, и другие позволяют запускать изолированные окружения. Но "под капотом" у них принципиально разные механизмы.

### Архитектура Виртуальных Машин (VM)

Виртуальная машина эмулирует физическое железо. Когда вы запускаете VM, происходит следующее:
1.  На физическом сервере работает **Хостовая ОС**.
2.  Поверх неё запускается **Гипервизор** (программа для управления виртуальными машинами).
3.  Гипервизор создает виртуальное "железо" (CPU, RAM, диск).
4.  Внутри этого виртуального железа вы устанавливаете **Гостевую ОС** (например, еще одну Ubuntu или Windows).
5.  И только внутри гостевой ОС вы запускаете свои приложения и библиотеки.

**Минусы VM:**
*   **Огромный оверхед (накладные расходы):** Каждая гостевая ОС потребляет сотни мегабайт (а то и гигабайты) оперативной памяти только на поддержание собственного ядра.
*   **Медленный запуск:** Загрузка виртуальной машины занимает минуты, так как нужно пройти весь цикл загрузки ОС (BIOS, загрузчик, инициализация ядра, запуск сервисов).
*   **Тяжелые образы:** Образ виртуальной машины весит гигабайты, даже если ваше приложение занимает 10 КБ.

### Архитектура Контейнеров (Docker)

Docker работает иначе. Он не виртуализирует железо. Он виртуализирует **операционную систему**.

```text
+---------------------------------------+
|             Containers                |
|  [ App A ]    [ App B ]    [ App C ]  |
+---------------------------------------+
|           Docker Engine               |
+---------------------------------------+
|              Host OS                  |
+---------------------------------------+
|             Hardware                  |
+---------------------------------------+
```

Все контейнеры на сервере используют **одно и то же ядро хостовой ОС**. Docker просто "отрезает" для каждого контейнера кусочек ресурсов и создает иллюзию отдельной системы с помощью механизмов ядра Linux:
*   **Namespaces (пространства имен)** — отвечают за то, что процесс видит вокруг. Они изолируют файловую систему, сеть, пользователей и процессы. Контейнер думает, что он — единственная система.
*   **Cgroups (контрольные группы)** — отвечают за то, сколько ресурсов процесс может съесть. Они устанавливают лимиты на CPU, оперативную память и диск.

**Плюсы контейнеров:**
*   **Минимальный оверхед:** Контейнер — это просто процесс в системе. Если приложение потребляет 10 МБ RAM, то и контейнер будет потреблять 10 МБ (плюс микроскопические доли на управление).
*   **Мгновенный запуск:** Запуск контейнера занимает миллисекунды. Это фактически просто запуск процесса.
*   **Компактность:** Образы контейнеров могут весить всего несколько мегабайт (например, образ Alpine Linux весит около 5 МБ).

### Сравнение в таблице

| Характеристика | Виртуальные машины | Контейнеры (Docker) |
| :--- | :--- | :--- |
| **Изоляция** | Полная (на уровне железа) | Процессная (на уровне ОС) |
| **Операционная система** | Своя гостевая ОС в каждой VM | Общее ядро хоста для всех |
| **Запуск** | Минуты | Секунды/Миллисекунды |
| **Использование RAM** | Фиксированное резервирование | Динамическое потребление |
| **Переносимость** | Зависит от гипервизора | Запустится везде, где есть Docker |

**Важное уточнение:** Docker — это не замена VM. Часто они работают вместе: вы арендуете виртуальную машину в облаке (например, AWS EC2) и запускаете внутри неё десятки Docker-контейнеров. Это дает двойную изоляцию и максимальную эффективность.

## Основные концепции Docker

Чтобы не запутаться в терминологии, давайте разберем четыре "кита", на которых держится Docker.

### 1. Образ (Image)
Образ — это "чертеж" или "слепок" вашего приложения. Это неизменяемый файл, который содержит всё необходимое: код, среду выполнения (runtime), библиотеки и настройки. 
*Аналогия: Это как установочный диск (ISO) или рецепт блюда.*

**Концепция слоев (Layers):**
Образ в Docker состоит из нескольких слоев. Каждый слой — это изменение файловой системы (например, установка пакета или копирование файлов). Слои накладываются друг на друга и доступны только для чтения.
Это дает два огромных плюса:
*   **Экономия места:** Если у вас 10 образов базируются на одной и той же Ubuntu, этот слой скачается на диск всего один раз.
*   **Скорость:** При обновлении приложения Docker скачает только те слои, которые изменились.

### 2. Контейнер (Container)
Контейнер — это запущенный экземпляр образа. Вы можете запустить 10 контейнеров из одного и того же образа. Каждый из них будет работать изолированно, но все они будут идентичны в момент старта.
*Аналогия: Это как установленная программа в памяти или готовое блюдо, приготовленное по рецепту.*

### 3. Реестр (Registry)
Хранилище для образов. Самый известный — **Docker Hub**. Когда вы пишете `docker pull nginx`, Docker идет в реестр и скачивает оттуда нужный образ. Компании часто используют приватные реестры для хранения своих проприетарных приложений.

### 4. Docker Engine
Это программное обеспечение, установленное на вашем хосте, которое занимается всей грязной работой: скачивает образы, создает контейнеры, управляет сетью и дисками.

## Установка Docker на Ubuntu 22.04

Хотя Docker можно установить через стандартные репозитории Ubuntu (`apt install docker.io`), это не рекомендуется, так как там часто лежат устаревшие версии. Официальный способ — использовать скрипт или репозиторий самого Docker.

Мы воспользуемся самым быстрым и надежным способом для обучения — официальным установочным скриптом.

### Шаг 1: Установка через скрипт
Подключитесь к вашей Ubuntu по SSH и выполните команды:
```bash
sudo apt update && sudo apt install -y curl
curl -fsSL https://get.docker.com | sudo sh
```
Этот скрипт автоматически определит вашу версию ОС, добавит нужные ключи и репозитории, а затем установит последнюю стабильную версию Docker Engine.

### Шаг 2: Настройка прав (Post-install)
По умолчанию команды Docker требуют прав суперпользователя (`sudo`). Постоянно вводить `sudo` неудобно и может привести к ошибкам. Чтобы разрешить текущему пользователю управлять Docker, нужно добавить его в специальную группу:

```bash
sudo usermod -aG docker $USER
```
**Внимание!** Чтобы изменения вступили в силу, вам нужно либо выйти и снова войти в систему (переподключиться по SSH), либо выполнить команду, которая обновит группы в текущей сессии (но только для этого окна терминала):
```bash
newgrp docker
```

### Шаг 3: Проверка версии
Убедитесь, что всё прошло успешно:
```bash
docker --version
```
Вы должны увидеть что-то вроде `Docker version 24.0.5, build ...`. Если команда выдала ошибку "permission denied", значит, вы забыли выполнить шаг 2 или применили `newgrp` в другом окне. Самый надежный способ — перелогиниться.

## Первые шаги: запускаем контейнер

Теперь, когда Docker установлен, пора что-нибудь запустить.

### Привет от Docker: hello-world
Самый первый тест, который делают все:
```bash
docker run hello-world
```
Что произошло после нажатия Enter?
1.  Docker Engine проверил, есть ли образ `hello-world` на вашем компьютере.
2.  Не нашел его локально и обратился к **Docker Hub**.
3.  Скачал (pull) образ.
4.  Создал контейнер на основе этого образа.
5.  Запустил его.
6.  Процесс внутри контейнера вывел текст в консоль и успешно завершился.

### Запуск веб-сервера Nginx
`hello-world` — это скучно. Давайте запустим настоящий веб-сервер, который будет работать в фоне и принимать запросы.

```bash
docker run -d -p 8080:80 --name my-nginx nginx
```

Разберем эту длинную команду по частям:
*   `docker run` — команда запуска.
*   `-d` (detached) — запустить в фоновом режиме. Если этого не сделать, ваш терминал "приклеится" к логам контейнера, и вы не сможете вводить команды.
*   `-p 8080:80` — проброс портов. Мы говорим Docker: "Любой запрос, приходящий на мой сервер (хост) на порт 8080, перенаправляй внутрь контейнера на порт 80". Помните, что nginx внутри контейнера по умолчанию слушает именно 80 порт.
*   `--name my-nginx` — мы даем контейнеру имя. Если его не указать, Docker придумает случайное имя вроде `flamboyant_hopper`. По имени гораздо удобнее обращаться к контейнеру в будущем.
*   `nginx` — название образа, который мы запускаем.

**Проверка:**
Теперь откройте браузер и введите IP вашего сервера и порт 8080 (например, `http://1.2.3.4:8080`). Вы увидите стандартную страницу "Welcome to nginx!". Вы только что развернули веб-сервер за пару секунд!

## Управление контейнерами и образами

В процессе работы у вас будет накапливаться много контейнеров и образов. Нужно уметь ими управлять.

### Просмотр контейнеров
*   `docker ps` — показывает только **запущенные** в данный момент контейнеры.
*   `docker ps -a` — показывает **все** контейнеры (запущенные, остановленные, завершенные с ошибкой).

Вы увидите таблицу с колонками: `CONTAINER ID`, `IMAGE`, `COMMAND`, `CREATED`, `STATUS`, `PORTS`, `NAMES`.

### Просмотр логов
Если приложение внутри контейнера ведет себя странно, первым делом смотрим логи:
```bash
docker logs -f my-nginx
```
Флаг `-f` (follow) позволяет видеть новые строки логов в реальном времени. Нажмите `Ctrl+C`, чтобы выйти из режима просмотра логов.

### Выполнение команд внутри контейнера
Иногда нужно "войти" внутрь контейнера, чтобы проверить файлы или конфигурацию.
```bash
docker exec -it my-nginx bash
```
*   `-it` — комбинация флагов для интерактивного режима с поддержкой ввода с клавиатуры.
*   `bash` — программа, которую мы хотим запустить внутри. (Если вы используете облегченные образы вроде `alpine`, вместо `bash` используйте `sh`).

Вы окажетесь внутри изолированной файловой системы контейнера. Введите `exit`, чтобы выйти.

### Остановка и удаление
Чтобы остановить наш nginx:
```bash
docker stop my-nginx
```
Теперь, если вы введете `docker ps`, его там не будет. Но он все еще существует в системе в состоянии `Exited`. Чтобы окончательно его удалить:
```bash
docker rm my-nginx
```

### Работа с образами
*   `docker images` — список всех образов, скачанных на диск.
*   `docker rmi nginx` — удалить образ. Это можно сделать только если нет контейнеров (даже остановленных), которые используют этот образ.

## Полезные флаги docker run

Команда `docker run` имеет сотни флагов, но для ежедневной работы вам понадобятся лишь несколько из них.

### Таблица: Ключевые параметры

| Флаг | Что делает | Пример использования |
| :--- | :--- | :--- |
| `-d` | Фоновый режим | `docker run -d mysql` |
| `-p` | Порты (host:container) | `-p 3306:3306` |
| `-v` | Тома (Volumes) | `-v /opt/data:/var/lib/mysql` |
| `-e` | Переменные окружения | `-e MYSQL_ROOT_PASSWORD=pass` |
| `--name` | Имя контейнера | `--name production_db` |
| `--rm` | Удалить после выхода | `docker run --rm alpine ls -l` |
| `-it` | Интерактивный режим | `docker run -it ubuntu bash` |
| `--restart` | Политика перезапуска | `--restart always` |

### Глубокое погружение: Volumes (Тома)
Это критически важная тема. Контейнеры по своей природе **эфемерны** (непостоянны). Если вы запишете данные внутрь контейнера и удалите его, данные исчезнут навсегда.

Чтобы сохранить данные, мы используем тома. В простейшем виде это **Bind Mounts** — прямая связь папки на хосте с папкой в контейнере.

**Ремарка:** Bind Mount — это самый простой тип томов, который напрямую связывает конкретный путь на хост-системе с путем внутри контейнера. Это удобно для разработки, так как изменения файлов на хосте мгновенно видны в контейнере. Но помните, что это делает ваш контейнер зависимым от структуры файловой системы хоста.

```bash
docker run -d -p 8081:80 -v $PWD/html:/usr/share/nginx/html nginx
```
В этом примере мы используем переменную `$PWD` (текущая рабочая директория). Всё, что вы положите в папку `html` в текущей директории на сервере, мгновенно появится в папке веб-сервера внутри контейнера. Мы использовали порт `8081`, чтобы он не конфликтовал с нашим предыдущим контейнером.

### Глубокое погружение: Переменные окружения (`-e`)
Современные приложения должны настраиваться через переменные окружения (это хорошая практика разработки). Например, чтобы запустить базу данных MySQL, ей нужно передать пароль:

```bash
docker run -d --name mydb -e MYSQL_ROOT_PASSWORD=mysecret mysql
```
Без этого флага образ MySQL просто откажется запускаться, так как он запрограммирован требовать пароль через переменную `MYSQL_ROOT_PASSWORD`.

## Частые ошибки новичков

1.  **Попытка запустить GUI-приложения:** Docker предназначен для консольных приложений и сервисов.
2.  **Хранение данных в контейнере:** Никогда не храните базу данных или логи внутри контейнера без Volume. Рано или поздно вы его удалите, и будет больно.
3.  **Огромные образы:** Не пытайтесь засунуть всё в один контейнер (и базу, и код, и кэш). Следуйте правилу: "Один процесс — один контейнер".
4.  **Забытые контейнеры:** Новички часто запускают контейнеры командой `run`, они падают с ошибкой, и люди запускают их снова. В итоге в системе висят десятки остановленных контейнеров.
    *   Используйте команду `docker container prune`, чтобы удалить все остановленные контейнеры разом:
    ```bash
    docker container prune
    ```
    *   Или запускайте временные контейнеры с флагом `--rm`, чтобы они удалялись сами после остановки.

## Ключевые выводы урока

1.  **Docker решает «Матрицу ада»:** упаковывает приложение со всеми зависимостями в изолированный контейнер — работает одинаково везде.
2.  **Контейнеры ≠ Виртуальные машины:** VM эмулирует железо (своя ОС), контейнеры виртуализируют ОС (общее ядро хоста).
3.  **Namespaces и Cgroups — основа изоляции:** namespaces изолируют процессы, сеть, ФС; cgroups ограничивают CPU, RAM.
4.  **Образ vs Контейнер:** образ — шаблон (чертёж), контейнер — запущенный экземпляр образа (процесс).
5.  **Слои в образах экономят место:** одинаковые слои скачиваются один раз; при обновлении скачиваются только изменённые слои.
6.  **Volumes для сохранения данных:** контейнеры эфемерны; используйте `-v` для привязки папки хоста к контейнеру.
7.  **`docker run` — основная команда:** `-d` (фон), `-p` (порты), `-e` (переменные), `--name` (имя), `--rm` (удалить после).
8.  **Очистка через `prune`:** `docker container prune` удаляет остановленные контейнеры; `--rm` делает это автоматически.

## Глоссарий терминов урока

| Термин | Расшифровка | Простое объяснение |
|--------|-------------|-------------------|
| **Docker** | — | Платформа для запуска приложений в изолированных контейнерах. |
| **VM** | Virtual Machine | Виртуальная машина с полной гостевой ОС и эмуляцией железа. |
| **Hypervisor** | Гипервизор | Программа для создания и управления виртуальными машинами (VMware, VirtualBox). |
| **Guest OS** | Гостевая ОС | Операционная система, установленная внутри виртуальной машины. |
| **Host OS** | Хостовая ОС | Основная операционная система на физическом сервере. |
| **Container** | Контейнер | Изолированный процесс со своей ФС, сетью и переменными окружения. |
| **Image** | Образ | Неизменяемый шаблон для создания контейнеров (слои ФС). |
| **Registry** | Реестр | Хранилище образов (Docker Hub, приватные реестры). |
| **Docker Hub** | — | Публичный реестр образов от Docker Inc. (hub.docker.com). |
| **Docker Engine** | — | Движок Docker, управляющий контейнерами на хосте. |
| **Namespaces** | Пространства имён | Механизм ядра Linux для изоляции процессов, сети, пользователей, ФС. |
| **Cgroups** | Control Groups | Механизм ядра Linux для ограничения ресурсов (CPU, RAM, диск). |
| **Layers** | Слои | Уровни файловой системы образа; каждый слой — изменение. |
| **Bind Mount** | — | Тип тома: прямая привязка папки хоста к папке в контейнере. |
| **Volume** | Том | Способ хранения данных контейнера вне его эфемерной ФС. |
| **Эфемерный** | Ephemeral | Временный, исчезающий после удаления контейнера. |
| **Detached mode** | Фоновый режим | Запуск контейнера в фоне (`-d`), без привязки к терминалу. |
| **Interactive mode** | Интерактивный режим | Режим с подключением к терминалу контейнера (`-it`). |
| **Port forwarding** | Проброс портов | Перенаправление трафика с порта хоста на порт контейнера (`-p`). |
| **glibc** | GNU C Library | Основная библиотека C, от которой зависят программы в Linux. |
| **Runtime** | Среда выполнения | Библиотеки и зависимости, необходимые для запуска приложения. |
| **Prune** | Очистка | Удаление неиспользуемых объектов (контейнеров, образов, томов). |
| **ISO** | International Organization for Standardization | Формат образа оптического диска (часто используется для ОС). |
| **Overhead** | Накладные расходы | Дополнительные ресурсы, потребляемые системой (VM тяжелее контейнера). |
