Представьте дата-центр с тысячами серверов. Чтобы настроить базу данных или обновить приложение, вам нужен «пропуск». В цифровом мире таким пропуском является **SSH (Secure Shell)**.

Для DevOps-инженера или разработчика SSH — это не просто команда, а экосистема протоколов, обеспечивающая безопасность всей современной интернет-инфраструктуры. В этом уроке мы пройдем путь от основ криптографии до настройки сложных туннелей, превратив SSH в ваш любимый швейцарский нож.

## Как работает SSH под капотом

SSH (Secure Shell) был создан в 1995 году Тату Юлёненом как замена уязвимым протоколам Telnet и rlogin, где данные передавались открытым текстом. SSH решил эту проблему с помощью шифрования.

### Метафора замка и ключа (Асимметричное шифрование)

Основная концепция SSH — **асимметричное шифрование**. Представьте это как работу с навесными замками.

**Публичный ключ** (Public Key) — это открытый замок. Вы можете раздать тысячи таких замков кому угодно. Если кто-то хочет отправить вам секретный груз, он запирает его вашим замком. Теперь никто, даже отправитель, не сможет открыть ящик.

**Приватный ключ** (Private Key) — это единственный физический ключ от ваших замков. Вы храните его в строгом секрете на своем устройстве.

В SSH это работает так:
1. Вы копируете «замок» (публичный ключ) на сервер.
2. При входе сервер «запирает» случайное проверочное число вашим замком.
3. Вы открываете замок своим «ключом» (приватным ключом) и возвращаете число.
4. Сервер убеждается, что вы — владелец, и впускает вас.

```text
      Локальная машина (Client)                Удаленный сервер (Server)
    +-------------------------+              +-------------------------+
    | [Приватный ключ (Key)]  |              | [Публичный ключ (Lock)] |
    |           |             |              |           |             |
    |    1. Запрос на вход    |------------->|           |             |
    |                         |              |    2. Шифрование числа  |
    |    3. Дешифрование      |<-------------|       этим "замком"     |
    |       своим ключом      |              |                         |
    |                         |              |                         |
    |    4. Возврат числа     |------------->|    5. Проверка: ОК      |
    +-------------------------+              +-------------------------+
```

### Симметричное шифрование и сессия

Асимметричное шифрование очень надежное, но медленное. Поэтому SSH использует его только в самом начале, чтобы «познакомиться» и договориться. Как только личности подтверждены, клиент и сервер создают временный «общий секрет» (сеансовый ключ). 

С этого момента используется **симметричное шифрование** (например, алгоритм AES). Это похоже на то, как если бы вы с сервером придумали уникальный шифр для этой конкретной беседы и сожгли инструкции после её завершения. Это происходит мгновенно и позволяет передавать гигабайты данных без задержек.

## Генерация ключевой пары

Первым делом нам нужно создать те самые «замок» и «ключ». Раньше стандартом был алгоритм RSA, но сегодня он считается громоздким и менее эффективным. Современный стандарт в мире DevOps — это **Ed25519**.

### Почему мы выбираем Ed25519?

1.  **Скорость:** Он работает значительно быстрее RSA.
2.  **Безопасность:** Основан на эллиптических кривых и устойчив к атакам, которые могут быть опасны для RSA.
3.  **Компактность:** Публичный ключ Ed25519 — это короткая строка (около 68 символов), в то время как RSA 4096 занимает пол-экрана.

### Создаем ключи

Откройте терминал на вашей локальной машине (Linux, macOS или Windows PowerShell) и введите команду:

```bash
ssh-keygen -t ed25519 -a 100 -C "your_email@example.com"
```

Разберем параметры:
*   `-t ed25519`: Указывает тип ключа.
*   `-a 100`: Количество раундов хеширования (KDF). Это делает «подбор» вашего пароля к ключу на диске в сто раз сложнее.
*   `-C "your_email@example.com"`: Комментарий, обычно это ваша почта или название устройства (например, "work-laptop"). Он поможет вам позже понять, чей это ключ в файле на сервере.

### Самый важный момент: Passphrase

Когда программа спросит `Enter passphrase (empty for no passphrase)`, **не нажимайте Enter просто так**. 

**Passphrase (кодовая фраза)** — это пароль, которым будет зашифрован ваш приватный ключ на диске. Если злоумышленник украдет ваш файл ключа, но у вас нет passphrase, он получит полный доступ ко всем вашим серверам. Если же ключ защищен фразой, у вас будет время сменить ключи на серверах, пока взломщик пытается подобрать пароль.

В результате в директории `~/.ssh/` появятся два файла:
1.  `id_ed25519` — ваш **приватный ключ**. Это святая святых. Никогда не передавайте его через мессенджеры, не выкладывайте в Git и не копируйте на флешки без крайней необходимости.
2.  `id_ed25519.pub` — ваш **публичный ключ**. Это ваш «замок», который вы будете раздавать серверам.

## Копирование публичного ключа на сервер

Теперь нам нужно установить наш «замок» на сервер. Чтобы сервер разрешил вам вход по ключу, ваш публичный ключ должен быть записан в файл `~/.ssh/authorized_keys` в домашней директории того пользователя, под которым вы входите.

### Способ 1: Автоматический (ssh-copy-id)

Если вы работаете в Linux или macOS, используйте встроенную утилиту:

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server-ip
```

Утилита сама подключится к серверу, создаст нужные папки, установит правильные права доступа (это критично!) и добавит ваш ключ в список разрешенных.

### Способ 2: Ручной

Если утилиты нет (например, вы на Windows без специального софта), можно сделать это одной командой, выполнив её на **локальном компьютере**:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server-ip "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

Эта «магическая» строка делает следующее: `cat` на вашей машине считывает публичный ключ, передает его через SSH-туннель на сервер, где вторая команда `cat` (уже внутри кавычек) принимает эти данные из стандартного ввода и дописывает их в файл `authorized_keys`.

**Обратите внимание на права доступа:** 
*   Папка `.ssh` должна иметь права `700` (только владелец может читать, писать и заходить).
*   Файл `authorized_keys` должен иметь права `600` (только владелец может читать и писать). 
Если права будут слишком свободными (например, `777`), SSH из соображений безопасности просто откажется работать с таким файлом.

## Отключаем пароли — только ключи

Поздравляю, теперь вы можете заходить на сервер без ввода пароля пользователя (вам нужно будет вводить только passphrase вашего ключа, если вы её установили). Но сервер все еще уязвим: боты со всего мира прямо сейчас пытаются подобрать пароль к вашему пользователю `root` или `admin`.

Чтобы закрыть эту дыру, мы полностью отключим вход по паролю.

1.  Зайдите на сервер: `ssh user@server-ip`.
2.  Откройте конфиг SSH-сервера: `sudo nano /etc/ssh/sshd_config`.
3.  Найдите и измените следующие параметры (убедитесь, что они не закомментированы символом `#`):

```ssh
# Отключаем аутентификацию по паролю
PasswordAuthentication no

# Отключаем пустые пароли
PermitEmptyPasswords no

# Запрещаем вход пользователю root напрямую (рекомендуется)
PermitRootLogin no

# Убеждаемся, что вход по ключам разрешен
PubkeyAuthentication yes
```

**Внимание! Опасность!** 
1. Перед отключением входа для `root` (`PermitRootLogin no`), убедитесь, что вы создали другого пользователя и добавили его в группу **sudo** (Ubuntu/Debian) или **wheel** (RHEL/CentOS), чтобы не потерять доступ к управлению сервером.
2. Перед тем как сохранить файл и перезапустить сервис, убедитесь, что вы открыли второе окно терминала и уже зашли по SSH-ключу. Если вы совершите ошибку в конфиге и закроете текущую сессию, вы можете навсегда потерять доступ к серверу (придется использовать консоль восстановления в панели провайдера).

4.  Проверьте конфиг на ошибки:
    ```bash
    sudo sshd -t
    ```
5.  Если ошибок нет, перезапустите службу. Обратите внимание, что название сервиса зависит от вашего дистрибутива:
    *   **Ubuntu/Debian:** `sudo systemctl restart ssh`
    *   **RHEL/CentOS/Fedora:** `sudo systemctl restart sshd`

Теперь ваш сервер — настоящая крепость. Злоумышленник даже не увидит приглашения ввести пароль, соединение будет обрываться сразу после неудачной попытки аутентификации по ключу.

## ~/.ssh/config — ваш менеджер подключений

Со временем у вас появится 5, 10, а то и 50 серверов. Запоминать их IP-адреса, порты и имена пользователей — невозможно. Конечно, можно использовать Excel или заметки, но SSH предлагает встроенное элегантное решение — файл конфигурации.

Создайте файл (или откройте существующий) на вашей локальной машине: `nano ~/.ssh/config`.

Пример настройки:

```ssh
# Общие настройки для всех хостов
Host *
  AddKeysToAgent yes
  ServerAliveInterval 60
  VisualHostKey yes

# Основной рабочий сервер
Host prod-app
  HostName 95.163.210.44
  User deploy
  Port 22
  IdentityFile ~/.ssh/id_ed25519

# Сервер базы данных с нестандартным портом
Host dev-db
  HostName db.example.com
  User postgres
  Port 54322
  IdentityFile ~/.ssh/id_ed25519_db

# Пример Jump-хоста (доступ к внутреннему серверу через шлюз)
Host internal-node
  HostName 10.0.1.50
  User admin
  ProxyJump prod-app
```

Теперь вместо длинной команды:
```bash
ssh -p 54322 postgres@db.example.com -i ~/.ssh/id_ed25519_db
```
Вам достаточно набрать:
```bash
ssh dev-db
```

Такое короткое имя называется **Host** (иногда его называют «алиасом» SSH-подключения, но не путайте его с алиасами команд в `.bashrc`). Это не только экономит время, но и позволяет использовать автодополнение в терминале.

## Передача файлов через SCP и SFTP

SSH умеет не только передавать команды, но и файлы. Для этого есть два основных инструмента.

### SCP (Secure Copy)

Самый быстрый и простой способ. Синтаксис почти такой же, как у обычной команды `cp`, только добавляется адрес хоста.

*   **Загрузить файл на сервер:**
```bash
scp local-file.txt prod-app:/home/deploy/
```
*   **Скачать папку с сервера:**
```bash
scp -r prod-app:/var/log/nginx/ ./backup_logs/
```

### SFTP (SSH File Transfer Protocol)

Это интерактивный протокол, похожий на FTP, но работающий через защищенный канал SSH. Он удобен, если вам нужно не просто скопировать файл, а «побродить» по папкам сервера, переименовать что-то или удалить.

Запуск:
```bash
sftp prod-app
```
Внутри работают привычные команды: `ls`, `cd`, `mkdir`, `rm`, а для передачи используются `put` (загрузить) и `get` (скачать).

## SSH-туннели: проброс портов

Это, пожалуй, самая «магическая» часть SSH. Представьте ситуацию: на сервере запущена база данных, которая слушает только `localhost` (внутренний адрес сервера). Снаружи к ней подключиться нельзя. Или вам нужно показать заказчику сайт, который запущен у вас на ноутбуке, но у вас нет публичного IP.

Для этих целей существуют туннели.

### Local Port Forwarding (`-L`)
*Сценарий: Хочу подключиться к базе на сервере из своей локальной среды.*

```bash
ssh -L 5432:localhost:5432 prod-app
```
**Как это работает:** Ваш компьютер начинает «слушать» порт 5432 локально. Все данные, приходящие на этот порт, SSH упаковывает, отправляет на `prod-app` и там разворачивает, направляя их на `localhost:5432`. **Важно:** в этом контексте `localhost` — это адрес **относительно удаленного сервера**, то есть обращение к самой машине `prod-app` изнутри. Для вашей IDE или GUI-клиента базы данных это выглядит так, будто база запущена прямо у вас на компьютере.

### Remote Port Forwarding (`-R`)
*Сценарий: Хочу пробросить свой локальный сайт на внешний сервер.*

```bash
ssh -R 8080:localhost:80 prod-app
```
Теперь вы сможете зайти на `prod-app:8080` и попасть на ваш локальный сервер, работающий на порту 80. Здесь `localhost` относится к **вашему локальному компьютеру** — это точка назначения, куда сервер будет пересылать трафик. Это незаменимо для тестирования вебхуков или демонстрации работы.

**Важное уточнение:** По умолчанию OpenSSH привязывает проброшенный порт к `localhost` самого сервера. Это значит, что сайт будет доступен только внутри сервера `prod-app`. Чтобы порт стал доступен "всему интернету" (привязка к внешнему IP), в конфиге сервера `/etc/ssh/sshd_config` должна быть установлена опция `GatewayPorts yes`.

## SSH-agent: Избавляемся от повторного ввода пароля

Если ваш ключ защищен длинной passphrase (а мы договорились, что это так), вводить её при каждом подключении утомительно. Для этого существует `ssh-agent` — фоновый процесс, который один раз запрашивает пароль, расшифровывает ключ и хранит его в оперативной памяти.

### Как запустить агент

**В Linux или macOS (bash/zsh):**
```bash
eval $(ssh-agent -s)
ssh-add ~/.ssh/id_ed25519
```

**В Windows (PowerShell):**
Запуск агента в PowerShell отличается, так как это системная служба:
```powershell
# Запускаем службу
Start-Service ssh-agent
# Добавляем ключ
ssh-add $HOME\.ssh\id_ed25519
```
*Если вы используете **Git Bash** на Windows, используйте те же команды, что и для Linux.*

Теперь, пока вы не перезагрузите компьютер, SSH будет сам брать ключ из памяти агента. В современных системах (macOS, Ubuntu с GNOME) это часто настроено автоматически и интегрировано с системным хранилищем ключей.

## Отладка: Когда что-то идет не так

Даже опытные инженеры иногда сталкиваются с тем, что SSH «висит» или выдает `Permission denied`. Главное правило — не гадать, а смотреть логи.

Используйте флаги детализации для поиска причин:

```bash
# Базовая информация (какие ключи пробует клиент)
ssh -v user@host

# Более подробно
ssh -vv user@host

# Максимальная детализация (весь процесс рукопожатия)
ssh -vvv user@host
```

**Типичные ошибки:**
1.  **Permission denied (publickey):** Ваш ключ не добавлен в `authorized_keys` или у файлов/папок на сервере неверные права доступа.
2.  **Connection timed out:** Сервер недоступен, неверный IP или порт закрыт файрволом. Проверьте настройки **ufw** или **iptables** — это стандартные программы-брандмауэры в Linux, которые могут блокировать входящие соединения.
3.  **Host key verification failed:** Вы заходите на сервер, IP которого раньше принадлежал другому серверу. SSH видит, что «отпечаток» замка изменился, и бьет тревогу (защита от атаки Man-in-the-Middle). Если вы уверены, что сервер переустанавливали, удалите старый отпечаток командой:
```bash
ssh-keygen -R server-ip
```

## Безопасность и антипаттерны

В завершение — список того, что делает DevOps-инженера профессионалом, и того, что может стоить компании безопасности.

### ✅ Best Practices:
*   **Один человек — один ключ.** Никогда не создавайте «общий» ключ для всей команды. Если сотрудник уволится, вам придется менять ключи везде. Если у каждого свой ключ — вы просто удаляете одну строку из `authorized_keys`.
*   **Используйте Jump-хосты.** Не выставляйте все сервера в интернет. Оставьте один защищенный шлюз (Bastion host), а к остальным подключайтесь через него с помощью `ProxyJump`.
*   **Двухфакторная аутентификация (2FA).** Для критически важных систем можно настроить связку SSH-ключ + код из Google Authenticator.

### ❌ Антипаттерны:
*   **Хранение приватного ключа в GitHub.** Даже в приватных репозиториях. Скрипты-сканеры находят их за секунды.
*   **Использование root для повседневной работы.** Заходите под обычным пользователем, а права повышайте через `sudo`.
*   **Отключение проверки `known_hosts`.** Никогда не добавляйте параметры, которые игнорируют проверку подлинности сервера, если вы не понимаете на 100%, зачем это нужно.

## Заключение

Мы разобрали SSH от теории до продвинутой практики. Теперь вы знаете, что это не просто способ «зайти на сервер», а мощный и гибкий инструмент обеспечения безопасности. Помните: в мире DevOps безопасность — это не состояние, а процесс. Регулярно обновляйте свои ключи, следите за конфигами и никогда не пренебрегайте защитой приватных данных.

### Ваше домашнее задание:
1.  Сгенерируйте свою первую пару ключей Ed25519 с надежной passphrase.
2.  Настройте локальную виртуальную машину (например, в VirtualBox) или арендуйте самый дешевый VPS.
3.  Установите ключ и **полностью отключите** вход по паролю.
4.  Настройте короткое имя (**Host**) в `~/.ssh/config` для вашего сервера.
5.  Попробуйте пробросить порт 80 с сервера на ваш локальный компьютер через `-L`.

## Ключевые выводы урока

1.  **SSH = асимметричное шифрование:** публичный ключ («замок») на сервере, приватный («ключ») у вас; вместе обеспечивают безопасный вход.
2.  **Ed25519 вместо RSA:** быстрее, безопаснее, компактнее; используйте `ssh-keygen -t ed25519 -a 100`.
3.  **Passphrase обязательна:** защищает приватный ключ на диске; без неё украденный ключ = доступ ко всем серверам.
4.  **Отключите вход по паролю:** после настройки ключей измените `sshd_config` (PasswordAuthentication no, PermitRootLogin no).
5.  **`~/.ssh/config` экономит время:** задайте алиасы (Host), порты, пути к ключам — подключайтесь по короткому имени.
6.  **SCP и SFTP для файлов:** `scp` для быстрой копии, `sftp` для интерактивной работы с файлами сервера.
7.  **Туннели (-L и -R):** Local Forwarding — доступ к сервису на сервере; Remote Forwarding — доступ к локальному сервису с сервера.
8.  **ssh-agent хранит ключи:** один раз введите passphrase, агент будет подставлять ключ из памяти.

## Глоссарий терминов урока

| Термин | Расшифровка | Простое объяснение |
|--------|-------------|-------------------|
| **SSH** | Secure Shell | Протокол безопасного удалённого доступа к серверам (шифрует весь трафик). |
| **Асимметричное шифрование** | — | Шифрование парой ключей: публичный шифрует, приватный расшифровывает. |
| **Симметричное шифрование** | — | Шифрование одним ключом (быстрее, используется для сессии после «знакомства»). |
| **Публичный ключ** | Public Key | «Замок», который копируется на сервер; не секретный. |
| **Приватный ключ** | Private Key | «Ключ», который хранится у вас на компьютере; держите в секрете. |
| **Ed25519** | — | Современный алгоритм SSH-ключей (быстрый, безопасный, на эллиптических кривых). |
| **RSA** | Rivest–Shamir–Adleman | Старый алгоритм ключей; надёжный, но медленный и громоздкий. |
| **KDF** | Key Derivation Function | Функция вывода ключа; `-a 100` делает 100 раундов хеширования passphrase. |
| **Passphrase** | Кодовая фраза | Пароль для защиты приватного ключа на диске. |
| **authorized_keys** | — | Файл на сервере со списком публичных ключей, которым разрешён вход. |
| **sshd_config** | SSH Daemon Config | Файл конфигурации SSH-сервера (`/etc/ssh/sshd_config`). |
| **ProxyJump** | — | Опция SSH для подключения через промежуточный сервер (Jump-хост). |
| **SCP** | Secure Copy | Утилита копирования файлов через SSH (как `cp`, но через сеть). |
| **SFTP** | SSH File Transfer Protocol | Интерактивный протокол передачи файлов через SSH (как FTP, но защищён). |
| **Local Port Forwarding** | Проброс локального порта | `-L`: доступ к сервису на сервере через локальный порт. |
| **Remote Port Forwarding** | Проброс удалённого порта | `-R`: доступ к локальному сервису с сервера через порт на сервере. |
| **ssh-agent** | SSH Agent | Фоновый процесс, хранящий расшифрованные ключи в памяти. |
| **known_hosts** | — | Файл с «отпечатками» серверов; SSH предупреждает, если отпечаток изменился. |
| **Man-in-the-Middle** | Человек посередине | Атака, когда злоумышленник перехватывает соединение между клиентом и сервером. |
| **Jump-хост** | Bastion Host | Промежуточный сервер-шлюз для доступа к внутренней сети. |
| **2FA** | Two-Factor Authentication | Двухфакторная аутентификация: ключ + код из приложения. |
| **ufw** | Uncomplicated Firewall | Простой брандмауэр в Ubuntu (может блокировать порт 22). |
| **iptables** | — | Система фильтрации трафика в Linux (может блокировать соединения). |

## Чек-лист готовности урока

- [ ] Объяснить разницу между асимметричным и симметричным шифрованием.
- [ ] Сгенерировать пару ключей Ed25519 через `ssh-keygen -t ed25519 -a 100 -C "email"`.
- [ ] Установить надёжную passphrase для приватного ключа.
- [ ] Скопировать публичный ключ на сервер через `ssh-copy-id`.
- [ ] Проверить права на `~/.ssh` (700) и `authorized_keys` (600) на сервере.
- [ ] Отключить вход по паролю в `/etc/ssh/sshd_config` (PasswordAuthentication no).
- [ ] Запретить вход root напрямую (PermitRootLogin no).
- [ ] Проверить конфиг SSH-сервера через `sudo sshd -t`.
- [ ] Перезапустить SSH-сервер (`systemctl restart ssh` или `sshd`).
- [ ] Настроить алиасы в `~/.ssh/config` (Host, HostName, User, Port, IdentityFile).
- [ ] Подключиться по короткому имени (`ssh prod-app`).
- [ ] Загрузить файл на сервер через `scp`.
- [ ] Скачать папку с сервера через `scp -r`.
- [ ] Подключиться через `sftp` и выполнить `put`/`get`.
- [ ] Пробросить локальный порт на сервер через `ssh -L`.
- [ ] Пробросить удалённый порт через `ssh -R`.
- [ ] Запустить ssh-agent и добавить ключ через `ssh-add`.
- [ ] Использовать отладку через `ssh -vvv` при проблемах.
- [ ] Удалить старый отпечаток сервера через `ssh-keygen -R`.
- [ ] Объяснить, почему нельзя хранить приватный ключ в Git.
- [ ] Объяснить преимущество Jump-хоста для безопасности.
