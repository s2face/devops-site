# SSH от А до Я: безопасный доступ к серверам

SSH (Secure Shell) — это не просто «команда для входа на сервер». Это фундаментальный протокол сетевого администрирования, обеспечивающий защищённый канал связи между клиентом и сервером в незащищённой сети. Если вы планируете работать в DevOps или системном администрировании, SSH станет вашим основным инструментом на ближайшие годы.

В этом уроке мы разберем, как работает SSH «под капотом», научимся генерировать современные ключи, настраивать удобные конфиги и использовать SSH-туннели для решения сложных сетевых задач.

---

## История и эволюция протокола

Прежде чем погрузиться в практику, важно понимать контекст. До появления SSH в 1995 году администраторы использовали такие протоколы, как **Telnet**, **rlogin** и **rsh**. Главная проблема этих протоколов заключалась в том, что данные (включая логины и пароли) передавались в открытом виде (plain text). Любой злоумышленник, имеющий доступ к сетевому трафику между вами и сервером, мог легко перехватить ваши учетные данные.

Тату Юлёнен (Tatu Ylönen), исследователь из Технологического университета Хельсинки, разработал первую версию SSH после того, как в сети его университета произошла атака с перехватом паролей.

Сегодня мы используем **SSH-2**, который был стандартизирован в 2006 году и полностью заменил уязвимый SSH-1. Современные реализации, такие как **OpenSSH**, стали де-факто стандартом в мире Linux и Unix-подобных систем.

## Как работает SSH под капотом

В основе безопасности SSH лежит комбинация различных криптографических методов: симметричное шифрование, асимметричное шифрование и хеширование.

### 1. Асимметричное шифрование (Public Key Cryptography)

Это основа процесса аутентификации. Здесь используется пара ключей:

*   **Приватный ключ (Private Key):** Хранится только у вас на локальной машине. Его нельзя передавать никому. Он используется для создания цифровой подписи и расшифровки данных, зашифрованных вашим публичным ключом.
*   **Публичный ключ (Public Key):** Копируется на все серверы, к которым вы хотите иметь доступ. Он используется для проверки вашей цифровой подписи и зашифровки данных, которые сможете прочитать только вы.

### 2. Симметричное шифрование

Асимметричное шифрование вычислительно дорого. Поэтому SSH использует его только для аутентификации и согласования сеансового ключа. Как только соединение установлено, все данные передаются с использованием **симметричного шифрования** (например, AES или ChaCha20). Сеансовый ключ генерируется заново для каждой сессии и уничтожается после её завершения.

### 3. Обмен ключами (Diffie-Hellman)

Чтобы клиент и сервер могли договориться о секретном сеансовом ключе, не передавая его по сети, используется алгоритм Диффи-Хеллмана. Это математический фокус, позволяющий двум сторонам создать общий секрет через открытый канал связи так, что перехватчик не сможет его вычислить.

### Схема аутентификации (Key-based)

Процесс подключения выглядит так:

1.  **Запрос:** Клиент инициирует соединение и сообщает серверу, что хочет зайти под пользователем `devops` с определенным публичным ключом.
2.  **Challenge:** Сервер проверяет файл `~/.ssh/authorized_keys` пользователя. Если ключ там есть, он генерирует случайное число (nonce), шифрует его публичным ключом клиента и отправляет обратно.
3.  **Response:** Клиент расшифровывает сообщение своим приватным ключом и отправляет результат серверу (вместе с другими данными сессии для предотвращения атак повтора).
4.  **Success:** Сервер сравнивает результаты. Если они совпали — личность клиента подтверждена, доступ разрешен.

## Генерация ключевой пары

Забудьте про старый формат RSA 2048/4096. Хотя он все еще безопасен при достаточной длине ключа, современный стандарт — **Ed25519**.

### Почему Ed25519?
1.  **Безопасность:** Основан на эллиптических кривых (Curve25519), устойчив к многим известным атакам.
2.  **Скорость:** Генерируется и проверяется быстрее, чем RSA.
3.  **Компактность:** Публичный ключ Ed25519 очень короткий (всего 68 символов против 500+ у RSA).

Генерируем ключ:

```bash
ssh-keygen -t ed25519 -a 100 -C "devops@work-laptop"
```

Разберем флаги:
*   `-t ed25519` — тип алгоритма.
*   `-a 100` — количество раундов хеширования для passphrase. Чем больше число, тем сложнее будет подобрать пароль к приватному ключу методом перебора (брутфорс файла на диске).
*   `-C "comment"` — комментарий. Помогает идентифицировать ключ (например, с какого устройства он был создан).

**Важно: Passphrase**. При генерации вас спросят пароль. **Никогда не оставляйте это поле пустым.** Passphrase шифрует приватный ключ на вашем диске. Даже если кто-то украдет файл вашего ключа, он не сможет им воспользоваться без этого пароля.

В результате вы получите два файла в `~/.ssh/`:
*   `id_ed25519` — приватный ключ (права доступа должны быть строго 600).
*   `id_ed25519.pub` — публичный ключ (права 644).

## Копирование публичного ключа на сервер

Чтобы сервер «узнал» ваш ключ, его нужно добавить в файл `~/.ssh/authorized_keys` конкретного пользователя на стороне сервера.

### Способ 1: Использование ssh-copy-id (Рекомендуемый)

Это самая удобная утилита, которая сама проверит наличие папки `.ssh`, создаст файл с нужными правами и добавит ключ:

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server-ip
```

### Способ 2: Вручную (если нет утилиты)

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server-ip "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

После этого попробуйте зайти: `ssh user@server-ip`. Если вы установили passphrase, система ее запросит.

## Настройка сервера: /etc/ssh/sshd_config

Безопасность SSH по умолчанию в большинстве дистрибутивов — это «минимально допустимый уровень». Для реального продакшена настройки нужно ужесточить.

Отредактируйте файл конфигурации демона SSH на сервере:

```bash
sudo nano /etc/ssh/sshd_config
```

### Основные параметры безопасности:

1.  **Отключаем пароли:**
    ```ssh
    PasswordAuthentication no
    ChallengeResponseAuthentication no
    ```
    Это самая важная настройка. Боты со всего интернета будут пытаться подобрать пароль к вашему SSH. Если пароли отключены, им просто не за что зацепиться.

2.  **Ограничиваем вход для Root:**
    ```ssh
    PermitRootLogin no
    ```
    Заходить напрямую под root — плохая практика. Заходите под обычным пользователем и используйте `sudo`.

3.  **Белые списки пользователей:**
    ```ssh
    AllowUsers devops admin
    ```
    Только перечисленные пользователи смогут войти. Все остальные будут отклонены.

4.  **Смена порта (Security by obscurity):**
    ```ssh
    Port 2222
    ```
    Смена порта с 22 на что-то другое не спасет от направленной атаки, но уберет 99% автоматизированного шума в логах.

**Внимание:** Перед перезагрузкой проверьте конфиг:
```bash
sudo sshd -t
```
Если всё в порядке, перезапускайте:
```bash
sudo systemctl restart ssh
```

## ~/.ssh/config — магия удобства

Вы не должны запоминать IP-адреса и порты. Файл `~/.ssh/config` позволяет создавать алиасы для ваших серверов.

Пример расширенного конфига:

```ssh
# Общие настройки для всех хостов
Host *
  AddKeysToAgent yes
  UseKeychain yes # Для macOS
  ServerAliveInterval 60 # Предотвращает разрыв соединения

# Рабочий сервер через нестандартный порт
Host prod-db
  HostName 1.2.3.4
  User admin
  Port 2222
  IdentityFile ~/.ssh/id_ed25519

# Сервер во внутренней сети через Jump-хост
Host internal-app
  HostName 10.0.1.5
  User devops
  ProxyJump prod-db
```

Теперь вместо длинных команд вам достаточно набрать:
*   `ssh prod-db`
*   `ssh internal-app` (SSH сам сначала подключится к `prod-db`, а через него пробросит туннель к `internal-app`).

## Передача файлов: scp и sftp

### SCP (Secure Copy)

Работает быстро и просто, синтаксис похож на `cp`.

Копирование на сервер:
```bash
scp image.png prod-db:/var/www/static/
```

Копирование с сервера:
```bash
scp prod-db:/etc/nginx/nginx.conf ./
```

### SFTP (SSH File Transfer Protocol)

Более продвинутый протокол. В отличие от SCP, он может возобновлять прерванные передачи и выполнять операции над файловой системой (создание папок, удаление и т.д.).

```bash
sftp prod-db
# Доступные команды: ls, cd, put (загрузить), get (скачать), rm, mkdir
```

## SSH-agent: Забудьте о вводе пароля

Если у вас 10 серверов и на каждом ключе стоит passphrase, вводить её каждый раз — мучение. `ssh-agent` — это программа, которая хранит ваши расшифрованные ключи в памяти.

1.  Запустите агент (обычно он запускается автоматически при входе в систему):
    ```bash
    eval `ssh-agent -s`
    ```
2.  Добавьте свой ключ:
    ```bash
    ssh-add ~/.ssh/id_ed25519
    ```
Теперь, пока вы не перезагрузитесь или не закроете сессию терминала, SSH будет брать ключ из агента автоматически.

## SSH-туннели: Решение сложных сетевых задач

### 1. Local Port Forwarding (`-L`)
Вы хотите подключиться к базе данных PostgreSQL на сервере, но порт 5432 закрыт файрволом для внешних подключений.

```bash
ssh -L 5433:localhost:5432 prod-db
```
**Что произошло?** Ваш локальный порт 5433 теперь «связан» с портом 5432 на сервере `prod-db`. Вы можете открыть DataGrip или pgAdmin, подключиться к `localhost:5433`, и вы попадете в базу на сервере.

### 2. Remote Port Forwarding (`-R`)
Вам нужно показать веб-сайт, запущенный у вас локально (порт 8080), коллеге в другом городе. У вас нет внешнего IP.

```bash
ssh -R 8080:localhost:8080 prod-db
```
Теперь любой, кто зайдет на `prod-db:8080` в браузере, попадет на ваш локальный компьютер.

### 3. Динамический туннель (SOCKS Proxy)
Хотите, чтобы ваш браузер работал через сервер (например, для доступа к ресурсам, доступным только из сети ДЦ)?

```bash
ssh -D 1080 prod-db
```
Настройте в браузере SOCKS5 прокси на `localhost:1080`, и весь ваш трафик пойдет через SSH-туннель.

## Отладка (Debugging)

Если соединение не устанавливается, не гадайте. Используйте флаг `-v` (verbose):

*   `ssh -v user@host` — базовая информация.
*   `ssh -vv user@host` — более подробно.
*   `ssh -vvv user@host` — максимум подробностей, включая процесс согласования алгоритмов шифрования.

Частые проблемы:
*   `Permission denied (publickey)` — сервер не принял ваш ключ. Проверьте `authorized_keys` на сервере и какой ключ отправляет клиент.
*   `Connection timed out` — скорее всего, порт закрыт файрволом (iptables/ufw) или неверный IP.
*   `Offending key in /home/user/.ssh/known_hosts:10` — сервер переустановили, и его идентификатор изменился. Если вы знаете, что это легитимное изменение, удалите старую запись: `ssh-keygen -f "~/.ssh/known_hosts" -R "server-ip"`.

## Безопасность: Best Practices и Антипаттерны

### ✅ Рекомендации:
*   **Используйте Ed25519.** Это стандарт современности.
*   **Всегда используйте Passphrase.** Ключ без пароля — это огромная дыра в безопасности.
*   **Периодически проводите аудит `authorized_keys`.** Удаляйте ключи бывших сотрудников или старых устройств.
*   **Используйте 2FA для SSH.** Можно настроить Google Authenticator или использовать аппаратные ключи (Yubikey) через протокол FIDO2 (ключи типа `ed25519-sk`).

### ❌ Антипаттерны:
*   **Хранение приватных ключей в Git.** Даже в приватных репозиториях. Это смертный грех DevOps-инженера.
*   **Рассылка одного и того же ключа всей команде.** У каждого сотрудника должен быть свой персональный ключ.
*   **Игнорирование `known_hosts`.** Если SSH предупреждает об изменении ключа хоста, не вводите `yes` машинально. Сначала выясните, почему он изменился.
*   **Использование RSA 1024.** Эти ключи уже давно считаются небезопасными и ломаются современными мощностями.
