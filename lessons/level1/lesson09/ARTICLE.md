# Сети для DevOps: IP, DNS, TCP/UDP — без воды

Добро пожаловать на девятый урок нашего курса. Если вы дошли до этого момента, значит, вы уже умеете базово работать с Linux и Bash. Но современный DevOps-инженер не может существовать в вакууме одного сервера. Вся магия облаков, Kubernetes, микросервисов и CI/CD строится на одном фундаменте — **компьютерных сетях**.

В этом уроке мы не будем заучивать сухую теорию тридцатилетней давности. Мы разберем ровно те инструменты и концепции, которые понадобятся вам, когда в 3 часа ночи сработает алерт "Service Unavailable", и вам нужно будет понять: упало приложение, "отвалилась" сеть или кто-то неправильно настроил DNS.

## Модель OSI — только то, что нужно знать

Когда говорят о сетях, всегда вспоминают модель OSI (Open Systems Interconnection). Это концептуальная модель, которая описывает, как данные проходят путь от электрического сигнала в кабеле до кнопки "Купить" в вашем браузере. В ней 7 уровней, но для DevOps-инженера критически важны только три. Именно на них происходит 99% работы по настройке и отладке систем.

### L3 — Сетевой уровень (Network Layer): Адресация
Здесь живет протокол **IP** (Internet Protocol). Его главная задача — доставить пакет от точки А до точки Б, используя IP-адреса. На этом уровне работают маршрутизаторы (роутеры). Они решают, через какой узел отправить пакет, чтобы он быстрее дошел до цели.
*   **Единица данных:** Пакет.
*   **ICMP:** Протокол служебных сообщений (используется `ping`). Он сообщает об ошибках доставки (например, "узел недоступен").
*   **Что проверяем на L3:** "Видит" ли один сервер другой? Есть ли маршрут? Не блокирует ли провайдер трафик?
*   **Инструменты:** `ping`, `traceroute`, `ip route`.

### L4 — Транспортный уровень (Transport Layer): Доставка
Здесь работают **TCP** и **UDP**. Этот уровень отвечает за то, чтобы данные дошли до конкретного приложения на сервере. Для этого используются **порты**. Если IP-адрес — это адрес дома, то порт — это номер квартиры.
*   **Сокет (Socket):** Комбинация IP-адреса и порта (например, `192.168.1.10:80`). Это конечная точка соединения.
*   **Единица данных:** Сегмент (TCP) или Датаграмма (UDP).
*   **Что проверяем на L4:** Открыт ли порт? Установлено ли соединение? Нет ли потерь пакетов на уровне передачи?
*   **Инструменты:** `ss`, `netstat`, `telnet`, `nc` (netcat).

### L7 — Прикладной уровень (Application Layer): Логика
Уровень логики приложений: **HTTP**, **DNS**, **SSH**, **FTP**, **SMTP**. Здесь данные превращаются в понятные приложению запросы (например, GET-запрос к веб-серверу).
*   **Единица данных:** Данные (**Payload** — полезная нагрузка, например, текст письма или HTML-код страницы).
*   **Что проверяем на L7:** Какой статус-код вернул веб-сервер (200 OK или 502 Bad Gateway)? Правильно ли работает DNS-резолвинг? Валиден ли SSL-сертификат?
*   **Инструменты:** `curl`, `wget`, `dig`, `nslookup`.

> **Золотое правило инженера:** Диагностика всегда идет "снизу вверх". Бессмысленно копаться в конфигах Nginx (L7), если сервер не пингуется (L3).

## IP-адреса и маски подсети: Основы выживания

IP-адрес — это уникальный идентификатор устройства в сети. Мы будем говорить про IPv4, который состоит из четырех **октетов** (один октет — это 8 бит или число от 0 до 255), разделенных точками. Например: `192.168.1.15`. В двоичном виде это 32 бита информации.

### CIDR-нотация и маски подсети
Чтобы компьютеры понимали, какие адреса находятся в их "родной" локальной сети, а какие — где-то в интернете, используются маски подсети. Раньше их писали громоздко: `255.255.255.0`. Сегодня стандарт — **CIDR** (Classless Inter-Domain Routing).

CIDR — это число после слеша, которое показывает количество "замороженных" бит, относящихся к адресу сети. Остальные биты отдаются под адреса конкретных устройств (хостов).

| CIDR | Маска | Количество адресов | Доступно хостам | Описание |
| :--- | :--- | :--- | :--- | :--- |
| `/32` | `255.255.255.255` | 1 | 1 | Конкретный IP-адрес (один хост) |
| `/30` | `255.255.255.252` | 4 | 2 | Технологические линки (точка-точка) |
| `/24` | `255.255.255.0` | 256 | 254 | Стандартная сеть офиса/дома |
| `/16` | `255.255.0.0` | 65,536 | 65,534 | Большая корпоративная сеть |
| `/8` | `255.0.0.0` | 16,777,216 | 16,777,214 | Огромные облачные сети |

**Важно:** В любой подсети два адреса всегда зарезервированы:
1.  **Первый адрес (например, .0):** Адрес самой сети.
2.  **Последний адрес (например, .255):** Broadcast-адрес для рассылки всем устройствам в сети.

### Приватные и публичные адреса (RFC 1918)
Публичных IPv4 адресов на всех не хватило, поэтому мир перешел на использование "серых" (приватных) адресов внутри локальных сетей. Они не маршрутизируются в глобальном интернете.

**Диапазоны приватных адресов:**
*   **10.0.0.0/8:** Самый частый выбор для дата-центров и облаков (AWS VPC, Google Cloud). Огромный запас адресов.
*   **172.16.0.0/12:** Часто встречается в виртуализации и Docker-сетях.
*   **192.168.0.0/16:** Классика домашних роутеров.

Если вы видите у сервера IP `10.5.0.12`, знайте — снаружи по этому адресу к нему не подключиться напрямую. Для этого используются:
*   **NAT (Network Address Translation):** Технология, позволяющая многим устройствам из локальной сети выходить в интернет через один общий публичный IP-адрес роутера.
*   **VPN (Virtual Private Network):** Шифрованный "туннель", который позволяет вашему компьютеру стать частью удаленной приватной сети, как если бы вы были подключены к ней кабелем.

## L4 Протоколы: TCP против UDP

Транспортный уровень определяет, как именно данные будут передаваться. Выбор протокола зависит от того, что важнее: надежность или скорость.

### TCP (Transmission Control Protocol): Надежный курьер
TCP — это протокол с гарантией доставки. Он следит, чтобы все пакеты дошли в правильном порядке. Если пакет потерялся из-за помех, TCP отправит его повторно.

**TCP 3-way Handshake (Трехэтапное рукопожатие):**
Перед передачей полезных данных клиент и сервер "договариваются":
1.  **SYN:** Клиент посылает запрос: "Привет, хочу установить соединение, мой порядковый номер X".
2.  **SYN-ACK:** Сервер отвечает: "Привет! Запрос принял, мой номер Y, подтверждаю твой X+1".
3.  **ACK:** Клиент подтверждает: "Окей, принял твой Y+1. Погнали!".

**Где используется:** Всё, где нельзя терять ни байта — HTTP (сайты), SSH (управление), Базы данных (PostgreSQL, Redis), Почта.

### UDP (User Datagram Protocol): Быстрый почтальон
UDP не устанавливает соединение и не проверяет доставку. Он просто "выстреливает" пакетами в сторону получателя.

**Где используется:** Стриминг видео, онлайн-игры, IP-телефония. Если в видео-звонке пропадет один кадр — не страшно, главное, чтобы картинка не замирала ради повторной отправки этого кадра. Также UDP используется в DNS для скорости.

### Популярные порты (KPI: Шпаргалка)

| Порт | Протокол | Назначение |
| :--- | :--- | :--- |
| 22 | TCP | SSH (Удаленный доступ) |
| 53 | UDP/TCP | DNS (Разрешение имен) |
| 80 | TCP | HTTP (Незащищенный веб) |
| 443 | TCP | HTTPS (Защищенный веб) |
| 3306 | TCP | MySQL (База данных) |
| 5432 | TCP | PostgreSQL (База данных) |
| 6379 | TCP | Redis (Кэш) |
| 9090 | TCP | Prometheus (Метрики) |
| 8080 | TCP | Альтернативный HTTP (Java, Dev-серверы) |

## DNS: Как работает магия имен

DNS (Domain Name System) — это распределенная база данных, которая превращает понятные нам имена (google.com) в понятные машинам IP-адреса (142.250.185.206).

### Алгоритм DNS-резолвинга (по шагам) — KPI
Когда вы вводите в браузере `app.example.com`:

1.  **Локальный кэш:** Система проверяет, не запрашивали ли мы это имя недавно.
2.  **Файл /etc/hosts:** Проверка локальной "ручной" таблицы. Если там есть запись, запрос завершается.
3.  **Рекурсивный резолвер:** Запрос летит к DNS-серверу вашего провайдера или публичному (8.8.8.8).
4.  **Корневые серверы (Root):** Это глобальные "указатели", которые знают, кто отвечает за зоны верхнего уровня. Резолвер спрашивает: "Где найти зону `.com`?".
5.  **TLD серверы (Top-Level Domain):** Серверы зон верхнего уровня (`.com`, `.ru`, `.org`). Они говорят: "За домен `example.com` отвечают такие-то NS-серверы".
6.  **Авторитативный сервер:** Резолвер идет к конечному владельцу зоны (например, Cloudflare или ваш собственный сервер) и получает IP-адрес для поддомена `app`.
7.  **Кэширование:** Резолвер отдает IP вашему компьютеру и запоминает его на время, указанное в **TTL** (Time To Live).

### Основные типы записей, которые создает DevOps:
*   **A:** Главная запись. Связывает домен с IPv4.
*   **AAAA:** То же самое, но для IPv6.
*   **CNAME:** Псевдоним. `www.google.com` может быть CNAME для `google.com`.
*   **MX:** Указывает, на какой сервер слать почту для этого домена.
*   **TXT:** Любой текст. Часто используется для верификации (Google Search Console) или настроек безопасности почты (SPF/DKIM).

### Где живут настройки DNS в Linux?
Как DevOps-инженер, вы будете постоянно сталкиваться с двумя файлами, которые определяют, как ваша система видит мир.

1.  **/etc/hosts:** Это "ручной" DNS. Здесь вы можете принудительно назначить IP любому домену.
    ```text
    127.0.0.1   localhost
    10.0.5.50   my-database.local
    ```
    Записи в этом файле имеют приоритет над DNS-серверами.

2.  **/etc/resolv.conf:** Здесь указано, к каким DNS-серверам обращаться системе.
    ```text
    nameserver 8.8.8.8
    nameserver 1.1.1.1
    ```
    **Важно:** В Ubuntu этот файл часто управляется `systemd-resolved`. Чтобы изменения сохранились навсегда, их нужно вносить в конфиги Netplan (файлы в `/etc/netplan/*.yaml`):
    ```yaml
    # Пример куска конфига netplan
    nameservers:
      addresses: [8.8.8.8, 1.1.1.1]
    ```

## Практика: Инструментарий DevOps-инженера

Прежде всего, убедитесь, что у вас установлены все необходимые инструменты. В минимальных образах их может не быть:
```bash
sudo apt update && sudo apt install traceroute dnsutils netcat-openbsd tcpdump -y
```

### 1. Работа с IP и маршрутами
Забудьте команду `ifconfig`. Используйте `ip`.

```bash
# Просмотр всех интерфейсов и их IP
ip addr show
```

Как узнать, через какой шлюз (роутер) мы выходим в мир?
```bash
ip route
# Пример вывода: default via 10.0.2.2 dev eth0
# default — маршрут по умолчанию (для всего трафика, которому нет места в локальной сети)
# via 10.0.2.2 — адрес шлюза (вашего роутера)
# dev eth0 — через какой сетевой интерфейс уйдет пакет
```

### 2. Проверка связности (`ping`, `traceroute`)
`ping` — проверка "жив или мертв" через ICMP.
```bash
ping -c 4 8.8.8.8
```

Если сервер пингуется, но соединение тормозит или обрывается, поможет `traceroute`. Она покажет каждый промежуточный узел на пути к цели.
```bash
# -n чтобы не тратить время на резолвинг имен промежуточных узлов (так быстрее)
traceroute -n google.com
```

### 3. Проверка портов и сокетов (`ss` и `nc`)
Утилита `ss` пришла на смену `netstat`. Она быстрее и информативнее. Для просмотра системных процессов используйте `sudo`.
```bash
# -t (tcp), -l (listening), -n (numeric), -p (processes)
sudo ss -tlnp

# Вывод покажет, какие процессы какие порты слушают:
# LISTEN 0 128 0.0.0.0:22 0.0.0.0:* users:(("sshd",pid=840,fd=3))
```

Чтобы проверить, открыт ли порт на удаленном сервере (не заходя на него), используйте `nc` (Netcat):
```bash
# -z (сканирование, не слать данные), -v (подробно)
nc -zv 1.1.1.1 443
# Connection to 1.1.1.1 443 port [tcp/https] succeeded!
```

### 4. Отладка DNS (`dig`)
`dig` — золотой стандарт. Показывает не только IP, но и время ответа, и через какой сервер прошел запрос.
```bash
# Короткий ответ (только IP)
dig google.com +short
```

### 5. Анализ трафика (`tcpdump`)
Когда нужно смотреть "в провода". `tcpdump` перехватывает пакеты в реальном времени.
```bash
# -i eth0 (интерфейс), port 80 (фильтр порта)
# -n (не превращать IP в имена), -A (выводить содержимое в ASCII — читаемом тексте)
sudo tcpdump -i eth0 port 80 -n -A
```

## Настройка Firewall с помощью UFW

В Ubuntu используется **UFW** (Uncomplicated Firewall).

**Базовая настройка:**
```bash
# 1. Сначала разрешаем SSH, иначе потеряем доступ к серверу!
sudo ufw allow 22/tcp

# 2. Разрешаем веб-трафик
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# 3. Включаем фаервол
sudo ufw enable
```

## Алгоритм диагностики: "Сеть не работает" (KPI)

1.  **Проверка L3 (Пинг):** Пингуем IP сервера.
    *   *Не идет?* Проверяем настройки сети на сервере (`ip addr`) и Firewall провайдера.
2.  **Проверка DNS:** Пингуем домен сайта.
    *   *По IP идет, а по домену нет?* Проблемы с DNS-записями или `resolv.conf`.
3.  **Проверка L4 (Порты):** На самом сервере запускаем `sudo ss -tlnp`.
    *   *Nginx не слушает 80 порт?* Проблема в конфиге Nginx.
    *   *Слушает, но извне не зайти?* Проверяем локальный Firewall (`sudo ufw status`).
4.  **Проверка L7 (Приложение):** Выполняем `curl -I http://localhost` на сервере.
    *   *Флаг -I (I-uppercase) запрашивает только HTTP-заголовки. Для полной отладки используйте -v.*
    *   *Отдает 500 ошибку?* Чиним код приложения.

## Резюме: Шпаргалка сетевика

| Команда | Что делает | Когда применять |
| :--- | :--- | :--- |
| `ip a` | Показывает IP | Узнать свой адрес |
| `ip r` | Показывает шлюз | Проверить маршруты (`default`, `via`, `dev`) |
| `ping -c 4` | Проверка связи | Понять, жив ли сервер (ICMP) |
| `sudo ss -tlnp` | Список портов | Узнать, кто занял порт (нужен sudo) |
| `dig <domain>` | DNS-запрос | Проверить записи домена |
| `curl -I` | Только заголовки | Проверить код ответа (200, 404, 500) |
| `curl -v` | Подробный запрос | Увидеть весь процесс обмена данными |
| `nc -zv <ip> <port>`| Проверка порта | Понять, открыт ли порт на удаленном хосте |
